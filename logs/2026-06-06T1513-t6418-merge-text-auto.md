## Root cause
Both failing subtests run `git checkout -m <tag>` where the target (`a`/`b`) is a
**tag**, not a branch (the repo only has branches `main` and `side`). In grit's
`grit/src/commands/checkout.rs`, branch targets route to `switch_branch`, which
honors the `-m` merge flag and calls `merge_branch_working_tree` (a real
three-way merge that re-applies local modifications). But a tag/commit target
falls through to `detach_head_with_label` -> `detach_head_inner`, which ignored
`branch_merge` entirely and called `switch_to_tree`. `switch_to_tree` runs
`check_dirty_worktree`, which detected the staged/local changes left by the
preceding `git restore --source=a -- .` and aborted with
"Your local changes to the following files would be overwritten by checkout".

Upstream `merge_working_tree` (git/builtin/checkout.c) performs the SAME merge
for both branch and detached checkouts — the detach is only a HEAD update.

## Fix
`grit/src/commands/checkout.rs`:
- Added `detach_head_with_merge(repo, oid, force, label)`: builds the conflict
  presentation (same as `switch_branch`), runs `merge_branch_working_tree` inside
  the checkout smudge context (which performs the three-way merge and writes the
  index but leaves HEAD alone), then writes the detached HEAD, reflog, runs the
  post-checkout hook, and prints the detached-HEAD message — mirroring
  `detach_head_inner`'s HEAD-update tail. Short-circuits to plain detach when
  already at the target and not forced.
- Routing (the `resolve_to_commit` arm in `run`): when `branch_merge_wanted &&
  !switch_force`, call `detach_head_with_merge` instead of `detach_head_with_label`.
  `-f` still discards local changes via the plain detach path.
