# t1007-hash-object.sh — ticket 813c7a

## Status
Was 38/40. Now 40/40.

## Failing subtests (before)
- 15: check that appropriate filter is invoke when --path is used
- 17: --path works in a subdirectory

## Root cause
`git hash-object --path=<file>` was not implemented at all. The `--path` flag was
not present in the clap `Args` struct, so clap rejected the option ("unexpected
argument '--path'"). The argument-checking subtests (`Can't use --path with
--stdin-paths` / `--no-filters`) passed only by coincidence (test_must_fail saw the
clap rejection), but the behavioral tests 15/17 failed because the path-driven clean
(CRLF) filter was never applied — OIDs differed from file0_sha/file1_sha.

## Fix (grit/src/commands/hash_object.rs)
1. Added `--path=<file>` to `Args` (`Option<PathBuf>`).
2. Added explicit conflict checks matching git/builtin/hash-object.c wording:
   - `Can't use --stdin-paths with --path`
   - `Can't use --path with --no-filters`
   (plus reworded the existing --stdin-paths conflicts to match git.)
3. Resolve `--path` into a work-tree-relative "vpath" via `filter_relative_path`
   (git does `prefix_filename(prefix, vpath)`), so `--path=../file1` from a
   subdirectory maps to work-tree path `file1`.
4. Thread the vpath through `read_file_for_hash`: content is still read from the
   real file argument, but the attribute (clean filter) lookup uses the vpath when
   set. Mirrors C: `hash_object(arg, type, no_filters ? NULL : vpath ? vpath : arg)`.
5. Honor `--path` with `--stdin`: stdin content is converted using the vpath's
   attributes (C passes `vpath` to `hash_fd(0, ...)`).
   Extracted `convert_with_attrs()` shared by the file and stdin paths.

`Some(None)` vpath (path outside work tree / no repo) means no attributes apply.
