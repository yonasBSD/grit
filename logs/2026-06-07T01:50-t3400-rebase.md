## Fix 1 (tests 19/20) — DONE
rebase_can_preemptive_ff in grit/src/commands/rebase.rs did not honor the
fork-point (git's `restrict_revision`). When the configured upstream is rewound
after the branch forked off it (default-base reset --hard HEAD^), the second
`git rebase --merge/--apply` wrongly reported "up to date" and left default~1 at
the OLD upstream commit. Git's can_fast_forward() (git/builtin/rebase.c:908)
returns false when restrict_revision (fork-point) != branch_base (onto).

Added `restrict_revision: Option<ObjectId>` param to rebase_can_preemptive_ff;
pass Some(upstream_oid) when fork_point_effective (upstream_oid carries the
effective fork-point/`effective_upstream_for_range`). Tests 19/20 now pass.
