## Test 108: batch missing blob request during checkout — FIXED

Test does `checkout HEAD^` on a `--filter=blob:limit=0` partial clone with
`GIT_TRACE_PACKET` + `GIT_TRACE2_EVENT` set, and asserts:
- exactly one `fetch> done` line in the packet trace (one negotiation round)
- exactly one `"key":"total_rounds","value":"1"` trace2 data event

Root causes in grit:
1. The checkout lazy-fetch path (`hydrate_tree_blobs_from_promisor` ->
   `flush_promisor_blob_batch`, Local branch) just *copied loose objects* from the
   promisor ODB, never spawning an `upload-pack` negotiation. So no packet trace
   (`fetch> done`) was produced and no `total_rounds` event was emitted.
2. grit never emitted `total_rounds` anywhere (`grep total_rounds` over src was empty).

Fixes (both in grit/src, command wiring + transport):
- `grit/src/commands/promisor_hydrate.rs` `flush_promisor_blob_batch`: for a Local
  promisor source, when `promisor_local_lazy_fetch_prefers_upload_pack()` is true
  (i.e. `GIT_TRACE_PACKET` is active, same gate t4067 already uses), route the whole
  batch through `try_lazy_fetch_promisor_objects_batch` (one `upload-pack` round for
  all missing blobs) instead of loose-copying. Non-traced path unchanged.
- `grit/src/fetch_transport.rs` `fetch_upload_pack_negotiate_pack_bytes` (the
  explicit-OID-wants path, used ONLY by promisor lazy fetch): emit
  `trace2_emit_data_intmax(.., "total_rounds", 1)` after the single negotiation round
  — `negotiation_v2` for v2, `negotiation_v0_v1` for v0/v1 — matching upstream
  fetch-pack.c. Explicit-want lazy fetch is always exactly one round.
- `grit/src/commands/promisor_hydrate.rs` `try_lazy_fetch_promisor_objects_batch`:
  post-fetch retention used `exists_local`, which deliberately SKIPS objects in
  `.promisor`-marked packs. After ingesting the freshly-fetched (thin) pack and
  writing its `.promisor` marker, the wanted objects lived in a promisor pack, so
  `exists_local` reported them absent -> the batch bailed "could not fetch N objects"
  even though the checkout had actually succeeded. Switched the two post-fetch
  `need.retain(... exists_local ...)` checks to `repo.odb.exists(...)` (which includes
  promisor packs), mirroring the single-object path's `repo.odb.read(..).is_ok()`.

Result: 112/115. Manual repro: exit 0, one `fetch> done`, one total_rounds=1 event.

## Remaining failures (113, 114, 115) — bundle-uri auto-discover over HTTP
### ROOT CAUSE: environment blocker (Apple Git 2.39.5 as REAL_GIT), NOT a grit bug.

grit ALREADY implements the full bundle-uri chain on both sides:
- Client: `run_http_clone` -> `maybe_apply_bundle_uri_after_http_fetch_with_client` ->
  `read_bundle_list_from_http` (POST `command=bundle-uri` over v2) ->
  `apply_bundle_list_for_fetch` -> `read_bundle_uri_bytes` (HTTP GET) ->
  `trace2_child_start_git_remote_https(uri)` which emits exactly the
  `child_start argv:["git-remote-https","<uri>"]` event the test greps for.
- Server: `serve_v2.rs` advertises `bundle-uri` (gated on `uploadpack.advertiseBundleURIs`)
  and handles `command=bundle-uri` (`cmd_bundle_uri`, streams `bundle.*` config).

But the v2 capability ADVERTISEMENT during smart-HTTP clone is produced by the harness's
`git-upload-pack` wrapper, which runs `$REAL_UP --advertise-refs` (REAL git) for the
advertise phase — only the fetch/negotiation is delegated to grit (see lib-httpd.sh
~L242-254; bundle-uri is only routed to grit when `BUNDLE_URI_PROTOCOL=http`, which
t5601 does NOT set).

In THIS environment `REAL_GIT=/usr/bin/git` is **Apple Git 2.39.5 (Apple Git-154)**,
which does NOT advertise the `bundle-uri` capability at all:
  `GIT_PROTOCOL=version=2 /usr/bin/git upload-pack --advertise-refs <repo>` ->
  advertises only `version 2, ls-refs, fetch, server-option, object-format, object-info`
  (NO `bundle-uri`). Homebrew git 2.52.0 on the same machine DOES advertise it.

Verified end-to-end with a standalone httpd repro: grit's clone GETs
`/info/refs?service=git-upload-pack` (Git-Protocol: version=2), the server reply
contains `version 2` but no `bundle-uri` cap, so grit (correctly) reports
"server does not advertise bundle-uri" and skips discovery — exactly as upstream git
would against the same advertisement. The ported test bodies for 113/114/115 are
byte-identical to git/t/t5601-clone.sh.

Conclusion: 113/114/115 are UN-PASSABLE in this environment without a newer REAL_GIT
or a harness change to route the bundle-uri advertise through grit's serve_v2 (both
out of scope: cannot modify tests / test-lib.sh, cannot change the system git). No grit
Rust change can satisfy them while the advertiser is Apple Git 2.39.5.
