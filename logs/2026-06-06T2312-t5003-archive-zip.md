# t5003-archive-zip

Ticket: 9651c6

## Starting state
- 81/82 passing. Failing subtest 78: "git-archive --format=zip with bigFile delta chains".

## Diagnosis
The failing subtest is misleadingly named: the failure is not in `git archive`.
The test builds three progressively-growing blobs (base, delta-1 = base + line,
delta-2 = delta-1 + line), runs `git repack -Ad` to deltify them into a chain,
then asserts with `git verify-pack -v`:

    chain length = 1: 1 object
    chain length = 2: 1 object

grit's `verify-pack` reported `chain length 0: 3` and `chain length 1: 2`. The
pack itself was correct (base full, 2d4527 delta on base, 60b225 delta on
2d4527 — a real depth-2 chain). The bug was in depth *reporting*.

`verify_pack_and_collect` in grit-lib/src/pack.rs computed each delta's depth in
a single pass in pack/offset order. For OFS_DELTA the base always precedes the
delta so that worked, but for REF_DELTA the base oid can be stored *after* the
delta in the pack. When the depth-2 delta was processed, its base (a depth-1
delta) had not yet had its depth assigned (still `None` → treated as 0), so the
delta was reported as depth 1 instead of 2.

## Fix
Rewrote depth resolution in `verify_pack_and_collect` to be order-independent:
- Capture each delta's base as a `DeltaBaseLink` (Oid for ref-delta, Offset for
  ofs-delta) while scanning headers, instead of computing depth inline.
- After scanning, resolve depths with a memoized recursive walk
  (`resolve_delta_depth`) that follows base links to their record index and sets
  `depth = base_depth + 1`. Full objects are depth 0; a base missing from the
  pack (thin) or a cyclic link counts as depth 1; memoization keeps it O(n) and
  a pre-set sentinel prevents infinite recursion on malformed cycles.

Files changed:
- grit-lib/src/pack.rs (added DeltaBaseLink enum + resolve_delta_depth, rewrote
  the depth loop in verify_pack_and_collect)
- data/tests/t5/t5003-archive-zip.toml
