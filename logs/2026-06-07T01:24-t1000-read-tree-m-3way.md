## Starting state
82/83 passing. Failing subtest 83: "16 - A matches in one and B matches in
another." which runs `git read-tree -m $tree0 $tree1 $tree1 $tree0` (a symmetric
4-tree merge).

## Diagnosis
`grit read-tree -m` errored with `too many trees for -m (max 3)`. The guard in
`grit/src/commands/read_tree.rs::run` rejected `args.merge && tree_oids.len() > 3`
*before* the merge dispatch, even though:
- A `four_tree_symmetric_merge` helper already existed in the file, and
- The merge dispatch already had a `4 =>` arm (requires the symmetric layout
  T0 T1 T1 T0) wiring it up.

Git's `builtin/read-tree.c` allows up to MAX_UNPACK_TREES (8) trees for `-m`;
for >= 3 trees it runs threeway_merge with `head_idx = stage - 2`.

Real git output for the test case:
```
100644 ecbec6db011caa3ae26739812f092f76875fe6b6 2	F16   (tree1 / E16)
100644 bcf4afda2c30299bf6538240d40bcb8fb3857691 3	F16   (tree0 / F16)
```

## Fix
Removed the `args.merge && len > 3` cap (kept the overall `len > 4` cap). Now 4
symmetric trees flow to the existing `four_tree_symmetric_merge`, which emits
stage 2 = t1 and stage 3 = t0 when the two trees disagree — matching git exactly.
