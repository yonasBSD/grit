## Verification of test 103 (no grit fix possible)
The test builds `expect` in `src` and `actual` in `dst`, then `test_cmp`:
- `expect` = `git for-each-ref | sed ... | sort -k 4`
- `actual` = `git for-each-ref` (plain, refname order)

Each `for-each-ref` line is `<oid><SP><type><TAB><refname>`. With default blank
field separators (space AND tab), this is **3 fields** (oid, type, refname) —
confirmed `awk '{print NF}'` = 3.

`sort -k 4` therefore names an **empty / out-of-range key**, so both BSD (this host:
`sort 2.3-Apple`) and GNU sort fall back to whole-line comparison, ordering by the
leading **OID**. Decisive control test:

```
printf 'zzz commit\trefs/a\naaa commit\trefs/z\n' | sort -k 4
# -> emits `aaa ... refs/z` FIRST (OID/whole-line order),
#    not `zzz ... refs/a` (which a refname key would put first)
```

Consequences:
- `expect` is OID-ordered: tag `0ddcfd7...` < commit `b9c09d6...`, so **tag is FIRST**.
- `actual` is refname-ordered, so `refs/tags/tag` is **LAST**.
- Inherent mismatch on this host.

Reconstructed the scenario with **real git's exact OIDs** (b9c09d6 commit, 0ddcfd7
tag — the same values grit emits in the verbose run): `diff expect actual` still
MISMATCH. So **real git 2.52.0 fails this test identically on this macOS host**.

Confirmed grit's object hashing is byte-identical to git: from identical input both
grit and `/opt/homebrew/bin/git` produced commit OID `7f0f539...`.
