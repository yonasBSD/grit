## Failing subtest
- 188: `format-patch --base errors out when base commit is not ancestor of revision list`

History under test (criss-cross):

    ---Q---P---Z---Y---*---X
       \             /
        ------------W

`git format-patch -3` selects the 3 non-merge commits Y, W, X (reversed for output: Y, W, X).
`--base=P` and `--base=Z` must FAIL; `--base=Q` (the fork point) must succeed.

## Root causes (three layered bugs, all in shared machinery)

1. **`rev-list --no-merges -N` / `format-patch -N` over a merge dropped real commits.**
   In `grit-lib/src/rev_list.rs`, parent-count filters (`--no-merges`, `--max-parents`,
   `--min-parents`) removed merge commits from `included` *before* the ordering walk, severing
   the graph: walking from X could not reach W/Y because X's only parent (the merge) was gone.
   Git only suppresses dropped commits from *output*; it still walks *through* them.
   Fix: added `date_order_walk_through_dropped` — a pure committer-date priority queue seeded
   from the user tips that traverses the full reachable set and emits only the surviving
   (filtered) commits, stepping over dropped merges. Replaced the old
   `date_order_walk_with_tips` (which stopped at dropped parents).

2. **Equal-date tiebreak diverged from Git.** `CommitDateKey` broke date ties by OID; Git's
   `prio_queue` (prio-queue.c) breaks them by *insertion order* (earlier insertion emitted
   first). Added a `seq` field to `CommitDateKey`: among equal-date commits the earlier-reached
   one (e.g. the first parent of a merge) wins. Walks that don't care leave `seq == 0` →
   falls through to the prior OID tiebreak, preserving existing behavior. With this, grit
   selects W before Y, matching `git rev-list --no-merges -3`.

3. **`--base` validation used the wrong reference.** grit only checked that base was an ancestor
   of the *first patch's parent*. Git's `get_base_commit` (builtin/log.c) checks base against the
   *pairwise-reduced merge base of every commit in the series* (`reduce(rev[])` then
   `in_merge_bases(base, rev[0])`). Added `reduced_pairwise_merge_base` in
   `grit/src/commands/format_patch.rs` mirroring Git's pairwise reduction (exactly-one merge base
   per pair, else "failed to find exact merge base"), and check `is_ancestor(base, rev0)`.
   Also corrected the two error messages to match Git
   ("base commit should be the ancestor of revision list" /
   "base commit shouldn't be in revision list").
