## Fixed: tests 118 & 119 (fetch-to-worktree refusal)

Two independent root causes, both now resolved:

### A. Maintenance-after-fetch detach race (test 119, and exposed 118)
Test 119 cleanup `rm -fr bare.git` raced a detached `git maintenance run --auto`
child spawned by fetch, causing `rm: bare.git/objects: Directory not empty`.

Upstream's `git/t/test-lib.sh` sets `GIT_TEST_MAINT_AUTO_DETACH=false` (line 1965)
"so tests cannot race with background maintenance by default". grit's ported
`tests/test-lib.sh` omitted this line (test-lib.sh is not editable per the rules;
`tests/t7900-maintenance.sh:29` `sane_unset GIT_TEST_MAINT_AUTO_DETACH` confirms the
harness is *expected* to set it).

- `scripts/run-tests.sh`: inject `GIT_TEST_MAINT_AUTO_DETACH=false` alongside the
  other `GIT_TEST_*` env vars (matches upstream test-lib; harness file, not test-lib.sh).
- `grit/src/commands/fetch.rs` `maybe_run_auto_maintenance_after_fetch`: honor
  `GIT_TEST_MAINT_AUTO_DETACH` when deciding foreground vs detached maintenance
  (config still wins if set), consistent with `run_auto_after_commit`. Without this
  the env var alone had no effect on the fetch path.

### B. tag-of-blob breaks submodule-change walk (test 118)
Once the race was gone, 118 failed with
`error: corrupt object: object <blob> is a Blob, not a commit` during the final
`git fetch -u . HEAD:wt`. Backtrace:
fetch::run -> recursive_fetch_submodules_after_fetch ->
collect_changed_submodules_for_fetch -> rev_list -> resolve_specs_with_options ->
peel_to_commit. The submodule-change detector feeds every freshly-fetched ref tip
(incl. `refs/tags/tag-of-blob`, created by test 111 and never deleted) into a commit
rev-list. grit's `peel_to_commit` hard-errored on a tag peeling to a blob.

Real git's `git rev-list tag-of-blob` (and even `git rev-list <blob-oid>`) exits 0
with no output — a non-commit positive tip simply contributes no commits.

Fix in `grit-lib/src/rev_list.rs`:
- `resolve_specs_with_options`: when a positive spec peels to a non-commit (blob/tree
  or tag-of-blob), skip it silently instead of returning `CorruptObject` — matching
  `git rev-list`. Genuinely missing/corrupt objects still error.
- Added helper `object_peels_to_non_commit(repo, oid)` to distinguish "readable object
  that peels to blob/tree" (skip) from "unreadable/corrupt" (error).

`git rev-list HEAD tag-of-blob` now correctly lists HEAD's commits (exit 0), which is
the case the submodule walk needs.

## Remaining: test 103 'fetch follows tags by default' — NOT a grit bug
The test builds `expect` via `... | sort -k 4` and compares to `dst`'s
`git for-each-ref` (`actual`). The for-each-ref line is `<oid> <type>\t<refname>` =
3 whitespace-separated fields, so `-k 4` is an EMPTY key; both GNU and BSD `sort` fall
back to the whole-line comparison (verified with `gsort --debug`: "no match for key"),
ordering by OID, NOT by refname. So the result hinges on whether the annotated tag's
OID sorts before/after the commit OID.

Proven this is identical for real git:
- first commit OID `b9c09d6…` and tag OID at tagger time 1112912533 = `0ddcfd7…`
  are produced byte-for-byte by both grit AND installed git 2.52.0 (independently
  hashed). grit's cumulative `test_tick` count at test 103 (10 ticks) is also correct.
- `0ddcfd7… < b9c09d6…` so `sort -k 4` puts the tag FIRST in expect while for-each-ref
  (refname order) puts it LAST in actual → mismatch for real git too on this host.

It is a host/coreutils-dependent upstream test (passes only when the random-ish tag OID
happens to sort after the commit OID). Cannot fix grit; fixing requires editing the
test (forbidden). Matches prior agents' "sort -k4 artifact" finding.
