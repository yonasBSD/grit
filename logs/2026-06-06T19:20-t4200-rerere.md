## Root cause

Subtest 32 exercises a merge where the "ours" side of a 3-way merge already
contains conflict markers from a *previously committed* unresolved conflict, and
"theirs" modifies the single base line that those markers surround:

- base   = `bar`
- ours   = `<<<<<<< HEAD` / `foo` / `=======` / `bar` / `>>>>>>> A~`
- theirs = `baz`

Git produces a single conflict whose ours-side is the *entire* ours content
(markers and all) vs `baz`, yielding well-formed *nested* markers:

```
<<<<<<< HEAD
<<<<<<< HEAD
foo
=======
bar
>>>>>>> A~     <- inner conflict closes BEFORE the outer separator
=======
baz
>>>>>>> A
```

Grit instead matched the `bar` line inside ours to the base `bar`, applied
theirs' `bar`→`baz` change "cleanly", and kept the surrounding markers as
ours-only insertions — producing malformed nesting (outer `=======` appearing
before the inner `>>>>>>>` closes) or even a clean (exit 0) merge. rerere then
could not parse / record a postimage, so the replayed resolution never applied
and `test` was left with broken markers.

This was NOT a rerere bug — it was a `grit merge-file` / `merge_file::merge`
conflict-region bug. Isolated repro:

    printf '<<<<<<< HEAD\nfoo\n=======\nbar\n>>>>>>> A~\n' > ours
    printf 'bar\n' > base ; printf 'baz\n' > theirs
    grit merge-file -p ours base theirs   # diverged from git

## Why grit diverged

`compute_hunks` emitted three one-sided hunks for this case:

1. `OnlyOurs { base:[], ours:[<<<, foo, ===] }`   (leading insert)
2. `OnlyTheirs { base:[bar], theirs:[baz] }`       (theirs modifies base line)
3. `OnlyOurs { base:[], ours:[>>>] }`              (trailing insert)

`merge_adjacent_replace_and_trailing_insert_conflicts` already folds the
*leading* insert (1+2) into a conflict, but the *trailing* insert (3) was left
dangling. Git's `xdl_merge` (`xdl_append_merge`: `i1 <= m->i1 + m->chg1`)
attaches a one-sided insertion that *touches* a change region — so the trailing
`>>>>>>> A~` insert at the base position immediately after the contested line is
absorbed into the conflict's ours side.

## Fix

`grit-lib/src/merge_file.rs`: new pass `fold_adjacent_inserts_into_conflict`,
run right after `merge_adjacent_replace_and_trailing_insert_conflicts`. It folds
a pure one-sided insertion (`Only* { base: [] }`) that *immediately follows* a
`Conflict` into that conflict's matching side (empty base ⇒ no intervening base
line ⇒ they touch, matching git's `xdl_append_merge`).
