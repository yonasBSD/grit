## What was fixed in grit (server side is now fully correct)

Three grit Rust changes, all verified correct end-to-end:

1. `grit/src/commands/upload_pack.rs` â€” `write_ref_advertisement`:
   the empty-repo (unborn HEAD, no refs) v0 carrier line now advertises
   `<zero-oid> capabilities^{}` instead of `<zero-oid> HEAD`. A real-git client
   that sees a `HEAD` ref at the all-zero OID tries to `want` it and the
   negotiation breaks; `capabilities^{}` is git's standard no-ref carrier that
   still ships the capability list (incl. `object-format`). This matches modern
   git byte-for-byte.

2. `grit/src/bin/test_httpd.rs` â€” `handle_smart_http_grit_upload_pack`:
   for the discovery GET it now prepends the `# service=git-upload-pack` pkt-line
   + flush banner that `git-http-backend` normally adds (grit's
   `upload-pack --http-backend-info-refs` emits only the bare advertisement).
   Real-git clients require this banner; grit's own client tolerates it (it calls
   `strip_v0_service_advertisement_if_present`).

3. `grit/src/bin/test_httpd.rs` â€” new `smart_sha256_upload_pack` routing:
   `/smart/<repo>` requests for a repo whose `config` has
   `extensions.objectformat = sha256` are now served by grit's own `upload-pack`
   (like the existing reftable / bundle-uri-http routes) instead of the CGI
   `git-http-backend` path. The CGI path wires `--advertise-refs` to the host's
   system git (see `lib-httpd.sh`), and on Apple git 2.39.5 the empty-sha256 v0
   info/refs body is just two flushes (no object-format).

### Verification that the grit server is correct

Serving the empty sha256 repo from grit's `test-httpd` (with the fixes above) and
cloning it:

| client                     | v0     | v2     |
| -------------------------- | ------ | ------ |
| grit (this repo)           | sha256 | sha256 |
| modern git 2.52.0 (brew)   | sha256 | sha256 |
| Apple git 2.39.5 (Xcode)   | sha1   | sha1   |

So grit advertises `object-format=sha256` correctly in both protocol versions and
both grit-as-client and modern-git-as-client record it. Upstream CI passes 36/37
because it uses a modern git.

## Remaining blocker (environment, not a grit defect)

The harness `tests/lib-httpd.sh` hybrid `git` wrapper routes any `clone http(s)://`
to `REAL_GIT`, which on this host resolves to Apple/Xcode git 2.39.5. That old
client does NOT record `object-format` for an EMPTY-repo clone even when the server
advertises it correctly (verified directly against the grit server, table above).
grit is therefore never the client for subtests 36/37, and the only ways to make
them pass are harness-level:
  1. route `*/smart/sha256*` clones to grit (GUST_BIN) in `lib-httpd.sh`, or
  2. point the harness REAL_GIT at a newer git.
Both require editing `tests/lib-httpd.sh`. AGENTS.md / the MOP-UP contract forbid
modifying files under `tests/` (only `test_expect_failure -> test_expect_success`
flips are allowed), and a prior agent's auto-mode classifier explicitly denied the
`lib-httpd.sh` edit. No grit-code-only path closes 36/37.
