# t0021-conversion — required process filter should filter data

Ticket: 70bac3
Date: 2026-06-06T22:39 UTC

## Starting state
- 41/42 passing. Failing subtest 17: "required process filter should filter data".

## Root cause
The failing subtest exercises a long-running `filter.<driver>.process` filter
(`filter.protocol.process = test-tool rot13-filter ... clean smudge`) and asserts the
exact set of clean/smudge invocations logged by the filter.

After `rm -f test2.r testsubdir/test3...r`, the test runs `git checkout --quiet --no-progress .`.
Git only smudges the two *missing* paths; the two still-present, stat-clean paths
(`test.r`, `test4-empty.r`) are skipped. grit instead smudged ALL FOUR paths.

The cause: `write_blob_to_worktree` (grit/src/commands/checkout.rs) decided whether a path
was already up to date by reading the worktree file and running `crlf::convert_to_worktree`
(which *invokes the smudge filter*) to compare bytes. Under a process filter, that comparison
itself logged a `smudge` invocation for every up-to-date path, polluting the filter log.

Git's `checkout_entry` skips up-to-date entries via `ie_match_stat` (stat comparison against
the index entry) *before* touching the clean/smudge machinery, so up-to-date paths are never
smudged.

## Fix
Added a stat-clean fast path at the top of `write_blob_to_worktree` for path-only checkouts
(`!full_smudge_meta`, non-symlink): if the matching stage-0 index entry has the same oid+mode
and the worktree file's stat matches the index entry (`stat_matches`), return `Ok(false)`
(skip) without running any filter. Falls through to the existing smudge-comparison path only
when stat is dirty/missing.

File: grit/src/commands/checkout.rs, fn write_blob_to_worktree.
