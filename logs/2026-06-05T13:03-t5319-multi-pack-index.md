## Fix 1 — subtest 8 "corrupt idx reports errors"
Root cause: `git rev-list --objects --all` with `core.multiPackIndex=true` over a
MIDX that names a pack whose `.idx` was truncated to 1064 bytes must emit
`error: packfile <pack> index unavailable` and still succeed. The objects in the
walk are all *loose* (the `.idx`/`.pack` from `pack-objects` are copies), so grit
found them loose and never reached the per-object MIDX lookup that already had the
warning. Git emits this at pack-store *preparation* time (open_pack_index on every
MIDX-referenced pack), independent of where the object is found.

Fix: added `midx::validate_midx_referenced_packs(objects_dir)` (runs once per
process per objects_dir) that opens every pack idx the MIDX names and emits
`error: packfile <pack> index unavailable` via the existing `midx_warn_once` guard
for any that fail to parse. Called from `Odb::read()` when MIDX is enabled, before
the loose lookup — mirroring `prepare_packed_git`.

## Fix 2 — subtest 25 "preferred packs must be non-empty"
Root cause: `git pack-objects <base> </dev/null` (empty stdin object list) must write
an *empty* pack file and print its hash, so the subsequent
`multi-pack-index write --preferred-pack=pack-<empty>.pack` can reject it with
"cannot select preferred pack ... with no objects". grit's pack-objects returned
early without writing the empty pack or printing a hash, so `$empty` was empty and
the preferred-pack name never matched.

Fix: in grit/src/commands/pack_objects.rs, when the object list is empty, output is a
file (base_name set, not --stdout) and `--non-empty` is NOT set, write the canonical
empty pack via new helper `write_empty_pack_to_file` and print its hash. Scoped to the
bare stdin-object-list mode (the existing `!args.stdin_packs && (!args.cruft || ...)`
guard) so repack/geometric/maintenance paths (which pass --non-empty or --stdin-packs)
are unaffected. Verified t5300 (63/63), t5616 (46/47), t7700 (36/47) unchanged.

## Fix 3 — subtest 56 "repack preserves multi-pack-index when creating packs"
Root cause: grit's `repack` unconditionally deleted the MIDX (`clear_pack_midx_state`)
at the start of any full repack (`args.all`). Git only drops the MIDX when it is about
to rewrite it (`--write-midx`) OR when a pack the MIDX *references* is removed as
redundant (git/repack.c `repack_remove_redundant_pack` -> `clear_midx_file` gated on
`midx_contains_pack`). In subtest 56 the MIDX was built from two `.keep` packs that are
NOT repacked, so `repack -ad` must leave the MIDX byte-for-byte intact; grit deleted it.

Fix (grit/src/commands/repack.rs, non-geometric path only — geometric returns early):
- Only clear the MIDX up front when `--write-midx` (it rewrites below).
- Otherwise snapshot the MIDX-referenced pack idx names (`read_midx_pack_idx_names`)
  before repacking, and after redundant-pack removal clear the MIDX iff any referenced
  pack's `.pack` no longer exists on disk. Kept packs survive -> MIDX preserved.
Verified t7700 (36), t5324 (17), t6501 (42) unchanged. t7900 sub29 still pre-existing fail.

Result: 86/98.

## Fix 4 — subtest 61 "multi-pack-index in an alternate"
Root cause: `git repack --local` ran `pack-objects --all --unpacked --incremental --local`,
which grit routes to `collect_incremental_repack_oids`. That function honored `--unpacked`
(skip objects in *local* packs) but ignored `--local`, so objects that live only in an
ALTERNATE ODB (here, packs moved into alt.git) counted as "unpacked" and were copied into
a new local pack -> the MIDX then reported 91 objects instead of 3.

Fix (grit/src/commands/pack_objects.rs): in `collect_incremental_repack_oids`, when
`args.local`, drop OIDs present in the alternate object store (`alternate_object_ids`,
the same exclusion the cruft path already used). Mirrors git/pack-objects.c
`want_object_in_pack` honoring `local`. Verified t7700 (36), t5616 (46), t6500 (35),
t5304 unchanged.

Result: 87/98.

## Fix 5 — subtest 97 "bitmapped packs are stored via the BTMP chunk" (+ 76 cascaded)
Root cause: `git repack -d` (incremental) over 5 commits should produce 5 packs of exactly
3 objects each (one commit's blob+tree+commit). grit produced packs of 4/5/2 objects: the
incremental pack-objects re-packed already-packed parent blobs/trees. grit's
`rev-list --objects --unpacked` walk correctly emits the FULL closure of unpacked commits
(matching git for t6000/t6113), so the dedup must happen in pack-objects via `--incremental`:
"an object already in a pack is ignored even if it would otherwise be packed." grit only
applied that exclusion when `--no-write-bitmap-index`, not for `--incremental` generally.

Fix (grit/src/commands/pack_objects.rs `collect_incremental_repack_oids`): change the
already-packed-object exclusion gate from `args.no_write_bitmap_index` to `args.incremental`.
This makes each per-commit incremental pack hold exactly its new objects, so the MIDX BTMP
chunk records bitmap_nr=3 per pack. Cascaded: subtest 76 (cruft batch-size) also now passes.
Verified t5300 (63), t6000 (23), t6113 (14), t6500 (35), t7700 (36), t5310 (151), t7701 (1,
pre-existing), t5332 (14) all unchanged.

Result: 89/98.
