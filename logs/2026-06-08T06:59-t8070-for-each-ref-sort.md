# t8070-for-each-ref-sort — port to passing

## Starting state
29/30 passing. Failing subtest: #23 "count: --count larger than total returns all".

## Root cause (differential-verified vs git 2.52.0)
The subtest body did:

```sh
git for-each-ref >all &&
total=$(wc -l <all) &&
git for-each-ref --count=100 >out &&
test_line_count = "$total" out
```

On macOS, BSD `wc -l` left-pads its output (`       7`). The command substitution
`total=$(wc -l <all)` preserves that leading whitespace, so `$total` = `"       7"`.
`test_line_count = "$total" out` then string-compares the (stripped, unquoted)
actual count `7` against the padded expected `"       7"` and fails:

```
test_line_count: expected        7 lines (=), got 7 in 'out'
```

This is a classic BSD `wc` padding TEST bug — a comparison-mechanism portability
issue, not a value mismatch.

### Differential proof
Built identical repo in /tmp, ran both binaries:
- real git 2.52.0 (`/opt/homebrew/bin/git`): `for-each-ref` => 7 refs, `--count=100` => 7 refs
- grit (`target/release/grit`):              `for-each-ref` => 7 refs, `--count=100` => 7 refs

Grit matches real git byte-for-byte. No grit bug.

## Fix (test-body only)
Strip the BSD padding from the captured count:

```sh
total=$(wc -l <all | tr -d " ") &&
```

## Result
30/30 passing. fully_passing = true.
