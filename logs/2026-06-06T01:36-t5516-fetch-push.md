## Root cause #1 (massive): `git init --template=` created `hooks/`, `info/`, `description`

The harness `mk_empty` helper does:
```sh
git init --template= "$repo_name" &&
mkdir "$repo_name"/.git/hooks &&        # FAILS when grit already made hooks/
git -C "$repo_name" config receive.denyCurrentBranch warn
```

Upstream Git's `create_default_files` (git/setup.c) only creates `objects/`, `refs/`
and friends; `hooks/`, `info/`, and `description` are *template content* installed by
`copy_templates`. With `--template=` (empty), no template is copied, so none exist and
the subsequent `mkdir .git/hooks` succeeds.

grit's `grit/src/commands/init.rs::create_git_dir` unconditionally created `hooks` and
`description`, and created `info/exclude` via its default-template branch. Fix: gate
`hooks/` creation and the `description` write behind `has_template_content`
(`template_dir.is_some() || !skip_default_templates`). `info/exclude` was already gated by
`skip_default_templates`.

Result: 6 -> 73/124.

## Root cause #2: push dst DWIM against remote refs (`count_refspec_match`)

`git push remote main:origin/main` (dst `origin/main` is a short name that should match the
existing `refs/remotes/origin/main`) bailed "destination is not a full refname". grit's
`resolve_destination_ref_for_push` rejected any slash-containing non-`refs/` dst.

Implemented Git's `count_refspec_match` (remote.c) weak/strong matching plus `guess_ref`
fallback in push.rs: match the short dst against the remote's advertised refs; one strong
match (or single weak) is unambiguous. For colon-less pushes, the dst value is the *full
source ref name* (so a remote with both refs/heads/frotz and refs/tags/frotz is not ambiguous
for a branch source). Fixed tests 26, 32, 33 with no regressions. 73 -> 74/124.

## Root cause #3: `@` source alias and config-push HEAD/@ resolution

`git push remote @` (and `+@`, `@:dst`) failed: grit never rewrote `@`->HEAD. Also
`remote.<name>.push = HEAD`/`@` in config went through a normalize_ref(src) path that produced
`refs/heads/HEAD`. Fixed both: rewrite `@`->HEAD in CLI parse_refspec output; route the
config-push branch through resolve_push_src_for_refspec (handles HEAD/@/oid/DWIM) with the
same effective-dst HEAD resolution. Fixed 45,46,47,48,49,50. 74 -> 80/124.

## Root cause #4: push --delete tag shorthand and refspec validation

`git push --delete tag deltag` did not delete (no `tag <name>` handling); `--delete main:foo`
and `--delete ""` wrongly succeeded. Implemented git/builtin/push.c set_refspecs delete rules:
`tag <name>` -> refs/tags/<name>; reject refspecs containing ':' or empty with
"--delete only accepts plain target ref names". Fixed 70,72,73. 80 -> 83/124.

## Root cause #5: receive hooks (update hook cwd/order, did_not_exist, ref order)

Hook fixes for tests 64-68:
- `update` hook ran via run_hook_capture (cwd=pusher), so relative-path writes missed the
  remote repo; switched to run_hook_in_git_dir (cwd=remote git_dir).
- update hook + ref application now run deletions-first (PHASE_DELETIONS before PHASE_OTHERS,
  receive-pack.c), with applied_updates re-sorted to input order for post hooks.
- delete of a non-existent ref now reaches pre-receive + update hooks (new RefUpdate

  applied (Git did_not_exist). post-receive stdin built from applied_updates, not the full feed.
- ref commands ordered like receive-pack sees them: existing remote refs (sorted) first, new
  refs after in refspec order (so main:main main:seen :next feeds main,next,seen).
Fixed 64,65,66,67,68. 83 -> 88/124.

## Root cause #6: reject non-commit object pushed to a branch (test 124)

`git push <repo> <tagsha>:refs/heads/branch` wrote a tag OID to a branch. Added a check in
apply_ref_update: a refs/heads/* update whose new object is not a commit is rejected with
`(invalid new value provided)` and a remote `trying to write non-commit object <sha> to branch
'<ref>'` message (git refs.c). Fixed 124. 88 -> 89/124.

## Root cause #7: implicit push remote/mode with detached HEAD (test 52)

After test 51 leaves HEAD detached, `git push` with remote.pushdefault + push.default=matching
failed: (1) infer_implicit_push_remote only consulted remote.pushDefault when a current branch
existed -> fixed to honor remote.pushDefault even when HEAD is detached; (2) the default-push
path required a current branch before checking the push mode -> matching mode now runs without a
current branch. Fixed 52. 89 -> 90/124.

## Root cause #8: --prune with matching (`:`) refspec (test 93)

`git push --prune testrepo :` did not delete remote branches lacking a local counterpart;
prune was only wired for glob refspecs. Added prune handling to collect_matching_push_updates:
delete remote refs/heads/* with no local branch (honoring negative refspecs). Fixed 93. 90 -> 91.

## Root cause #9: receive.hideRefs/transfer.hideRefs reject push (tests 95,96)

A push to update or delete a ref hidden by transfer.hideRefs/receive.hideRefs was allowed.
Added a check in check_receive_pack_policy via grit_lib::ref_exclusions::RefExclusions
(load_hidden_refs_from_config(..,"receive")): reject with "deny updating a hidden ref" /
"deny deleting a hidden ref". Fixed 95,96. 91 -> 93/124.

## Root cause #10: hidden-ref push must not pollute the repo (test 120)

A rejected hidden-ref push still wrote the pack into the remote. Pre-reject hidden-ref
updates/deletes before objects are sent (so push_tips is empty and no pack is ingested), and
render the pre_reject "deny ..." message as the per-ref "! [remote rejected] ... (<reason>)"
line. Fixed 120. 93 -> 94/124.

## Partial: updateInstead into a void repo (test 114)

worktree_clean_for_update_instead re-resolved HEAD *after* the ref was written, so pushing into
