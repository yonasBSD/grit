## Fixed: 72, 75, 79 (--simplify-by-decoration) -> 143 -> 146/149

Root causes (three separate bugs):

1. **Hidden-ref decorations not revealed when a decorate filter is active.**
   `collect_decorations_inner` (grit/src/commands/log.rs) only decorated the known
   namespaces (heads/tags/remotes/stash/HEAD/replace); refs like `refs/hidden/*`,
   `refs/prefetch/*`, `refs/rewritten/*` were silently dropped except under
   `--clear-decorations`. Git's `set_default_decoration_filter` (builtin/log.c) only
   restricts to the known namespaces when NO `--decorate-refs[-exclude]` /
   `log.excludeDecoration` is given; as soon as any such filter is active, the default
   include list is dropped and every ref that passes the filter is decorated with its
   full name and `DECORATION_NONE` (reset) color. Added a new `DecorationKind::Other`
   and a fallthrough that decorates unknown-namespace refs when
   `clear_decorations || !filter.is_empty()`.

2. **--simplify-by-decoration used a crude retain instead of real history simplification.**
   Both the legacy log walk (log.rs) and the rev-list engine (rev_list.rs:1026) just
   kept commits that had a ref tip, which dropped needed commits, severed the graph, and
   produced wrong ordering. Git treats a decorated commit as TREE_DIFFERENT and everything
   else as TREESAME (rev_compare_tree), then runs standard parent-rewriting simplification.
   - Added `RevListOptions::simplify_by_decoration_oids` (the filter-aware decorated set).
   - Engine: replaced the crude retain with `compute_simplify_by_decoration_keep_set`
     (rev_list.rs) — processes the topo-ordered walk in reverse (parents first), rewrites
     each commit's relevant parents past pruned/undecorated commits, removes redundant
     parents, and keeps a commit iff decorated OR root OR a relevant merge (>=2 distinct
     rewritten relevant parents).
   - Routing: `--simplify-by-decoration` without pathspecs now goes through
     `run_rev_list_log`; for default (unspecified) ordering it walks in Topo order, which
     is what Git emits for the rewritten topology (e.g. `reach` before the older `seventh`).
   - `run_rev_list_log` computes the decorated set from `collect_decorations_inner` (same
     filter-aware map the display uses).

3. **--clear-decorations was not order-sensitive.**
   Git's `clear_decorations_callback` clears `--decorate-refs[-exclude]` patterns given
   *before* it but keeps those given *after*. clap collects them order-insensitively.
   Added `ordered_decorate_ref_patterns` which rebuilds include/exclude from
   `raw_argv_tail`, clearing on `--clear-decorations`. Fixes subtest 79 both blocks.
