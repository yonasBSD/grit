# t3412-rebase-root.sh — rebase -i --root with conflict

Ticket: caa825

## Starting state
23/25. Failing subtests:
- 22: `rebase -i --root with conflict (first part)`
- 24: `rebase -i --root with conflict (second part)`

## Root cause
At subtest 22 the `other` branch history contains two merge commits
(`Merge branch 'side'`, `Merge branch 'third'`). `git rebase -i --root --onto main`
generated a todo list containing `pick <merge>` lines, which `pick` rejects:

```
error: 'pick' does not accept merge commits
error: invalid line 3: pick ddb0186 # Merge branch 'side' into other
```

Upstream `sequencer_make_script` (git/sequencer.c) sets `revs.max_parents = 1`
unless `--rebase-merges` is given, so a plain interactive rebase NEVER emits
`pick <merge>` lines — merge commits are excluded from the walk entirely, while
their non-merge ancestors on every side are still picked individually.

grit's `collect_commits_for_root_rebase` always used a `--first-parent` walk and
kept merge commits in the list (the sequential/apply root backend later flattens
them via `message_for_root_replayed_commit`, recording the second parent's
message). That is correct for the non-interactive `--root` backend (tests 19/21
pass), but wrong for the interactive backend, which builds a literal `pick` list.

The filter that drops merges (`filter_merge_commits`) was explicitly skipped for
`args.root` (`if !args.root && !rebase_merges_on { ... }`), so nothing removed
the merges in the interactive root case.

## Fix (grit/src/commands/rebase.rs)
- `collect_commits_for_root_rebase` gained an `exclude_merges` parameter. When
  true it mirrors upstream: `max_parents = Some(1)` over a topo walk (reverse),
  excluding merge commits. When false it keeps the original first-parent
  flattening walk for the sequential/apply backend.
- Call site passes `exclude_merges = args.interactive && !rebase_merges_on`.

The existing `filter_redundant_patch_commits` step (driven by `args.onto.is_some()`)
still drops the `1b` cherry-pick equivalent of `1` on `main`, so the interactive
todo becomes the expected linear `3 4 5 6` (+ `conflict` from `main`), producing
`expect-conflict` = `6 5 4 3 conflict 2 1`.
