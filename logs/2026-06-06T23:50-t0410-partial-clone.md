# t0410-partial-clone

Ticket 235337. File regressed to 37/38 after its original ticket closed.

## Failing subtest

14 "fetching from another promisor remote".

Symptom: `git -C repo fetch server2` died with
`error: unexpected line: 'filter blob:none'` and `upload-pack exited with exit status: 1`.

## Root cause

`repo` is a partial clone (origin has `partialclonefilter=blob:none`). The test adds a
second promisor remote `server2` (only `remote.server2.promisor=true`, NO filter, and
`server2` has NO `uploadpack.allowFilter`). The earlier `cat-file -p $HASH2` lazily
fetched the commit and â€” correctly, matching git's lazy fetch which internally runs
`git fetch <remote> --filter=blob:none` (see `git/promisor-remote.c` fetch_objects) â€”
registered `remote.server2.partialclonefilter=blob:none`
(`register_promisor_default_filter`, git's `partial_clone_register`).

The bug was on the *send* side: the later plain `git fetch server2` inherited that
filter (`effective_fetch_filter` returns `blob:none` for a promisor remote) and sent a
`filter blob:none` line UNCONDITIONALLY. Git's `fetch-pack.c` `send_filter` only writes
the filter line when the server advertised the `filter` feature
(`server_supports_feature("fetch", "filter", 0)`); otherwise it warns
("filtering not recognized by server, ignoring") and fetches unfiltered. server2 does
not advertise `filter` (no `uploadpack.allowFilter`), so our unconditional send tripped
its `die("unexpected line: ...")`.

## Fix

grit/src/file_upload_pack_v2.rs: new `v2_fetch_supports_filter(caps)` helper (checks the
server's `fetch=` capability for `filter`).

grit/src/fetch_transport.rs:
- `fetch_via_upload_pack_skipping` (v2 path): compute `wire_filter_spec` = the requested
  filter only when `v2_fetch_supports_filter(&caps)`; otherwise warn once and send None.
  Used for all three `write_v2_fetch_request` calls (single-round + 2 multi-round).
- `fetch_upload_pack_negotiate_pack_bytes` (explicit-wants v2 path): same gating.

Left the v0/v1 path unchanged: t0410's local fetch uses protocol v2 (`repo` has
`protocol.version 2`), and threading server caps into the v0 stream helper risked
regressions for no benefit here.
