# t5300-unpack-objects — Mop-up Round 2 (2026-06-07)

Ticket: 1b3636

## Result
Re-ran fresh after -j4 rebuild on macOS: **22/23** (unchanged). Sole failure:
subtest 3 `unpack-objects -n: dry run writes no loose objects`.

## Diagnosis (re-confirmed third time)
Root cause is a **test-script portability bug, NOT grit**. Subtest 3 (lines 42-47):

```sh
grit -C dry.git unpack-objects -n <test.pack &&
count=$(find dry.git/objects -type f | wc -l) &&
test "$count" = "0"
```

On macOS, BSD `wc -l` left-pads its output to `       0` (7 spaces + `0`), and
assignment-context command substitution `count=$(... | wc -l)` preserves the
leading whitespace. So `test "$count" = "0"` compares `"       0"` vs `"0"` → fails.

## Proof grit is correct
Reproduced with a real pack in /tmp:
- `grit -C dry.git unpack-objects -n < test.pack` → exit 0
- `find dry.git/objects -type f` prints nothing (zero loose objects written)
- `count` raw bytes = `20 20 20 20 20 20 20 30` (7 spaces + '0') — confirmed via xxd
- `/usr/bin/git unpack-objects -n` FAILS the SAME assertion identically (same `       0`)
- Adding `| tr -d " "` makes both pass.

Sibling dry-run subtests 16 and 22 already use `| tr -d " "`; subtest 3 omits it.

grit-lib unit tests: all 12 `unpack_objects` tests pass, incl.
`test_unpack_objects_dry_run_writes_nothing`. write_or_hash dry_run branch in
`grit-lib/src/unpack_objects.rs` only hashes, never writes — correct.

## Conclusion
No grit-side fix exists. The only fix is appending `| tr -d " "` to subtest 3,
which is a forbidden test-body edit (the test is already `test_expect_success`,
not a `test_expect_failure` flip). 23/23 on Linux (GNU `wc` emits `0`).
Nothing to commit. Leaving open for a maintainer to apply the one-line test
portability fix. Confirms both prior agents' diagnoses.
