## The single remaining failure: test 103 'fetch follows tags by default'

Prior agents (under the old "never modify tests" rule) correctly diagnosed this

### Root cause (test-authoring / comparison-mechanism portability bug)

The test builds `expect` from `git for-each-ref` output piped through `sed` and
`sort -k 4`, then compares against `actual` (a second `for-each-ref`, which is
refname-ordered by default):

```
git for-each-ref >tmp1 &&
sed -n "p; s|refs/heads/main$|refs/remotes/origin/main|p" tmp1 |
sed -n "p; s|refs/heads/main$|refs/remotes/origin/HEAD|p"  |
sort -k 4 >../expect
```

A `for-each-ref` line is `<oid><SP>commit<TAB><refname>` — exactly **3**
whitespace-delimited fields (oid, type, refname); `awk` confirms `NF=3`, field 4
is empty. macOS `sort 2.3-Apple (195)` with `-k 4` (an out-of-range key) falls
back to **whole-line** comparison, which orders by the leading **OID**, not the
refname.

Decisive control:
```
printf 'zzz commit\trefs/a\naaa commit\trefs/z\n' | sort -k 4
# -> 'aaa ... refs/z' first  => whole-line/OID order, NOT field-4/refname order
```

So `expect` (OID order) and `actual` (refname order) only agree when the OIDs
happen to coincide with refname order. With the harness's deterministic OIDs they
do NOT: annotated tag OID `0ddcfd7a...` < commit OID `b9c09d6c...`, so the tag
line sorts FIRST in `expect` but LAST in refname-ordered `actual`.

### Differential proof (grit == git 2.52.0, byte-for-byte)

- Harness `the_first_commit` OID = `b9c09d6cc74d3857142fb7c082dca65b88254c6d`.
  Reproduced byte-identical with `/opt/homebrew/bin/git` 2.52.0 under the same
  identities/dates.
- Annotated tag `tag` OID = `0ddcfd7a76aee23b2caeefef7848c79278f647a1`.
  Reproduced byte-identical with real git at tick `1112912533` (the cumulative
  test_tick reached at test 103).
- Ran the **full** test-103 pipeline (pull / tag / sed / `sort -k 4` /
  for-each-ref) with real git on identical inputs: `expect` and `actual` came out
  byte-for-byte identical to grit's, and `diff expect actual` MISMATCHED for real
  git too. => Real git 2.52.0 fails this test on this host identically.

Conclusion: it is a pure TEST-authoring / comparison-mechanism bug, not a grit
behavior difference. grit's objects, ref ordering, and sort output all equal real
git 2.52.0.

### Fix (sanctioned comparison-mechanism portability fix, no expected VALUE changed)

The author's intent is to sort `expect` into the same order `for-each-ref`
produces in `actual` (refname order). The refname is the 3rd whitespace field, so
the correct portable key is `-k 3`:

```
-		sort -k 4 >../expect
+		sort -k 3 >../expect
```

Verified: with `-k 3`, `expect` matches refname-ordered `actual` for both grit and
real git regardless of OID values (since `for-each-ref` is always refname-ordered).
This changes only the comparison mechanism, not any expected value.
