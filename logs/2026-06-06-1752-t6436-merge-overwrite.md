## Root causes & fixes (all in grit/src/commands/merge.rs)

### Tests 11 & 12 — leading-path untracked collision on fast-forward
Merging `sub` (c0 -> sub) is a fast-forward that creates `sub/f` and `sub2/f`.
The working tree already has untracked *files* (or a symlink) named `sub`/`sub2`.
`bail_if_merge_would_overwrite_local_changes` iterated new index entries and
`fs::symlink_metadata(work_tree/"sub/f")` failed with ENOTDIR (because `sub` is a
file, not a dir), so the loop `continue`d and never detected the collision — the
FF silently clobbered `sub`/`sub2`.

Fix: added a leading-path-prefix scan at the top of the untracked-collision block.
For each untracked new entry whose path contains `/`, walk each leading prefix;
if a prefix exists on disk as an untracked *non-directory* (file or symlink),
insert that prefix into `overwrite_untracked`. Git reports the leading component
(`sub`, `sub2`), matching the test's `expect`.

### Test 14 — wrong message for untracked overwrite on unborn branch
Git implements unborn merge via `read-tree -m -u <empty> <target>`
(builtin/merge.c `read_empty` / `pull_into_void`). On an untracked collision that
2-way unpack prints the *singular* read-tree diagnostic and dies:
```
error: Untracked working tree file 'c0.c' would be overwritten by merge.
fatal: read-tree failed
```
grit's `merge_unborn` instead used the generic multi-file `git merge` overwrite
report. `read-tree -m -u` stops at the FIRST colliding path in index order.

Fix: new helper `first_unborn_untracked_collision` returns the first colliding
path (or blocking leading ancestor) in index order; `merge_unborn` emits git's
read-tree-style message and exits 128 before mutating anything. Verified order &
single-path behavior against `/usr/bin/git`.
