# t3452-history-split — ticket 4d6ac5

## Starting point
Fresh harness run: 24/25 (ticket TOML said 13 failing at last scan, but earlier
rebase-core fixes in the history-rewrite group already lifted it to 24/25).
Only one subtest failed at a time; fixing one surfaced the next.

## Root causes

### 1. Split-out commits ignored `GIT_COMMITTER_DATE`
`grit/src/commands/history.rs::resolve_committer_for_split` built the committer
line from `OffsetDateTime::now_utc()`, so split-out commits got a wall-clock
(year 2026) timestamp. Upstream `git history split` passes `NULL` committer info
to `commit_tree_extended`, letting `git_committer_info()` pick up the environment
date (the harness `test_tick` exports `GIT_COMMITTER_DATE`). With the wrong date
the split commits sorted as the newest in `git log --graph`, breaking the
expected graph in test 10 (`can split commit in unrelated branch`).

Fix: delegate to `commit::resolve_committer` (made `pub(crate)`), which already
honours `GIT_COMMITTER_DATE` and resolves committer name/email from config the
same way `git commit` does. Author date is already preserved (history.rs copies
`based_on.author` / `author_raw`), matching the C code's `original_author`.

### 2. `git log --branches` graph tie-break used object id, not ref order
After fix #1, split-out commits share the exact `GIT_COMMITTER_DATE` of a sibling
branch tip. `git log --graph --branches` then has equal-date tips and must break
the tie by *insertion order* (Git's `prio_queue` FIFO), where `--branches`
expands refs in ref-name order (`main` before `unrelated`). grit's
`sort_revision_specs_by_committer_desc` (grit/src/commands/log.rs) broke equal
dates with `a.spec.cmp(&b.spec)` — i.e. by object-id hex — so the
alphabetically-smaller OID won regardless of ref name. That put the wrong branch
in the leftmost graph column (tests 8, and would also affect 10/11).

Fix: make the sort *stable* and key only on the committer epoch (`b.0.cmp(&a.0)`),
so equal-date tips keep their ref-name input order.
