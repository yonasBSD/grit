# t4216-log-bloom — bloom-filter pathspec acceleration in path-limited log

Ticket: a81709. File: tests/t4216-log-bloom.sh. Subsystem: log-graph (thread B).

## Starting state

163/167 passing. Failing subtests:
- 133: use Bloom filters even if the latest graph does not have Bloom filters
- 135: Use Bloom filters if they exist in the latest but not all commit graphs in the chain
- 165: Bloom reader notices out-of-bounds filter offsets
- 167: Bloom reader notices out-of-order index offsets

## Root cause

All four failures trace to the same architectural gap in `grit_lib::rev_list::rev_list`.

`git log -- <path>` (no `--graph`) is handled by `run_rev_list_log` → `rev_list`. For the
default (dense, non-`--full-history`) case, `rev_list` does the real history-simplification
traversal in `walk_dense_path_limited_closure` / `dense_path_limited_action` (a TREESAME walk
that *prunes* the traversal — it follows only the TREESAME parent of a merge). The changed-path
Bloom precheck + `GIT_TRACE2_PERF` statistics were only being run later, in the `ordered.retain(...)`
re-check over the **already-reduced** output set.

Git instead consults the Bloom filter for the first-parent comparison of **every** commit its
single `limit_list` walk reaches (`rev_compare_tree` → `check_maybe_different_in_bloom_filter`),
including commits later pruned by TREESAME merge simplification, and does so **unconditionally**
(not just when trace2 stats are enabled). Consequences of grit consulting only the reduced set:

- **133**: the latest split-graph layer has no Bloom data; its 3 commits (c14/c15/c16) should each
  count `filter_not_present`. grit only counted the 2 that survived path filtering (c14 doesn't
  touch `A/B`, so it was dropped before the precheck). Expected `filter_not_present:3`.
- **135**: a side-branch commit (`side-1`) reachable only through the differing parent of a
  TREESAME merge is pruned by the dense walk, so its first-parent precheck (`definitely_not`) was
  never counted. Expected `definitely_not:10`, grit produced `9`.
- **165 / 167**: corrupt-graph integrity warnings (`ignoring out-of-range offset …`,
  `ignoring decreasing changed-path index offsets …`) are emitted as a side effect of *loading*
  the Bloom filter for the commit at the corrupted lex position. That commit (c1 at lex pos 3 for
  165; side-1 at pos 2 for 167) does not touch the queried path `A/B/file2`, so it was never in
  grit's reduced precheck set and the warning never fired (`err` was empty).

## Fix (grit-lib/src/rev_list.rs only)

1. Compute the Bloom configuration (`bloom_chain`, `read_changed`, `version`, `cwd`) once, up
   front, before the dense walk (was previously computed inside the later retain block).
2. Add `consult_dense_bloom_filters`: a pass over the **full reachable set** (`included` before
   dense limiting), looking up each commit's first-parent Bloom filter exactly once — mirroring
   Git's single `limit_list` pass. It records the `filter_not_present`/`maybe`/`definitely_not`/
   `false_positive` counters when a `BloomWalkStats` handle is present, and runs even without one
   so corrupt-graph reader warnings still fire. It never changes selection (a `definitely_not`
   only confirms what the tree diff would also conclude).
3. The later `ordered.retain(...)` re-check now passes `None` for stats when `dense_path_limited`
   (to avoid double-counting); for `--full-history`/`--simplify-merges`/`--ancestry-path` (dense
   walk skipped) it still records stats as before.
4. `dense_path_limited_action` reverted to its pure tree-diff selection logic (no Bloom).
