## Failure
Subtest 3 `indirectly clone patch_clone` failed. The test:
1. `git pull ../.git` into user_clone (base repo A-B chain) — succeeds, HEAD = B.
2. `git pull ../patch_clone/.git` — patch_clone has C (parent B), C^tree, C:text,
   B, B^tree but is MISSING B:text, A (parent of B) and A:side.

Second pull errored:
```
error: missing object 53030622... while copying from remote: object not found
```
where `53030622` is commit A — which patch_clone legitimately does not have.

## Root cause
`copy_reachable_objects_internal` (grit/src/commands/fetch.rs, the non-upload-pack
local-copy fetch path) walked the FULL reachable closure of the fetched tip (C):
C -> C^tree, C:text, parent B -> B^tree, parent A -> ERROR (A absent in patch_clone).

It never pruned the walk at objects the destination already holds. In real Git the
client's negotiation "have B" marks B (and its closure) uninteresting, so the remote
`pack-objects` never walks below B and never touches A.

A prune-at-`dst_odb.exists_local` already existed but was gated behind
`skip_missing_promisor` (promisor path only).

## Fix
Removed the `skip_missing_promisor &&` gate so the "destination already has this
object => prune the walk here" rule applies to the ordinary local copy path too.
The subsequent `check_connectivity` pass still validates the destination actually
holds the pruned closure, so a genuinely incomplete fetch (subtest 4, direct clone
of patch_clone into an empty repo) still fails as required.

One file changed: grit/src/commands/fetch.rs (one condition + comment).
