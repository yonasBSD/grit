## Status: 29/30 — remaining failure is a macOS harness/platform bug, NOT a grit defect.

## Failing subtest
- 23: `count: --count larger than total returns all`

## Root cause (definitive)
The subtest body:
```sh
git for-each-ref >all &&
total=$(wc -l <all) &&        # macOS /usr/bin/wc -l pads: total="       7" (leading spaces)
git for-each-ref --count=100 >out &&
test_line_count = "$total" out
```

grit's `for-each-ref --count=100` output is byte-for-byte correct: it returns all 7 refs,
identical to upstream `git for-each-ref --count=100` (verified side by side, same `.git`).
Both `for-each-ref` (no count) and `--count=100` emit exactly 7 lines.

The failure comes entirely from the test harness, not grit:

1. macOS `/usr/bin/wc -l` left-pads its count with spaces, so
   `total=$(wc -l <all)` captures `"       7"` (with leading spaces), not `"7"`.
   (Linux GNU `wc -l <file` emits `7` with no padding — so this test passes on Linux/CI.)

2. `tests/test-lib.sh` line 875 REDEFINES `test_line_count` (shadowing the canonical
   `tests/test-lib-functions.sh:1055` version, because test-lib.sh sources functions at
   line 102 and then redefines at line 875). The redefinition:
   ```sh
   actual=$(wc -l <"$file")
   actual=$(echo "$actual" | tr -d ' ')   # strips spaces from ACTUAL only
   if test "$actual" "$op" "$count"        # string compare; $count NOT stripped
   ```
   So it does `test "7" = "       7"` (string comparison) which is FALSE.
   Note the verbose output even reports "expected 7 lines (=), got 7" yet fails —
   the giveaway that it's a whitespace string-compare bug.

The fix that would make this pass is to also strip spaces from `$count` in
`test-lib.sh`'s `test_line_count` (line 879-880), i.e. `count=$(echo "$count" | tr -d ' ')`.
