# t5510-fetch — ticket 7ace33

Fresh run: 204/215. Failing subtests: 35, 44, 48, 56, 59, 187, 192, 204, 211, 213, 214.

## Fixes

### Bundle header / ref ordering (tests 44, 56)
`grit/src/commands/bundle.rs`:
- Prerequisite lines now record the boundary commit's oneline subject
  (`-<oid> <subject>`), matching git's `write_bundle_prerequisites`. Previously
  emitted an empty comment (`-<oid> `).
- Bundle refs are now stored in an insertion-order-preserving `BundleRefs`
  wrapper instead of a `BTreeMap`, so `git bundle create HEAD main` records refs
  in command-line order (HEAD first) and `list-heads`/parse preserve header
  order. `--all` still emits sorted refs with HEAD last (collected via a
  temporary sorted map then HEAD appended).

### Bundle fetch ref update (test 48)
`grit/src/commands/fetch.rs`:
- `git fetch ../bundle main:main` was unbundling objects but not updating the
  local ref, because the CLI refspec source `main` was compared verbatim against
  the full bundle ref name `refs/heads/main`. Added
  `resolve_bundle_fetch_ref_updates`, which DWIMs each refspec source against the
  bundle ref list (reusing `resolve_advertised_ref_for_fetch_src`) and matches
  wildcard refspecs directly.

## Reference lock detection (tests 35, 204, 211, 213, 214)
`grit-lib/src/refs.rs` `write_ref`: the loose-ref lock file was written with
`fs::write`, which silently clobbers an existing `<ref>.lock`. Switched to
`OpenOptions::create_new(true)` so a pre-existing lock surfaces `AlreadyExists`,
matching git's `lock_ref_oid_basic`. Fetch already translates that into
`error: cannot lock ref '<ref>': Unable to create '<lock>': File exists.` and
continues updating the other refs (non-atomic), so FETCH_HEAD / upstream config
/ HEAD updates still happen while the overall fetch exits non-zero.

## Explicit pull tracking update (test 59)
`grit/src/commands/pull.rs`: the local-path pull shortcut with an explicit
refspec wrote FETCH_HEAD but never refreshed the configured remote-tracking ref.
Added `update_opportunistic_tracking_ref`, which maps the fetched branch through
`remote.<name>.fetch` (reusing `fetch::remote_fetch_refspecs` /
`map_ref_through_refspecs`) and writes `refs/remotes/<remote>/<branch>`.

## Auto-gc after fetch (test 192)
`grit/src/commands/fetch.rs` `maybe_run_auto_maintenance_after_fetch`: fetch
spawned `maintenance run --auto --detach` with the child's stdout/stderr sent to
/dev/null, so the gc task's "Auto packing the repository" notice never reached
the caller. Git's `gc --auto` always prints that line (foreground) when
`gc.autoDetach` is false. Now fetch runs maintenance in the foreground
(`--no-detach`) when `maintenance.autoDetach` / `gc.autoDetach` is explicitly
false, so the message is captured. (autoDetach=true still detaches as before, so
t5537's rm-race and normal non-blocking behavior are unchanged — verified 16/16.)

## Regression fix: bundle verify prerequisite display
Recording the prerequisite subject in the bundle header (test 44) also made
`git bundle verify` print the subject after each prerequisite oid, which broke
t6020 (6 subtests: it expects `<oid> ` with an empty name). Git stores
prerequisites with an empty display name (`bundle.c` keeps the subject only in
`->util` for connectivity diagnostics), so `print_bundle_verify_info` now prints
`<oid> ` with an empty comment. t6020 restored to 37/37; t5510 test 44 (which
checks the raw header bytes, not the verify output) still passes.
