## Root cause for 10–13 (FIXED)
All four failed with `fatal: failed to apply delta` / `error: remote unpack
failed: unpack-objects abnormal exit`.

The smart-HTTP push server in this harness is **real git** (lib-httpd.sh routes
`git-receive-pack` to `$_real_exec_path/git-receive-pack`, and test-httpd spawns
the real `git-http-backend` found via GIT_EXEC_PATH). So real git's
`unpack-objects --strict` was rejecting the thin pack grit's `pack-objects
--thin --stdout` produced.

Captured the exact push packs (temporary GRIT_DUMP_PUSH_PACK hook in
`build_thin_push_pack_from_have_set`) and confirmed real `git unpack-objects
--strict` rejects them while grit's lenient unpack accepts them.

Bad object: a REF_DELTA for the **empty blob** (`e69de29b…`) encoded against the
7-byte blob `01f9a2aac3` (`commit\n`), delta bytes `0700` = `src_size=7,
dst_size=0, no ops`. Reproduced standalone: real git rejects this degenerate
delta with "failed to apply delta" even when the base is present.

Cause in grit: `optimize_blob_deltas` in `grit/src/commands/pack_objects.rs`
selects a delta base via the prefix heuristic
`b.data.starts_with(&t.data) && b.data.len() > t.data.len()`. For the empty
target blob `t.data` is empty, so EVERY non-empty base "starts with" it and the
empty blob gets deltified against the smallest blob — producing the degenerate
delta. Git never deltifies a zero-length object.

## Fix
`grit/src/commands/pack_objects.rs`, `optimize_blob_deltas`: skip empty target
blobs (`if t.data.is_empty() { continue; }`) before the base-selection loop, so
the empty blob is always emitted as a full object.

Verified: regenerated the thin pack with the fixed binary → 0 ref_deltas, and
`git unpack-objects --strict` (real git 2.52) accepts it (rc=0).

Full suite after fix: **20/21** (tests 10,11,12,13 now pass).

## Test 21 (FIXED)
`clone/fetch scrubs password from reflogs`. The clone itself was never the
problem — it succeeds. The failure was the final assertion
`! grep "$HTTPD_URL_USER_PASS" reflog`: the test fetches
`$HTTPD_URL_USER_PASS/smart/test_repo.git +main:main` (URL with
`user%40host:pass%40host@` userinfo), and grit's reflog message embedded the
full URL including credentials:

    fetch http://user%40host:pass%40host@127.0.0.1:PORT/smart/test_repo.git +main:main: forced-update

Real git anonymizes the URL (`transport_anonymize_url`) so credentials never
reach the reflog. The lib-httpd wrapper routes the clone to real git (`_http=1`
for any http:// arg) but routes `fetch +main:main` to grit (`_http=0` for a
non-glob fetch refspec), so grit's reflog writer is what leaked.

Fix: `grit/src/commands/fetch.rs` — scrub URL userinfo via
`crate::http_client::scrub_url_credentials` in:
- `fetch_reflog_action_prefix` (the `fetch <remote> <refspecs>` action prefix
  used by `write_fetch_explicit_reflog`), and
- `append_fetch_reflog` (the `--append --prune` reflog path).

A named remote (e.g. `origin`) is not a URL so `scrub_url_credentials` (which
parses with `url::Url` and clears username/password) leaves it unchanged.

Verified reflog now reads `fetch http://127.0.0.1:PORT/smart/test_repo.git ...`
with the password gone.
