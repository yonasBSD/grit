# t6402-merge-rename — ticket 3a7bf9

Date: 2026-06-06T12:53Z

## Goal
Make subtest 36 "avoid unnecessary update, modify/delete" pass. File was 45/46.

## Root cause
For a modify/delete conflict (HEAD modifies `file`, side deletes it), the merge
correctly fails and leaves HEAD's version in the tree. The test sets `file`'s mtime
to 3600s in the past, runs the failing merge, and asserts the mtime is unchanged
("file should have stayed intact") — Git's twoway_merge avoids rewriting a file
whose content is already correct.

grit was rewriting `file` during the merge in two ways:
1. `remove_deleted_files` deleted `file` because it only survives at conflicted
   (non-zero) index stages (stage 1 base + stage 2 ours), so it was absent from the
   stage-0 `new_paths` set and got removed.
2. The conflict-file write pass then re-created `file` unconditionally via
   `fs::write`, giving it a fresh mtime.

## Fix (grit/src/commands/merge.rs)
- `remove_deleted_files`: do not delete a path that survives the merge only as a
  conflicted (non-zero stage) entry at that exact path (and has no children needing
  the slot, and is not a tree/gitlink). Git keeps those working files.
- Conflict-file write loop (both the two-head path ~line 2793 and the octopus path
  ~line 5285): added an "avoid unnecessary update" guard `worktree_file_matches`
  that skips `fs::write` when the on-disk regular file already holds identical bytes,
  preserving the mtime.
- New helper `worktree_file_matches(path, expected)` — true only for an existing
  regular file (symlinks excluded) whose bytes equal `expected`.

## Regression caught & fixed (t6416 symlink modify/modify, add/add)
Skipping removal of conflicted paths left a conflicted **symlink** in place. The
conflict-file write loop only handled `is_dir`, not symlinks, so `fs::write` followed
the symlink and clobbered its target, creating a spurious untracked file (e.g.
`fickle -> disneyland` produced an untracked `disneyland`). Fixed by unlinking an
existing symlink before writing the conflict content as a regular file, in both
write loops.
