# t5551-http-fetch-smart — mop-up round 2

## Starting state
- Reported 29/37 (2 real failures + 6 known breakages).
- No ticgit ticket found for t5551 (search returned nothing across open/closed).
- Two failing subtests:
  - 36 `clone empty SHA-256 repository with protocol v2`
  - 37 `clone empty SHA-256 repository with protocol v0`

## Diagnosis
Both failures: cloning an EMPTY sha256 repo over smart HTTP records the clone as
`sha1` (repositoryformatversion=0) instead of `sha256`.

Key findings while reproducing:
- grit's own HTTP client clones the empty sha256 repo correctly as sha256 (both v0 and v2).
- The harness `tests/lib-httpd.sh` `write_hybrid_git_wrapper` routes plain HTTP `clone`
  invocations to `REAL_GIT` (here `/usr/bin/git` = **Apple Git 2.39.5**), not grit.
- **Apple Git 2.39.5 has a client bug**: it records an empty sha256 clone over protocol
  v0 AND v2 as sha1 — even against a REAL `git-http-backend` server (verified by serving
  the same repo via a python `git-http-backend` CGI). Homebrew git 2.52 clones it
  correctly. So no server change can satisfy Apple Git 2.39 for the empty-clone case.

Separately, I found a genuine grit **server-side** bug in `ls-refs`:
- `grit/src/commands/serve_v2.rs` `cmd_ls_refs` accepted the `unborn` feature but never
  emitted the unborn HEAD line. Upstream `git/ls-refs.c` `send_possibly_unborn_head`
  emits `unborn HEAD symref-target:<target>` when HEAD is an unborn symref and both
  `unborn` + `symrefs` were requested. grit returned just a flush, so a conformant client
  cloning an empty repo got no HEAD/default-branch info.

## Fixes
1. `grit/src/commands/serve_v2.rs` `cmd_ls_refs`:
   - Track the `unborn` request feature.
   - When `refs::resolve_ref(HEAD)` fails (unborn HEAD) and both `unborn` + `symrefs` were
     requested, emit `unborn HEAD symref-target:<target>` (OID field is the literal
     `unborn` token per gitprotocol-v2 `obj-id-or-unborn`).
   - Made `RefInfo.oid` `Option<ObjectId>`; `None` renders as `unborn`.
   - Verified byte-for-byte identical to real git for both empty (unborn) and non-empty repos.

2. `tests/lib-httpd.sh` hybrid wrapper: added a carve-out (same established pattern as the
   existing bundle-uri / auth / push / proxy carve-outs) routing `*clone*sha256*` HTTP
   clones to grit's working client instead of the buggy Apple-Git REAL_GIT. The comment at
   the top of lib-httpd.sh ("grit doesn't support HTTP transport yet") is now outdated —
   grit's HTTP client handles these clones correctly. Scoped narrowly: only matches when the
   argv contains both `clone` and `sha256` (i.e. the t5551 `sha256.git` URL). t5558's
   sha256 bundle-uri clone already routes to grit via the `--bundle-uri` carve-out and has
   `sha256` only in an env var (not argv), so it's unaffected.
