## Fix for test 23 (MIDX hash-version threading)

The test writes a MIDX in a `--object-format=sha256` repo and a sha1 repo, swaps
the two MIDX files, runs `git log -1`, and expects:
`multi-pack-index hash version 2 does not match version 1` (and the symmetric form).

Upstream (`git/midx.c:157-160`, `git/chunk-format.c oid_version`): the MIDX header
byte 5 stores `oid_version(repo->hash_algo)` (SHA-1 → 1, SHA-256 → 2); the reader
errors and skips the MIDX when that byte differs from the repository's own
`oid_version`.

Prior agents concluded this needed a full SHA-256 object model. That is true for
reading sha256 *objects*, but **test 23 never reads a MIDX object** — `git log -1`
reads HEAD (a loose/packed commit) and the MIDX warning is a pure header-byte
check that fires before any OID lookup. So the test is satisfiable by threading
the repo's configured object format into MIDX write/read, even though grit's
sha256 repos still store SHA-1-length OIDs internally.

Changes, all in `grit-lib/src/midx.rs`:

- Added `HASH_VERSION_SHA256 = 2` and two helpers
  `repo_midx_hash_version(pack_dir)` / `repo_midx_hash_version_for_objects_dir(objects_dir)`
  that read `<gitdir>/config` `[extensions] objectformat = sha256` (minimal,
  case-insensitive section/key scan) and return 2 for sha256, else 1. grit already
  records `extensions.objectformat = sha256` in config for sha256-init repos
  (verified), so the gitdir config is authoritative.
- Writer: `build_midx_bytes_filtered` now takes a `hash_version` arg and emits it
  as header byte 5 (was hard-coded `HASH_VERSION_SHA1`). Both call sites
  (`write_multi_pack_index_with_options`, the compaction writer) pass
  `repo_midx_hash_version(pack_dir)`.
- Reader: `midx_load_for_read` now takes `expected_hash_version` and compares the
  on-disk byte against it (was hard-coded `HASH_VERSION_SHA1`), emitting the exact
  `error: multi-pack-index hash version {on-disk} does not match version {expected}`
  and skipping the MIDX. All three callers (`midx_oid_listed_in_tip`,
  `validate_midx_referenced_packs`, `try_read_object_via_midx`) derive the expected
  version from their `objects_dir`. `verify_midx` updated the same way for
  consistency (it would otherwise reject a valid sha256 MIDX).

Because the swapped MIDX always mismatches the host repo's version, the reader
warns and skips before touching the (SHA-1-length) OID table, so the SHA-1-vs-SHA-256
OID-length discrepancy never surfaces in this test.

## Remaining 4 (unchanged, deep/out-of-scope — see ticket comments 4 & 5)

- **77 / 78 / 81 — expire/repack batch.** `select_packs_for_batch`
  (`grit/src/commands/multi_pack_index.rs:560`) already mirrors
  `fill_included_packs_batch` exactly and per-pack MIDX counts match git
  (A=24,B=15,C=9,D=6,E=7). Divergence is purely grit's compressed pack *byte sizes*
  (A=5367,B=2565,C=1252,D=690,E=4646) vs git's: `BATCH_SIZE = THIRD_SMALLEST+1 = 2566`

  expire deletes it; git's exact bytes keep B. Needs byte-identical pack-objects
  output (separate large effort). 81 cascades from 77/78 state.
