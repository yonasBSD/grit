## Root cause

Test 3 does:
  git rebase --onto main HEAD^      # conflicts
  git read-tree --reset -u HEAD     # resolve conflict to a no-op (worktree == HEAD)
  git rebase --continue             # should finish + remove .git/MERGE_MSG

In `do_continue` (grit/src/commands/rebase.rs), the branch
`if stopped_oid.is_some() && worktree_matches_head(...)` handles the "resolved to
nothing" case. The non-obstructed `else` sub-path recorded the rewrite and removed
MERGE_MSG/current/REBASE_HEAD, then called `replay_remaining` — but it did NOT pop
the just-completed `pick` line from the todo.

The conflict stop wrote the todo as `todo[i..]`, so the resolved commit is still
`todo[0]` with `msgnum=1`. `replay_remaining` iterates `for i in (msgnum-1)..todo.len()`,
i.e. from index 0, re-applying the SAME pick and re-conflicting — leaving MERGE_MSG.

The normal successful-commit continuation path (a few hundred lines later) already
pops via `pop_first_nonempty_todo_line` with a comment "otherwise replay_remaining
would re-apply it". The empty no-op path was missing the equivalent pop.

## Fix

In the empty no-op `else` path: compute `todo_still_starts_with_current` (first todo
step's oid == current_oid or stopped_oid), use it for the rewrite flush peek index,
and `pop_first_nonempty_todo_line` before `replay_remaining` when true. Mirrors the
existing successful-commit path.
