# t5620-backfill: 4/10 -> 10/10

Branch: wf/p2/t5620-backfill
Base: 7a7844cb7

## Summary

Fixed `git backfill` on partial clones. Three root causes, all in grit Rust
(no test edits, maintenance.rs untouched).

## Changes

### 1. clone.rs — materialize partial-clone skeleton as loose objects
`materialize_blob_none_partial_layout` and `materialize_tree_zero_partial_layout`
re-wrote the reachable commit/tree skeleton with `Odb::write`, then deleted all
local packs. `Odb::write` short-circuits on `exists()`, which was true because the
object was still in the just-copied pack — so no loose file was written. After the
packs were deleted the skeleton objects existed nowhere and
`write_partial_clone_promisor_pack` failed with
`missing object in non-promisor repository`. Switched both call sites to
`Odb::write_loose_materialize` (checks only the loose path, writes
unconditionally). Fixes subtests 3, 4, 6, 7, 8.

### 2. clone.rs — race-tolerant copy_objects
`copy_objects` / `copy_dir_contents` aborted the whole clone when a source pack
or fan-out dir vanished mid-copy. A detached `git maintenance` geometric repack
fires after the test setup's commits and can delete a source pack while the bare
clone in subtest 2 is copying it (`copying objects: No such file or directory`).
Added `copy_skip_vanished_source`: treat `ErrorKind::NotFound` on the source
file/dir as a skip. Also handle a vanished inner fan-out dir in `copy_dir_contents`.
Fixes subtest 2. Verified stable across 11 t5620 runs.

NOTE: The diagnosis suggested raising `maintenance.loose-objects.auto` /
`geometric-repack.auto` from 100 to 6700. That is WRONG: git's
`loose_object_auto_condition` (git/builtin/gc.c:1318) defaults to 100 with an
exact `count >= limit` (no 256 rounding), and changing it regressed
t7900-maintenance by ~7 subtests. The defensive copy fix alone removes the race,
so maintenance.rs was left untouched.

### 3. checkout.rs + promisor_hydrate.rs — sparse-aware checkout hydration
`switch_to_tree` hydrated *every* missing blob in the target tree from the
promisor, ignoring an active sparse-checkout, so `checkout main` in a sparse
partial clone fetched all blobs instead of just the matching ones (subtests 7/8
expected 6 and 18 fetched, got 24/24). Added
`hydrate_sparse_tree_blobs_from_promisor` (reuses the existing
`collect_sparse_missing_blobs_from_tree`, including non-cone mode) and a
`sparse_checkout_patterns_for_hydration` helper that loads patterns + cone mode
like `backfill --sparse`. When sparse-checkout is enabled, checkout now hydrates
only the sparse working set.
