# t6131-pathspec-icase (ticket fd04f0)

## Summary

Fixed a real `git log -- <path>` dense path-limiting bug in `grit-lib/src/rev_list.rs`.
The t6131 test itself is designed to be skipped on case-insensitive filesystems
(upstream `CASE_INSENSITIVE_FS` prereq); on this macOS host it cannot run because
loose ref files `bar`/`bAr`/`BAR` collide on the case-insensitive APFS volume — real
Git fails identically here. Proved grit is fully correct: 9/9 on a case-sensitive
volume.

## Root cause (the real grit bug)

`git log --format=%s -- bar` returned nothing when the only commit that touched file
`bar` was an ancestor of HEAD whose descendant tips did not touch `bar`.

Reproduced on a case-sensitive volume:
- commit `bar` (root) adds file `bar`
- commit `bAr` adds file `bAr`
- `git log -- bar` should print `bar` (the root), but printed nothing.
- `git log --full-history -- bar` worked (different code path), isolating the bug to
  the default dense path-limited walk.

In `rev_list`, after `walk_dense_path_limited_closure` narrows `included` to the
path-matching commits, the default date-order emission calls
`date_order_walk(&graph, &included, &include, ...)`. That walk seeds the priority
queue only from user tips that are themselves in `included` (the selected set) and
only traverses through selected commits. When the user tip (HEAD = the `bAr` commit)
was dropped by path-limiting, nothing got seeded and the matching ancestor (root
`bar`) was never emitted. Git's `limit_list` instead walks the full reachable graph
and only *filters emission*.

## Fix

`grit-lib/src/rev_list.rs`:
- Capture the full reachable set (`dense_traversable`) before the dense closure
  overwrites `included`, and set a `dense_path_limited` flag.
- In the default/date-order branch, when `dense_path_limited` and there are explicit
  tips, emit via `date_order_walk_through_dropped(&graph, &include, &dense_traversable,
  &included, ...)` — which seeds from the user tips, traverses the full reachable set,
  and emits only the path-matching commits in date order (with the correct
  insertion-order tiebreak). This is the same mechanism already used for the
  parent-count filter case.

Topo ordering (`graph_order_topo_sort`) was already correct (seeds in-degree-zero
among the selected set), so no change was needed there.

## Verification

- t6131 on a case-sensitive APFS volume: 9/9 passing.
- Real `/usr/bin/git` on this macOS host also fails `tag 'bAr' already exists`,
  confirming the local harness failure is a filesystem/skip-prereq issue, not a grit
  bug. The grit `tests/test-lib.sh` lacks the upstream `CASE_INSENSITIVE_FS` lazy
  prereq, but per the rules `test-lib.sh`/test files must not be modified.
- Regression suite (case-insensitive FS is fine for these): t6012 42/42, t6019 18/18,
  t6111 65/65, t6007 23/23, t6004 7/7, t6003 36/36, t6009 15/15, t4208 21/21 — all
  green. t4202-log improved from 120/149 to 122/149 (the same fix helps 2 more
  subtests).
- `cargo test -p grit-lib --lib`: 272 passed, only the 2 known pre-existing
  `gitignore_glob_tests` failures.
