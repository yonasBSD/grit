## Verification of each remaining failure

### 23 — warn on improper hash version  (needs full SHA-256 object model)

The test writes a MIDX in a `--object-format=sha256` repo and a sha1 repo, swaps them,
and expects `git log` to print `multi-pack-index hash version 2 does not match version 1`
(and the symmetric message).

Root blockers in grit:
- `grit-lib/src/objects.rs:29` — `pub struct ObjectId([u8; 20])`. The object model is
  hard-wired to 20-byte SHA-1. In a `--object-format=sha256` repo grit still emits
  **40-char** object ids (confirmed: `git rev-parse HEAD` in the sha256 repo returns a
  40-hex id, not 64-hex). So grit's "sha256" repo is not actually sha256.
- `grit-lib/src/midx.rs:463` — the MIDX writer does `if e.oid.len() != 20 { continue; }`,
  i.e. it **skips every 32-byte OID**, producing a structurally empty sha256 MIDX
  (OIDL/OOFF chunks are zero-length; inspected the bytes).
- `grit-lib/src/midx.rs:666` — the writer always emits `HASH_VERSION_SHA1` (byte 5 = 1)
  regardless of object format, so the swapped MIDX never has byte5=2 and the mismatch is
  never detected.
- `grit-lib/src/midx.rs:1701` and `:1709` — `midx_load_for_read` hard-codes the comparison
  against `HASH_VERSION_SHA1` and `hash_len = 20`. The reader entry points
  (`midx_oid_listed_in_tip`, `read_midx_objects`, etc.) take only `objects_dir: &Path`
  and have no access to the repository's hash format, so they cannot emit
  `does not match version 2` in the sha256 direction.

A correct fix requires making `ObjectId` hash-agnostic and threading the repo hash algo
through MIDX read+write (and the Sha1 trailer). That is a cross-cutting change far larger
than the MIDX ticket. A header-only hack would be incoherent because grit's sha256 repos
aren't real sha256.

### 77 / 78 / 81 — expire/repack batch  (needs byte-identical pack-objects output)

grit's batch-selection logic (`grit/src/commands/multi_pack_index.rs:560`
`select_packs_for_batch`) already mirrors `fill_included_packs_batch` in
`git/midx-write.c:2001` exactly (mtime order, `expected_size = referenced<<14 / num_obj *
pack_size`, `(x + 1<<13) >> 14`, the `expected_size >= batch_size` and `total_size <
batch_size` guards). Per-pack MIDX reference counts match git (A=24, B=15, C=9, D=6, E=7;
A/B share 0 objects).

The divergence is purely **pack byte sizes**. In the harness run the `dup` packs are
A=5367, B=2565, C=1252, D=690, E=4646. `BATCH_SIZE` in tests 73/74 is
`THIRD_SMALLEST + 1` (= 2566, i.e. pack-B's own size + 1). The batch walks packs in mtime
order D, C, B (after the test-72 chmtimes) accumulating `total_size`:
690 + 1252 = 1942 < 2566, so the loop reaches pack-B, whose `expected_size` (= 2565) is
`< 2566` → grit includes pack-B in the combined pack. Pack-B's 15 unique objects then
move to the new pack, its MIDX count drops to 0, and `multi-pack-index expire` deletes it
(test 77 expects it kept among the top-4-by-size). git keeps pack-B because git's exact
pack byte sizes make `D + C >= B + 1` (loop stops before B) or B's `expected_size >=
batch_size`. This is governed entirely by compressed pack bytes, which differ between
grit and git pack-objects (delta/zlib/object-order differences). Tests 78 and 81 cascade
from the same attribution.

Making grit pack-objects byte-identical to git is a large independent effort.

### 98 — pack.preferBitmapTips interprets patterns as hierarchy  (needs EWAH commit bitmaps)

Requires real pack bitmap commit-selection (which commits receive an EWAH bitmap),
`pack.preferBitmapTips` hierarchy semantics (full refname treated as `refname/`; parent
namespace matches), and a `test-tool bitmap list-commits` that reads the real EWAH commit
table. grit currently writes a placeholder empty `.bitmap`; `run_test_tool_bitmap`
(`grit/src/main.rs:846`) is a heuristic stub that lists all commits and omits only the
last one. Implementing the bitmap commit-selection subsystem is well outside MIDX core.
