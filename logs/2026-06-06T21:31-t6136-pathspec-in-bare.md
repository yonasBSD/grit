## Root cause
The test runs `git log -- ..` and `git ls-files -- ..` from inside a bare repo
root and from inside a `.git` directory. In both cases there is no work tree.

- `ls-files` already had the correct behavior: when `repo.work_tree` is `None`
  it checks `pathspec_escapes_repo` and bails with
  `pathspec '..' is outside repository` (matches upstream
  `prefix_path_gently` → die "'%s' is outside repository").
- `git log` did NOT: `validate_pathspec_scope` in
  `grit/src/commands/log.rs` returned `Ok(())` unconditionally when there was
  no work tree, so `git log -- ..` silently succeeded (exit 0, empty output),
  failing `test_must_fail` and `test_grep "outside repository"`.

## Fix
`grit/src/commands/log.rs`, `validate_pathspec_scope`:
In the no-work-tree branch (bare repo / running inside `.git`), iterate the
non-magic pathspecs and reject any that lexically escape the repository (leading
`..` above the root, or absolute/rooted path) with
`pathspec '<spec>' is outside repository`. Normal pathspecs (e.g. `file1.t`)
still pass through so they keep limiting history (t0410's
`git -C partial.git log --follow -- new-file.txt` is unaffected).

Added a small helper `pathspec_escapes_repo_lexically` mirroring ls-files'
`pathspec_escapes_repo`.
