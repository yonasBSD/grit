# t2501-cwd-empty

Ticket: bff6f6 (test, t2)

## Starting state
Fresh run showed 23/24 passing (status TOML claimed 3 failing — stale; earlier
tickets in the worktree group fixed shared cwd machinery). The single remaining
failure:

- not ok 8 - merge fails if cwd needs to be removed

## Root cause
`test_required_dir_removal git merge fd_conflict` checks out `df_conflict` (where
`dirORfile/` is a tracked directory containing `dirORfile/foo.t`), cd's into
`dirORfile`, and runs `git merge fd_conflict` (where `dirORfile` becomes a *file*).
This is a **conflict-free** ort merge ("Merge made by the 'ort' strategy") that
replaces the cwd directory with a file.

In `do_real_merge` (grit/src/commands/merge.rs), `preflight_merge_worktree_for_cwd`
— which refuses (`ERROR_CWD_IN_THE_WAY`) *before* mutating the work tree and is
the only path that restores the index/worktree on failure — was gated on
`merge_result.has_conflicts`. For a clean merge the preflight was skipped, so
`remove_deleted_files` deleted `dirORfile/foo.t` first; the cwd refusal then fired
later (from inside remove/checkout) with no rollback, leaving the work tree dirty.
`git diff --exit-code HEAD` therefore failed (file gone), and `dirORfile` was empty.

## Fix
Run the preflight whenever `exit_on_merge_conflict && !trial_for_multi_strategy`
(dropped the `has_conflicts` requirement). On failure it restores from
`pre_merge_index_snapshot` and returns the refusal, before any worktree mutation.

The preflight is a no-op when the cwd is the work-tree root
(`process_cwd_repo_relative` returns None), so clean merges run from the repo root
are unaffected — only merges run from inside a directory the result would remove
now refuse cleanly.
