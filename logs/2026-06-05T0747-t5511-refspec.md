# t5511-refspec — refspec parsing/validation

Ticket: 44722f — tests/t5511-refspec.sh
Subsystem group: refspec-core (thread B)

## Starting state
Fresh run: 24/47 passing. The 23 failures were all "invalid" refspecs that
grit accepted (and ran `ls-remote` successfully) instead of rejecting.

## Root cause
`grit ls-remote <remote>` resolved the remote name to its URL but never
parsed/validated the configured `remote.<name>.fetch` / `remote.<name>.push`
refspecs. Upstream Git loads the remote definition first (via
`remote_get` -> `refspec_append`), which parses every refspec and dies with
`fatal: invalid refspec '<spec>'` on the first malformed one. grit had no
refspec-parsing machinery at all.

## Fix
1. New shared module `grit-lib/src/refspec.rs` — a faithful port of
   `parse_refspec()` from `git/refspec.c`. Handles the `+`/`^` modifiers, the
   special `:`/`+:` matching push refspec, glob (`*`) balance rules between LHS
   and RHS, exact-SHA1 LHS, and fetch-vs-push validity differences. Refname
   validation reuses the existing `check_ref_format::check_refname_format` with
   `allow_onelevel` (+ `refspec_pattern` for globs), matching Git's
   `REFNAME_ALLOW_ONELEVEL | REFNAME_REFSPEC_PATTERN` flags.
   Public API: `parse_fetch_refspec`, `parse_push_refspec`,
   `valid_fetch_refspec`, `valid_push_refspec`, plus `RefspecItem`.
2. Registered `pub mod refspec;` in `grit-lib/src/lib.rs`.
3. `grit/src/commands/ls_remote.rs`: when the repository argument resolves to a
   configured remote name, validate all of that remote's fetch and push
   refspecs up front (`validate_remote_refspecs`) and `bail!` with
   `invalid refspec '<spec>'` on the first invalid one — mirroring Git's early
   die during remote loading.

This is shared machinery: the same module/validation should serve the
fetch/push/clone tickets later in this thread.
