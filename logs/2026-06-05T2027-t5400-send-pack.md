# t5400-send-pack — ticket 8427c3

Goal: make `tests/t5400-send-pack.sh` fully pass. Started at 11/17 (fresh run), reached 17/17.

## Failing subtests and root causes

### 6 / 9 — push can be used to delete a ref / override denyDeletes with `git -c receive-pack`
`grit/src/commands/send_pack.rs` had a naive `parse_refspec` that stripped `+`, ignored
deletions, and resolved the (empty) source of `:extra` as `refs/heads/` → "src ref '' does not
match any". Rewrote refspec handling to use the library `grit_lib::refspec::parse_push_refspec`,
which yields `force`, `pattern`, `src`, `dst`. Empty src + dst now means deletion (`new_oid = None`,
null OID on the wire); a deletion-only push streams an empty packfile (`empty_pack_bytes`).

Two more bugs surfaced once deletions worked:
- `receive_pack.rs` `write_status_lines` skipped deletions in the report-status stream. Git's
  `report()` (builtin/receive-pack.c) emits `ok`/`ng` for *every* command including deletes;
  without it, send-pack could not detect a rejected deletion (denyDeletes). Fixed to report all.
- `git -c receive.denyDeletes=false receive-pack` was ignored: receive-pack built its config from
  the repo file only. Now it also merges `GIT_CONFIG_PARAMETERS` (the `git -c` payload) on top.
  To keep subtest 8 passing (client-side `git -c send-pack` must NOT leak to the server),
  `spawn_receive_pack`'s `apply_proto` now strips `GIT_CONFIG_PARAMETERS` from the receive-pack
  child env; a custom `--receive-pack="git -c ... receive-pack"` re-establishes it when the spawned
  git re-parses its own `-c`.

### 14 / 15 — pushing explicit / wildcard refspecs respects forcing
`+refs/heads/main:refs/heads/main` only forced via the global `--force`; the `+` was dropped.
Now per-spec `force` (`spec.force || global`) gates the non-fast-forward check. Wildcards
(`refs/heads/*:refs/heads/*`) were unsupported (`ref not found: refs/heads/*`); added
`expand_wildcard_refspec` matching local refs against the source pattern and substituting the
matched segment into the destination. Concrete srcs now resolve via `resolve_revision` (handles
`main^` etc.).

### 13 — receive-pack runs auto-gc in remote repo
send-pack shells out to `/usr/bin/git pack-objects`, which trips on grit's v2 multi-pack-index
("multi-pack-index version 2 not recognized"). Refactored pack generation into `run_pack_objects`
and made send-pack try the system git first, then fall back to grit's own pack-objects (which
reads its v2 MIDX) when the system git fails. Robust against MIDX-presence timing.

### 17 — receive-pack de-dupes .have lines
Two fixes:
- `advertise_refs_phase` advertised a `HEAD` line (git receive-pack does NOT — it iterates only
  the ref store) and de-duplicated local refs by OID. Rewrote to match `show_ref_cb`: every local
  ref is advertised under its (namespace-stripped) name regardless of duplicate OIDs; only
  out-of-namespace refs and alternate refs collapse into de-duplicated `.have` lines.
- The push (delegated to send_pack for local `--receive-pack`) fed `^<oid>` boundaries from the
  remote `.have` advertisement (e.g. a `clone -s` alternate's objects) that the source repo does
  not have → grit pack-objects aborted with "bad tree object". Now only locally-present objects
  are fed as negatives (`repo.odb.read(oid).is_ok()`), matching git ignoring unavailable haves.
  Also pinned `build_thin_push_pack_from_have_set` pack-objects to an explicit absolute `GIT_DIR`.
