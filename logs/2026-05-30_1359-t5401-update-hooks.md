# t5401-update-hooks — per-ref receive-pack rejection

Result: 8/13 -> 13/13 (green)

## Problem

`grit receive-pack` treated a push as one all-or-nothing transaction. When the
`update` hook declined a single ref (the `tofail` ref in the test), the per-ref
loop in `run_hooks_and_update_refs` (grit/src/commands/receive_pack.rs) called
`bail!("update hook declined the update")`, aborting the entire push. As a
result:

- post-receive / post-update hooks never ran (subtests 4, 7, 8)
- the report-status stream said `ok refs/heads/tofail` because the status was
  written BEFORE hooks ran, so send.out was non-empty and `tofail` reported ok
  instead of `ng ... hook declined` (subtest 11)
- stderr showed `error: update hook declined the update` +
  `error: receive-pack failed` instead of the per-ref
  `error: hook declined to update refs/heads/tofail` plus the post-* hook output
  (subtest 12)

This mirrors the divergence from git/builtin/receive-pack.c:1614-1618, where a
declined ref is marked rejected (`goto out` for THAT command only) and the push
continues; post-receive/post-update then run with only the accepted refs.

## Changes (grit/src/commands/receive_pack.rs, grit/src/commands/send_pack.rs)

1. Per-ref rejection: each ref's outcome is recorded instead of bailing. An
   update-hook decline emits `error: hook declined to update {refname}`, records
   the ref as rejected with reason "hook declined", and continues. Policy
   declines (deny-deletes / non-fast-forward / deny-current-branch) reject only
   the offending ref. The function returns Ok(()) (exit 0) when refs were
   rejected per-ref, so send-pack does not print "receive-pack failed".

2. Report after hooks: report-status-v2 is written AFTER hooks/ref-updates run,
   using each ref's real outcome — `ok {refname}` for committed refs,
   `ng {refname} hook declined` (or the policy reason) for rejected ones.

3. post-receive stdin and post-update args are built from ONLY the
   successfully-updated refs; pre-receive stdin and the update hook still see
   ALL proposed refs (keeps subtests 5 and 6 passing). post-receive/post-update
   are skipped entirely when no ref succeeded.

4. side-band-64k: receive-pack advertises and (when negotiated) multiplexes
   hook/diagnostic output on band 2 and the report on band 1; send-pack
   demultiplexes band 2/3 to stderr with a `remote: ` prefix and consumes the
   report stream without echoing it to stdout.

5. Final refinement (this run): traverse/unpack failure diagnostics are written
   to the receive-pack process's real stderr rather than band 2, matching
   `git receive-pack` (these originate from the unpack subprocess and are not
   relayed over side-band). Only the report-status stays band-1 framed.
