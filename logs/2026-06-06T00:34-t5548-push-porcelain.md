# t5548-push-porcelain — push --porcelain output

Ticket: f40b58 — tests/t5548-push-porcelain.sh
Result: 25/25 fully passing (was 5/25).

## Problem

`git push --porcelain` output was wrong in both the in-process local (file) push
path and the smart-HTTP push path in `grit/src/commands/push.rs`. Grit emitted a
human-ish per-ref line (`{flag}\t{src}:{dst}\t{old}..{new}\t{src} -> {dst}`) that
diverged from Git's machine-readable format on several axes:

- Missing the `To <url>` header and the trailing `Done` line.
- Summary column used `old..new` quickrefs everywhere instead of the bracketed
  status (`[up to date]`, `[new branch]`, `[deleted]`, `[rejected] (<reason>)`).
- Refs rejected client-side (non-fast-forward without `--force`) were dropped
  entirely (the code `bail!`ed before reporting them).
- Up-to-date refs were `continue`d away and not reported.
- No status ordering (Git prints up-to-date, then OK, then errors; sorted by ref
  name within each bucket).
- `--dry-run` produced only `To <url>` and returned.
- `--atomic` rejections weren't cascaded to `[rejected] (atomic push failed)`.
- A declined `pre-receive` hook bailed with no porcelain report.

## Fix

New library module `grit-lib/src/push_report.rs` mirrors Git's
`transport_print_push_status` / `print_ok_ref_status` (`git/transport.c`):

- `PushRefStatus` enum (UpToDate / Ok / RejectNonFastForward / RejectAlreadyExists /
  RemoteRejected / AtomicPushFailed / …).
- `PushRefResult` per-ref record.
- `format_push_status(dest, results, porcelain, quiet)` renders both the porcelain
  (stdout, `Done`-terminated) and human (stderr) forms, with Git's bucket ordering
  + alphabetical-by-refname secondary sort.

Subtlety reproduced exactly: the porcelain source field for a *deletion* is `:dst`
for a successful delete and for `REMOTE_REJECT` (Git passes `ref->deletion ? NULL`),
but `(delete):dst` for other error statuses (atomic push failed, non-fast-forward),
since those pass `ref->peer_ref`.

`grit/src/commands/push.rs`:
- Added `up_to_date` / `client_reject` fields to `RefUpdate` (and a `#[derive(Default)]`).
- In-process file path: collect a `Vec<Option<PushRefResult>>` per update at every
  outcome (pre-reject, atomic cascade, applied, remote-rejected, pre-receive-declined)
  and emit once via `emit_porcelain_push_report`. Suppressed the old inline porcelain
  `println!`s inside `apply_ref_update`.
- HTTP path: record up-to-date / non-fast-forward as `up_to_date` / `client_reject`
  instead of dropping/`bail!`ing; only send pushable refs to receive-pack; build the
  report from all refs; handle dry-run and atomic locally.
- `emit_porcelain_push_report` scrubs URL credentials (matches `transport_anonymize_url`).

## Regressions caught and fixed during the work

Keeping up-to-date refs in `updates` broke two t5541 cases until fixed:
- `push already up-to-date`: must short-circuit ("Everything up-to-date", exit 0)
  when nothing is pushable and nothing is rejected — added `nothing_to_push`.
- `push status output scrubs password`: the porcelain `To` line must use the
  scrubbed URL — `emit_porcelain_push_report` now scrubs.
