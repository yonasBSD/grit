## Root causes fixed

Two long-standing heuristics in `ls_tree.rs` diverged from Git's
`builtin/ls-tree.c` `show_recursive` + `read_tree_at` semantics.

1. Ad-hoc match clause matched a trailing-slash pathspec against a *blob*.
   `git ls-tree $tree path0/` (blob `path0`) and the `1.txt/` / `path3/2.txt///`
   cases printed the blob; Git treats a trailing slash as directory-only and
   shows nothing. Removed the extra `args.paths.iter().any(... trim_end_matches('/') ...)`
   clause; `matches_pathspec_set_for_object_ls_tree` already enforces dir context.

2. Descent (`is_ancestor` via `pathspec_wants_descent_into_tree`) recursed into a
   subtree that was itself the *leaf* match of the pathspec, instead of showing it
   as a tree node. `git ls-tree $tree path2/` showed `path2/baz/b` (recursed) instead
   of `040000 tree path2/baz`; likewise the t3101 "filter directories" case recursed
   into `path0/a/b/c`. Replaced with `ls_tree_show_recursive`, a faithful port of Git's
   `show_recursive`: a pathspec descends into an entry only when (relative to the current
   walk base) the spec is strictly *longer* than the entry name and the next char is `/`.

Also: pathspecs are now canonicalized once in `run()` via the existing
`git_normalize_pathspec_preserve_trailing_slash` helper (collapses `.`/`..`/empty
components and runs of `/`, keeping at most one trailing slash) so `path3/2.txt///`
behaves like `path3/2.txt/`. The matching/descent block in `list_tree` was unified to
mirror Git's `show_tree_fmt`: skip non-interesting entries, then `recurse = -r || pathspec
descends`, suppressing the tree node unless `-t`/`-d`.

Removed now-unused import `pathspec_wants_descent_into_tree`.
