# t1700-split-index — d294ed

## Goal
Make `tests/t1700-split-index.sh` fully pass. Baseline: 28/29.

## Failing subtest
- not ok 22 — `same mode for index & split index` (POSIXPERM)
  - Body: `test_commit A`, record `.git/index` mode, then `git -c core.splitIndex=true status`,
    then `shared=$(ls .git/sharedindex.*)` and compare its mode to the index mode.
  - grit left only `.git/index`; no `.git/sharedindex.*` was created by `status`.

## Root cause
The split-index *writing* path (`grit-lib/src/split_index.rs::write_index_file_split`) was already
complete and correct — it creates `.git/sharedindex.<oid>` and the `link` extension. The gap was the
`status` command never *triggered* a write in this scenario.

Upstream Git: after every index read, `tweak_split_index` (read-cache.c) runs. With
`core.splitIndex=true` and an index that is not yet split, `add_split_index` sets
`cache_changed |= SPLIT_INDEX_ORDERED` (split-index.c). During `status`,
`repo_update_index_if_able` (read-cache.c:2755) then writes the index because `cache_changed` is set
— materializing the shared index even when nothing else changed.

grit's `status` only wrote the index opportunistically when `refreshed || racy ||
force_full_index_write || force_sparse_index_write || write_index_for_untracked_cache`. A
freshly-committed repo had none of those, so no shared index was ever written.

## Fix
- `grit-lib/src/repo.rs`: new `Repository::split_index_would_force_write(&Index) -> bool`, mirroring
  `tweak_split_index`/`add_split_index`: returns true iff split index is enabled (config
  `core.splitIndex=true` or `GIT_TEST_SPLIT_INDEX`) AND the loaded index is not already split
  (no `link` extension / base OID).
- `grit/src/commands/status.rs`: add `force_split_index_write = repo.split_index_would_force_write(&index)`
  to the opportunistic index-write condition. The existing write goes through
  `write_index_at` -> `write_index_at_split` with `WriteSplitIndexRequest::default()`, whose
  `want_write_split` already returns true when config enables split index, so the shared index is
  written.

The new condition only fires when split-index is enabled, so normal `status` runs are unaffected.
