## Root cause — HARNESS PORTABILITY BUG, not a grit bug

The subtest unsets GIT_EDITOR/VISUAL/EDITOR and expects `git var GIT_EDITOR`
to exit 1 with empty output. That outcome depends entirely on the terminal
being "dumb": upstream `git_editor()` (git/editor.c) only returns NULL (→ exit 1)
when `is_terminal_dumb()` is true; otherwise it falls back to DEFAULT_EDITOR ("vi")
and exits 0. There is no `isatty` check in `git_editor()`.

The upstream test harness forces this by sanitizing the environment:
`git/t/test-lib.sh:654-656`:
```
USER_TERM="$TERM"
TERM=dumb
export TERM USER_TERM
```

Our ported `tests/test-lib.sh` does NOT set `TERM=dumb` (grep finds no `TERM`
assignment). During the harness run `TERM` inherits the outer shell value
(observed `xterm-ghostty`), so `is_terminal_dumb()` is false and BOTH grit and
real git return `vi` / exit 0.

### Proof (identical behavior, grit == upstream git)
With editors unset, HOME=trash, GIT_CONFIG_NOSYSTEM=1:

| TERM   | grit            | upstream git    |
|--------|-----------------|-----------------|
| dumb   | empty, exit 1   | empty, exit 1   |
| xterm  | `vi`, exit 0    | `vi`, exit 0    |

(The `nvim` seen without HOME override is the developer's global
`~/.gitconfig core.editor=nvim`; irrelevant under the harness because HOME is
the trash dir, so no global config leaks.)

Probe inside the actual harness confirmed: `TERM=xterm-ghostty`, HOME=trash,
GIT_CONFIG_NOSYSTEM empty, GIT_CONFIG_GLOBAL empty → `git var GIT_EDITOR` = `vi`,
exit 0. Every other editor subtest passes because each explicitly sets
GIT_EDITOR/EDITOR/core.editor, which is independent of TERM.

## Conclusion
grit's `grit/src/editor.rs::resolve_git_editor` is a faithful port of upstream
`git_editor()`. The only fix is `TERM=dumb` in `tests/test-lib.sh`, which is
explicitly disallowed by the working contract ("Do NOT modify tests/test-lib.sh").
