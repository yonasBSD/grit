## What I changed (one commit, 3f46efd10)

1. **Fast-forward pick was not recorded for post-rewrite.**
   In `cherry_pick_for_rebase`, the true fast-forward branch (HEAD already equals the
   picked commit's parent, no force-rewrite / no date rewrite) only wrote HEAD and
   returned — it never recorded the commit in `rewritten-pending`. Git's sequencer calls
   `record_in_rewritten` after every successful pick, including the `allow_ff &&
   !is_fixup` fast-forward (sequencer.c ~2344-2356, 4962-4964). Added a
   `record_rebase_in_rewritten_pending` call in that branch when `record_rewrite` is set.
   This is what made the picked commit show up in the post-rewrite hook input, and lets a
   following fixup/squash fold it into the final mapping. Fixes tests 13, 15, 16
   (and is required for 14).

2. **`do_continue` peeked the wrong todo index when recording the resolved commit.**
   After a conflict the todo is rewritten as `todo[i..]`, so the just-resolved commit is
   `todo[0]` and we pop it only at the end of `do_continue`. The flush decision must look
   at the command AFTER it, i.e. index 1. The code used index 0 in the `stopped_oid`
   branch, so it saw the resolved commit's own `pick` and flushed early, mapping it to the
   intermediate (pre-squash) commit. Changed both branches to peek index 1. Fixes test 14
   (`-i squash`); tests 4-10 still pass (index 1 == index 0 for the plain pick case).

3. **Failed `exec` was re-added to the todo and re-run forever.**
   In `replay_remaining`, a standalone `exec` writes the todo as `todo[i+1..]` BEFORE
   running (matching Git `save_todo` with `!reschedule`). On failure the code then
   overwrote it back to `todo[i..]`, re-adding the exec, so the next `rebase --continue`
   re-ran it. With `rebase.rescheduleFailedExec` unset (Git default) a failed exec is
   consumed; removed the re-write so continue resumes at the next command. Fixes test 17.

4. **Cherry-pick base selection simplified** (carried over from prior uncommitted work):
   always use the picked commit's parent tree as the cherry-pick base (matching Git's
   sequencer) instead of sometimes merging against the onto tree, which produced spurious
   conflicts on the commit after a resolved conflict during `--continue`.

## Remaining failure (test 11 — "git rebase with failed pick")
Out of scope for the post-rewrite hook. It exercises interactive `merge -C <rev>` +
`exec >FILE` creating untracked files, and expects each `rebase --continue` to abort with
"untracked working tree files would be overwritten" until the user removes the file.
grit's pick/merge-reuse paths do not yet do untracked-file-overwrite preflight detection
(`preflight_cherry_pick_cwd_obstruction` only guards the current working directory, not
arbitrary untracked-file clobbering). Implementing that is a separate feature with broad
regression surface across the rebase suites.

## Environment caveat (why standard env is 11, not 16)
t5407 uses `lib-rebase.sh set_fake_editor`, which sets `EDITOR` (not
`GIT_SEQUENCE_EDITOR`). Upstream `git/t/test-lib.sh` unsets `VISUAL` and all `GIT_*`
(including `GIT_EDITOR`) at startup; the grit copy of `tests/test-lib.sh` does NOT. So when
the caller's environment has `GIT_EDITOR=true` (common in agent shells) or
`scripts/run-tests.sh`'s `VISUAL=:`, grit's Git-accurate editor resolution
(GIT_EDITOR -> core.editor -> VISUAL[if not dumb] -> EDITOR) picks `true`/`:` instead of
the fake editor, so the `fixup`/`squash`/`drop`/`edit` todo edits never happen and tests
13-17 cannot run as intended. This is identical to how real Git would behave under the
same leaked env. I did not modify `tests/test-lib.sh` (forbidden) and did not make grit
deviate from Git's documented editor order. With the env sanitized the same way upstream
does it, the file is 16/17.
