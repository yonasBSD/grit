## Fix 1 — subtests 63 / 64 / 65 / 94: MIDX 64-bit large offsets

Root cause cluster: `git pack-objects --index-version=2,0x40` forces most pack
offsets into the v2 idx 64-bit large-offset table (off32_limit=64). The test then
corrupts one large-offset value (`corrupt_data $idx64 idxoff=2999 "\02"`) so the
resolved offset exceeds 2^32, and expects the MIDX writer to store it in the MIDX
LOFF chunk (`midx_read_expect 1 63 5 ... " large-offsets"`).

grit's MIDX writer (`build_midx_bytes_filtered` in grit-lib/src/midx.rs) rejected
any offset > u32::MAX with `error: corrupt object: object offset does not fit in
multi-pack-index`, and never emitted a LOFF chunk.

Fixes in grit-lib/src/midx.rs:
- Removed the hard rejection. Now mirrors git/midx-write.c exactly:
  `large_offsets_needed = any(offset > 0xffffffff)`. In that mode every offset with
  `offset >> 31 != 0` goes into the LOFF chunk with slot encoding
  `MIDX_LARGE_OFFSET_NEEDED | slot`; otherwise raw 32-bit. The LOFF chunk is only
  emitted when large offsets are needed.
- Added `MIDX_CHUNKID_LARGEOFFSETS => "large-offsets"` to the read-midx test-tool
  chunk-name table (was printing "unknown" for the 5th chunk).
- MIDX `verify` now loads referenced pack idx files with `read_pack_index_no_verify`
  (matching git's `open_pack_index`, which does not validate the trailing checksum).
  The 64-bit tests corrupt a pack `.idx` in place (stale checksum, valid structure);
  the verifying reader was failing it with "failed to load pack in position 0".

Result: 63, 64, 65 pass -> 92/98.

## Fix 2 — subtest 94: reader bounds-checks large offset table

Test 94 re-uses objects64, writes a MIDX with a LOFF chunk, then `corrupt_chunk_file
LOFF clear` removes the LOFF chunk. `git cat-file --batch-all-objects
--batch-check="%(objectsize)"` must then die with exactly
`fatal: multi-pack-index large offset out of bounds` (and nothing else on stderr).

Three grit bugs:
1. `collect_pack_object_ids` (grit/src/commands/cat_file.rs) used the *verifying*
   `read_local_pack_indexes`, so the corrupt-checksum idx was skipped and
   `--batch-all-objects` listed nothing. Switched to
   `read_local_pack_indexes_cached` (non-verifying, matches `open_pack_index`).
2. `validate_midx_referenced_packs` and `try_read_object_via_midx` both opened the
   referenced pack idx with the verifying `read_pack_index`, emitting a spurious
   `error: packfile <pack> index unavailable` before the die. Switched both to
   `read_pack_index_no_verify`. (Subtest 8's truncated-1064-byte idx still fails the
   non-verifying parse, so "index unavailable" still fires there.)

After this, the cleared LOFF correctly triggers the die at the large-offset slot
lookup with no preceding error line.

Result: 94 passes -> 93/98.
