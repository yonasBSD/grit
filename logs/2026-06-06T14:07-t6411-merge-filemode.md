# t6411-merge-filemode — ticket edfc3e

Date: 2026-06-06T14:07
Result: 19/19 passing (was 17/19), fully passing.

## Failing subtests fixed
- 11: detect conflict on double mode change (recursive)
- 13: detect conflict on double mode change (resolve)

Both exercise an add/add with no merge base where both sides introduce `file2`
as an empty regular file but with different modes: a2 stages it executable
(100755), b2 stages it non-executable (100644). The content (empty) is
identical, so grit's add/add content merge auto-resolved cleanly and recorded a
single stage-0 entry, silently keeping ours' mode — losing the mode conflict.
Real git reports `CONFLICT (add/add)` and records unmerged stages 2/3:

    100755 e69de29… 2  file2
    100644 e69de29… 3  file2

## Root cause / fix
`git merge -s recursive|resolve` runs through `grit/src/commands/merge.rs`
`merge_trees` → add/add arm `(None, Some, Some)` → `try_content_merge_add_add`.
That helper always set `mode = ours.mode` and only conflicted on textual
conflicts. With no base there is nothing to reconcile a mode clash, so a
plain executable-bit difference between two regular files must be reported as
an add/add conflict even when content merges cleanly (the clean merged blob is
still written to the work tree, matching git's empty `file2`).

Fix (`grit/src/commands/merge.rs`, `try_content_merge_add_add`): when both
sides are regular files (MODE_REGULAR/MODE_EXECUTABLE) with differing modes and
`favor == None`, force a `Conflict` result so the call site records unmerged
stages 2/3 (each carrying its own mode). Gitlink/symlink type clashes and
explicit ours/theirs favours are excluded.

Mirrored the same rule in `grit-lib/src/merge_trees.rs`
`add_add_content_conflict` (used by cherry-pick/revert/checkout -m) for
consistency, gated identically to regular-file mode clashes.
