## Changes (4 commits, all regression-free)

### 1. `grit-lib/src/rev_parse.rs` — gitlink in `rev-parse <treeish>:<path>`
`resolve_tree_path_rev_parse` errored `'<path>' is a gitlink, not a blob`.
Git returns the gitlink's recorded commit OID (it lives in the submodule, not
this ODB — do not read it). This single fix unblocks `test_submodule_content`
used throughout `lib-submodule-update.sh`.

### 2. `grit/src/commands/checkout.rs` — stat uid/gid + empty gitlink dir
- The two post-checkout index stat-refresh loops (`switch_to_tree` and
  `merge_branch_working_tree`) set ctime/mtime/dev/ino/size but **not uid/gid**.
  Carried-over unchanged files (built from `tree_to_flat_entries`, uid/gid=0)
  were then reported `M` by real `git diff-files` (uid mismatch). Now copy
  `meta.uid()/gid()` to match `entry_from_metadata`. (Fixes the
  `git_test_func: added submodule creates empty directory` family.)
- `checkout_index_to_worktree` gitlink branch: `checkout_gitlink_worktree_entry`
  returns early (no dir) for an uninitialized submodule, so an added gitlink
  left no worktree entry. Now `mkdir` the empty placeholder, matching Git's
  `write_entry`/`S_IFGITLINK` which always creates the directory.

### 3. `grit/src/commands/checkout.rs` — path-checkout of a gitlink
`git checkout <tree> -- <submodule>` tried to read the gitlink commit as a blob
(`object not found`). New gitlink branch in `checkout_paths`: update the index
gitlink OID + ensure the dir, without reading the submodule-owned commit.
(t2013 `"checkout <submodule>" updates the index only`.)

### 4. `grit/src/commands/_submodule_run_update_inner.rs.inc`
- `submodule update --force` took an "already at recorded commit" fast path and
  skipped the worktree checkout; now skipped under `--force` so the submodule is
  forcibly reset (t7406 "update --force forcibly checks out submodules").
- Custom update command (`submodule.<name>.update = !<cmd>`) now appends the
  recorded commit as git's `run_update_command` does, so `!git checkout` runs
  `git checkout <sha>` (t7406 "command in .git/config").

## Out of v1 scope — fundamentally incompatible with t7400's model

The bulk of remaining t2013 failures (the `--recurse-submodules` family: 7, 9,
10, 11, 12, 13, 16, 17, 18, 26 …, and the `git_test_func` "modified submodule
does not update work tree" family: 51, 52, 57, 65, 66) require ONE of two
checkout-model changes that each regress t7400:

1. **Allow a populated submodule dir through the untracked-overwrite check**
   (`verify_clean_submodule`). Measured: t2013 16→24 (+8) but t7400 111→103
   (−8): it makes t7400 64/66/70 pass, which shifts the deeply chain-coupled
   `submodule deinit` tests 99–109 into a state where `init` loses its index
   gitlink + config registration → all 11 deinit tests fail.
2. **Stop inline-populating an initialized submodule on a plain (non-recurse)
   checkout/reset** (Git leaves the worktree alone). Measured: directly breaks
   t7400 deinit 99–103 because the t7400 chain relies on `git reset --hard` /
   `git checkout` keeping `init` synced to the recorded SHA.

Both are *correct per upstream git* but collide with how t7400's submodule
chain was tuned to 111 (gitlinks synced inline on every checkout/reset). The
brief mandates zero t7400 regression over a larger t2013 gain, so these were
deliberately left out. Reconciling them would require fixing the
`submodule update --init` re-registration path (e.g. t7400 test 66/67 leave
`init` unregistered) so the deinit chain is robust regardless of earlier-test
state — a larger, separate change.
