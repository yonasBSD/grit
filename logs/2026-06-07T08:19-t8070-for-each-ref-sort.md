## Result
Re-ran fresh: still 29/30 passing. Only failing subtest: 23 'count: --count larger than total returns all'.

## Diagnosis (confirms prior two agents, now a third independent confirmation)
NOT a grit defect. grit `for-each-ref --count=100` output is byte-for-byte correct — it returns all
refs (count.unwrap_or(usize::MAX) at grit/src/commands/for_each_ref.rs:128), and the line count of
`out` is identical to the line count of `all`.

Root cause is a macOS BSD `wc -l` left-padding artifact in the test harness:

- Test line 232: `total=$(wc -l <all)` — on macOS BSD `wc -l` left-pads with spaces, so `total`
  becomes e.g. `       7` (or `       1` in a minimal repro).
- `test_line_count` (tests/test-lib.sh, BOTH definitions at line 875 and the shadowing one at 1547)
  strips spaces from `actual` via `tr -d ' '` but does NOT strip spaces from `count` ($2 == $total).
- So the comparison becomes `test "7" = "       7"` → false → subtest fails.

Reproduced exactly with a minimal repo:
```
total var=[       1]
actual=[1]
FAIL: test "1" = "       1"
```

## Why no fix is possible here
Fixable ONLY by:
1. Editing tests/test-lib.sh test_line_count to also strip spaces from `$count` (FORBIDDEN — no
   test-lib.sh edits), OR
2. Editing test line 232 to `total=$(wc -l <all | tr -d ' ')` (FORBIDDEN — test edits limited to
   flipping test_expect_failure → test_expect_success; this subtest is already expect_success).

Passes on Linux/CI where GNU `wc -l` emits no leading padding. No grit Rust change applicable.
