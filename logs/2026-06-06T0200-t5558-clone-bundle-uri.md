## Root cause of the hang (the timeout)
Subtest 33 "expand incremental bundle list" runs, in its setup:

    git -C "$HTTPD_DOCUMENT_ROOT_PATH/fetch.git" fetch origin +refs/heads/*:refs/heads/*

The hybrid git wrapper (lib-httpd.sh) routes glob-refspec fetches to REAL git, but
`fetch.git`'s `origin` is a local path (`clone-from`) so REAL git's fetch-pack speaks
protocol v2 to GRIT's `upload-pack` (resolved via PATH).

Two distinct bugs combined to deadlock that exchange:

1. **grit upload-pack v2 never streamed the pack after `ready` on a stateful transport.**
   `serve_v2::cmd_fetch` sent the `acknowledgments`/`ready`/delim section and then
   `return`ed, looping back to read another request. Upstream `upload-pack.c`'s state
   machine (`UPLOAD_SEND_ACKS` -> `UPLOAD_SEND_PACK`) instead falls through and sends the
   `packfile` section in the SAME response when ready. A stateful (file/ssh) client does
   not send a follow-up request after `ready`, so grit blocked in
   `read_until_flush_or_delim` while the client waited for the pack -> deadlock.
   Fix: thread a `stateless_rpc` flag through `serve_loop`/`process_one_v2_request`/
   `cmd_fetch`. When NOT stateless and `ready`, fall through to stream the pack inline.
   When stateless (smart-HTTP), keep the old per-round behavior (client sends a follow-up
   request with `done`). `upload-pack --stateless-rpc` now calls `process_one_v2_request`
   directly with `stateless_rpc=true`.

2. **grit `index-pack` rejected `--pack_header`.** REAL git's fetch-pack invokes
   `git index-pack --stdin --fix-thin --pack_header=2,8 --strict` (the wrapper resolves
   `git` -> grit). grit's index-pack errored with "unsupported option: --pack_header",
   which (with bug 1) helped wedge the connection. Fix: accept
   `--pack_header=<version>,<nr>` like `unpack-objects` does — reconstruct the 12-byte
   `PACK`/version/count header in front of the stdin body (the caller already consumed it).

## Bug 3: v2 clone fetch sent no `have` lines (subtests 17, 19, 20)
The `negotiation:` tests clone with `--no-local --bundle-uri=...` then grep the packet trace
for `clone> have <tip>` — the bundle tips must be advertised as `have` during the clone's
upload-pack negotiation. grit's protocol-v2 fetch request builder (`write_v2_fetch_request`)
only emitted `want` lines then `done`, never any `have` (single-round, no negotiation), so
the server received no commonality and the trace lacked the required `have` line.

Fix: add a `haves: &[ObjectId]` parameter to `write_v2_fetch_request` (emit `have <oid>`
lines before `done`); all existing callers pass `&[]`, except the clone `--no-local` v2 path
which now computes haves via a new `local_negotiation_haves()` helper. That helper gathers
local tips (refs/bundles/ applied by --bundle-uri, refs/heads/, refs/tags/, HEAD), peels each
to a commit present in the local ODB, excludes wants, and — crucially — honors the
`--negotiation-tip` filter (`negotiation_tip_oids`) when present. Haves are suppressed for an
actual shallow/deepen request (note `shallow_options` is `Some` on every clone, so gate on a
real depth/since/exclude/unshallow request, not its presence).

`read_v2_fetch_pack_response` already skips the `acknowledgments` section before `packfile`,
so the single-round `have ... done` flow streams the pack correctly.
