## Root causes found & fixed (grit/src/commands/rebase.rs)

### 1. Cousins reset target (generate_rebase_merge_script) — tests 2, 16, and unblocks cousins
Git's `make_script_with_merges` (sequencer.c) labels the BOTTOM rev of the `upstream...head`
symmetric walk as `onto` — that bottom is the **merge-base of upstream and head**, NOT the
`--onto` target. grit was labeling `onto_oid` directly. They differ when upstream is not an
ancestor of head (e.g. `rebase -r HEAD^` of a branch with a merge: onto=E but the second
branch's boundary is C). Fix:
- Label `merge_bases_first_vs_rest(upstream, head)[0]` as `onto` instead of `onto_oid`.
- Rewrote the boundary reset block to mirror git exactly: prefer an existing label; else, when
  `!rebase_cousins`, synthesize a label from the boundary's abbreviated hash (git `label_oid`,
  yielding `reset bf8c5d5 # C`); cousins mode leaves it unlabeled → `reset onto`. Removed the
  prior `!interesting.contains(&b)` short-circuit (it forced `reset onto`, breaking the
  no-cousins branch-point reset). t3421 keep-empty still resets to onto because there the
  boundary IS the merge-base (pre-labeled onto).

With this, `git rebase -r HEAD^` of `main` is a true whole-graph no-op (picks F/G fast-forward
since their parent C is unchanged, then the merge fast-forwards back to H).

### 2. Failed `merge -C` continue: staged-changes gate + content-conflict patch (test 7)
In `do_continue`, the blocked-merge-retry path (`rebase-merge-source` exists, no MERGE_HEAD):
- Added git's `commit_staged_changes` gate: if `.git/rebase-merge/message` is absent AND the
  worktree/index differs from HEAD, bail `error: you have staged changes in your working tree`
  BEFORE retrying. A blocked merge writes no `message` file, so a dirty index → error
  (stage 2). After `git reset --hard`, index is clean → proceed (stage 3). Genuine
  conflict-resolution continues take the MERGE_HEAD branch (which always has a `message`), so
  the gate never fires there (no regression to 12/15/21 — those fail only via cascade now).
- On a *content* conflict during the retry (MERGE_HEAD created), drop the merge from
  git-rebase-todo, write `patch` + REBASE_HEAD + stopped-sha (git's `error_with_patch`), then
  bail. Previously it bailed without dropping/patching.

### 3. Failed plain `merge <branch>` content conflict (test 8)
MergePlain conflict handler: on a content conflict, write the synthesized merge message
(`Merge branch '<x>'`) to `.git/rebase-merge/message` and drop the merge from the todo
(`todo[i+1..]`), matching git's `error_with_patch` for a merge with no original commit (no
`patch` written). Previously it kept the whole merge in the todo.

### 4. `cleanup_rebase_state` leaked MERGE_HEAD (the real cascade root) — unblocked 9,11,13-19,21
`cleanup_rebase_state` (run on abort/quit/finish) removed MERGE_MSG/SQUASH_MSG/REBASE_HEAD/
CHERRY_PICK_HEAD but NOT `MERGE_HEAD`/`MERGE_MODE`/`AUTO_MERGE`. A conflicted `merge` step
leaves `MERGE_HEAD`; without clearing it, test 8's `--abort` cleanup leaked MERGE_HEAD into
test 9, whose first `grit merge` died with "You have not concluded your merge (MERGE_HEAD
exists)" — cascading to *every* later test. Added the three files to cleanup (Git's
`remove_branch_state`). This single fix unblocked the bulk of the suite.

### 5. `merge -c` (commit-then-reword) — tests 9 and 34
`merge -c <commit>` now: (a) fast-forwards with reword (`fast_forward_edit`) — new
`reword_fast_forwarded_merge` opens the commit editor seeded with the merge message and
rewrites the FF'd merge in place; (b) for a non-FF `merge -c`, runs the merge `--no-edit`
(NOT `--edit`, which would launch the editor while MERGE_HEAD is live and break a reword editor
that runs `git rebase --edit-todo`), then rewords afterward, taking the AUTHOR from the
ORIGINAL merge commit (the child `grit merge` strips GIT_AUTHOR_* and would otherwise write
`Unknown <>`); (c) writes the remaining todo (`todo[i+1..]`) to disk *before* the merge so the
reword editor's `git rebase --edit-todo` reload sees the right list, and the Completed arm no
longer clobbers that file with the stale in-memory slice.

### 6. Merge strategy/options + unrelated histories + `reset [new root]` pick — tests 10/29 and root cascade
- `run_rebase_merge_subprocess` / `run_plain_merge_for_rebase` now forward the rebase's
  `strategy` and `strategy-opts` state (`-s <strategy>`, `-X<opt>`) to the child `grit merge`, so
  `git rebase -ir -s override -Xxopt` and `-Xtheirs` reach the merge (tests 10, 29).
- Both also pass `--allow-unrelated-histories` (Git's in-process `do_merge` has no such guard):
  a `merge <label>` joining independent roots no longer errors "refusing to merge unrelated
  histories" (test 22 first half).
- A `pick` after `reset [new root]` now becomes a *root* commit. `cherry_pick_for_rebase`
  detects HEAD == the squash-onto sentinel (`pick_is_root`) and emits no parent, instead of
  carrying the sentinel as a parent (which added a spurious commit to the graph). This fixed the
  root-rebase graph and, crucially, stopped test 22 leaving a broken rebase — unblocking the
  cascade through 23-34.

### 7. Merge-tip label from the merge subject — test 25 (and partial 33)
`generate_rebase_merge_script` now derives a merge tip's label from the merge subject when there
is no branch decoration, matching Git's `label_from_message`. `merge_subject_label_from_oneline`
was rewritten to Git's actual pattern: `Merge <anything> '<label>' …` → text in the first quote
pair (so `Merge commit '<oid>' into labels` → `<oid>`), else `Merge pull request … from <x>` →
`<x>`, else the whole subject. A label that is a full object id is then uniquified with a `-N`
suffix by `ensure_unique_label_name` (test 25 `labels that are object IDs are rewritten`). The
subject is extracted from the ` # <subject>` tail of the formatted todo line, not the raw
`pick <hash> # …` prefix.
