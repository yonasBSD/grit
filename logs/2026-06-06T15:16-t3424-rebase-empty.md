## Root causes (two bugs, both shared machinery)

### 1. Three-way content merge wrongly conflicted on identical both-side changes

`grit-lib/src/merge_file.rs` `compute_hunks()` unconditionally emitted a
`Hunk::Conflict` whenever both sides touched the same base region — even when both
sides changed it to **textually identical** content. A stale comment claimed git
always conflicts here ("see t4108-apply-threeway"), but git's `xdl_merge`
auto-resolves: when the two postimages match (`xdl_merge_cmp_lines`), the change is
taken once, not conflicted.

Repro (the t3424 scenario): base line `5`, ours `five`, theirs `five` produced
`<<<<<<< five ||||||| 5 ======= five >>>>>>>` — a bogus conflict. Real `git
merge-file` resolves it cleanly.

Fix: added `merge_lines_equal()` and, in both the changed-region `(true,true)` branch
and the both-sides-insert branch of `emit_inserts_at`, emit `OnlyOurs` (resolved)
instead of `Conflict` when ours == theirs. This is what makes commit C2 (whose
`5->five` change is already present upstream via C) merge cleanly and *become empty*,
instead of hitting a spurious conflict.

This change also improved t4200-rerere (34->35) and t4108-apply-threeway (3->8 ok,
file is `in_scope = skip`); no regressions in t6403/t7201/t3030/t6404.

### 2. Rebase ignored the `--empty` mode for commits that become empty

`grit/src/commands/rebase.rs` hardcoded the became-empty policy: it dropped
became-empty commits on non-interactive rebases and stopped on interactive ones,
never consulting `--empty=drop|keep|stop|ask`.

Fix (mirrors git `builtin/rebase.c` + `sequencer.c allow_empty()`):
- Added `BecomeEmptyMode {Drop,Keep,Stop}` + `resolve_become_empty_mode(args)`
  applying git's defaults: explicit `--interactive` -> stop, `--exec` -> keep,
  else (merge) -> drop. `--empty=stop`/`ask` both halt.
- Persist the resolved mode to state files `drop_redundant_commits` /
  `keep_redundant_commits` (git's names) at setup; read back via
  `rebase_become_empty_mode(rb_dir)` so it survives `--continue`/`--skip`.
- In `cherry_pick_for_rebase`, the became-empty block now keys off the mode:
  Keep -> fall through and commit the empty commit; Drop -> silently drop;
  Stop -> halt with "previous cherry-pick is now empty" message.
  A commit that *started* empty still uses the existing `--keep-empty` flag path
  (`became_empty == commit_tree_oid != parent_tree_oid` distinguishes the two,
  matching git `is_original_commit_empty`).
