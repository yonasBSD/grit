# t5538-push-shallow — subtest 8 "good deltas" (push enumeration + path-walk trace2)

Ticket 375555. Upstream: push from/to a shallow clone. Started at 7/8 (prior agent
fixed subtests 3,4,5,6 for shallow-push handling). This pass closes subtest 8.

## Starting state
7/8 passing. Only failing: subtest 8 "push new commit from shallow clone has good deltas".

The prior agent deemed subtest 8 out of scope ("pack-objects internals"). It is in
fact fixable in the local-push fast path. Subtest 8 asserts three things our push
output got wrong:

1. `Enumerating objects: 5, done` on the first push — we emitted `3`.
2. `Writing objects: .* bytes` — already correct (delta base found => bytes).
3. After `commit --amend` with `-c pack.usePathWalk=true`:
   - `Enumerating objects: 1, done` — we emitted `3`.
   - `test_region pack-objects path-walk config-push.txt` — we emitted no region.

## Root cause
- `grit/src/commands/push.rs` computed the "Enumerating objects" count with
  `estimate_push_progress_enumerated_objects`, a placeholder returning the number of
  refs being sent (clamped up to the written-object count). Git's `nr_seen`
  (`builtin/pack-objects.c`) additionally counts the *preferred-base* (delta-base)
  objects pulled from the receiver's boundary trees during a thin pack: for each
  distinct tree path among the interesting tree/blob objects, the same-path object in
  each boundary (`--not`) commit's tree (root tree for the empty path) is added and
  counted. For the `bigger a` push that is 3 packed + old root tree + old `a` blob = 5.
- The progress was also being computed AFTER the pack was ingested into the receiver,
  so every pushed object already appeared present (interesting set was empty).
- grit never emitted the `pack-objects`/`path-walk` trace2 region that Git's
  pack-objects emits under `--path-walk` / `pack.usePathWalk`. The local-push fast path
  builds the pack in a child whose trace2 stream is stripped, so the region never
  reached the parent's `GIT_TRACE2_EVENT` file.

## Fix
- `grit/src/commands/pack_objects.rs`:
  - `count_thin_push_enumerated_objects(repo, push_tips, remote_git_dir, fallback)` —
    replicates Git's `nr_seen` for a thin push: walks the interesting closure
    (commits-first, stopping at the receiver's objects), then for each distinct
    interesting tree path adds the matching preferred-base objects from the boundary
    commits' trees. Falls back to the supplied count on any traversal error.
  - Supporting helpers: `collect_interesting_with_boundaries`,
    `collect_interesting_tree_objects`, `collect_interesting_object_paths`,
    `collect_tree_path_oids`.
  - Refactored the receiver have-set computation into `local_push_have_roots`
    (closure-filtered, used by the pack builder) and `local_push_remote_object_ids`
    (unfiltered membership; keeps shallow boundary commits whose parents are absent
    locally, which the enumeration needs to recognize boundary trees).
- `grit/src/commands/push.rs`:
  - Compute `push_enumerated_objects` via `count_thin_push_enumerated_objects` BEFORE
    ingesting the pack (against the pre-push receiver state) and use it for the
    "Enumerating objects" progress line.
  - `maybe_emit_push_path_walk_region(config)` — when `pack.usePathWalk` is on and
    `GIT_TRACE2_EVENT` is a file, emit the `pack-objects`/`path-walk` region
    (enter+leave) via the existing `trace2_region_json`, mirroring
    `builtin/pack-objects.c` `get_object_list_path_walk`.
