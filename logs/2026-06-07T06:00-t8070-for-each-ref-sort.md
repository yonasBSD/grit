## Root cause (confirmed, NOT a grit defect)
The failing block (test file lines 228-236):
```sh
git for-each-ref >all &&
total=$(wc -l <all) &&          # macOS BSD wc -l left-pads -> total="       7"
git for-each-ref --count=100 >out &&
test_line_count = "$total" out  # calls test_line_count "=" "       7" "out"
```
`test_line_count` in `tests/test-lib.sh` (~line 875) strips spaces from the
*actual* count only:
```sh
actual=$(wc -l <"$file"); actual=$(echo "$actual" | tr -d ' ')   # -> "7"
test "$actual" "$op" "$count"                                    # test "7" = "       7" -> FALSE
```
The expected `$count` (= `$total` = `"       7"`) is never stripped, so the
string compare fails.

## Proof it is host-tooling, not grit
- grit `for-each-ref --count=100` output is byte-for-byte correct: 7 refs,
  proper trailing newline (verified with xxd).
- Reproduced the *identical* failure using real homebrew `git` 
  (`/opt/homebrew/bin/git`): `total=[       3] actual=[3]` -> FAIL. Real
  upstream git fails this exact pattern on macOS too.
- Passes on Linux/CI where GNU `wc -l` emits no leading padding.
