# t7102-reset.sh — 36/38 → 38/38

## Summary
Fixed `git reset HEAD -- <paths>` so it correctly re-creates index entries for
paths that exist in the target tree (HEAD) but have no stage-0 index entry
(e.g. after `git rm`), and for unmerged paths referenced by pathspec.

## Failing subtests fixed
- `24 - test --mixed <paths>`
- `30 - resetting specific path that is unmerged`

Both shared one root cause.

## Root cause
In `grit/src/commands/reset.rs`, `reset_paths()` built its candidate path set
(`expanded_paths`) for non-`.` pathspecs ONLY from stage-0 index entries that
matched the pathspec. A path removed by `git rm` (no stage-0 index entry) or an
unmerged path could thus be skipped entirely, so `reset HEAD -- file` never
restored the entry from HEAD and emitted a spurious
`pathspec '...' did not match any file(s) known to git`.

Upstream git (`builtin/reset.c` `read_from_tree` → `do_diff_cache`) diffs the
target tree against the pathspec-filtered index, so any path present in the tree
is matched and re-added even when absent from the index.

## Fix
Changed the non-`.` pathspec branch of `reset_paths()` to build the candidate
set as the UNION of:
1. stage-0 index entries matching the pathspec (existing logic), and
2. tree (HEAD) entries matching the pathspec (new).

This mirrors the existing `.` branch, which already unions tree keys + stage-0
index entries. The empty-set bail (`did not match any file(s) known to git`) now
only fires when neither index nor tree has a matching path, matching git. The
per-path loop was unchanged: it already handles in-tree/not-in-index (re-add
from tree) and unmerged (`Index::remove` clears all stages, then re-add stage 0).
