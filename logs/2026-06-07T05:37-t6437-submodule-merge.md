# t6437-submodule-merge — mop-up round 1 (2026-06-07)

Ticket: 981df7. File: tests/t6437-submodule-merge.sh. Was 20/22, now 22/22.

## Failing subtests fixed

### Test 16 — "file/submodule conflict; merge --abort works afterward"

Root cause: the file-vs-submodule "distinct types" add/add conflict
(`conflict_submodule_vs_non_gitlink` in grit/src/commands/merge.rs) staged the
non-gitlink file at the ORIGINAL path `path` (stage 3) and dropped the conflict
blob in the working tree at `path~<oid>` WITHOUT tracking it in the index. Git
merge-ort instead renames the file side to `path~<label>` (e.g. `path~B^0`) and
stages it there at stage 3 (tracked); the gitlink stays at `path` stage 2.

Because grit's `path~<oid>` was UNTRACKED, test 15's `test_when_finished
"git -C file-submodule reset --hard"` left it behind. Test 16 then re-ran the same
merge and it aborted with "untracked working tree files would be overwritten by
merge: path~<oid>" — MERGE_HEAD was never written, so `test_path_is_file
.git/MERGE_HEAD` failed.

Fix: `conflict_submodule_vs_non_gitlink` gained a `relocate_file` flag. For a
normal (non-criss-cross) merge it now stages the file at `path~<label>` (tracked)
and writes the working-tree copy there, producing index `path` (gitlink, s2) +
`path~B^0` (file, s3) — exactly matching upstream (`ls-files -s` = 3,
`ls-files -u` = 2). `reset --hard` then cleans the tracked `path~B^0`, so test 16's
re-merge succeeds.

IMPORTANT gating: `relocate_file = !criss_cross_outer_merge`. The criss-cross /
virtual-merge-base path (e.g. t6416 test 34 "submodule vs symlink") must keep
staging BOTH sides at `path` so `create_virtual_merge_base`'s add/add-omit
heuristic still fires and the path is dropped from the virtual base. Relocating
there regressed t6416 test 34 (the recursive merge auto-resolved cleanly instead
of conflicting). With the gate, t6437 single-base merges relocate; criss-cross /
with-base distinct-types calls pass `false` and keep the prior behavior.

### Test 22 — "merging should fail with no merge base"

Root cause: a submodule add/add with NO common merge base (each superproject
branch introduced the submodule independently) hit the `(None, Some(oe),
Some(te))` both-gitlink arm in `merge_trees`, which just staged 2/3 and emitted a
bare "CONFLICT (submodule)". It never ran the gitlink merge, so the
"go to submodule (sub), and either merge commit <HEAD^1>" advice (on stderr) that
the test greps for was absent.

Fix: route that arm through `try_merge_gitlink_entries` with a synthetic null-base
entry (`be.oid = zero_oid()`, mode gitlink). That reaches the existing
`base_c == zero_oid()` branch, which records the conflict with
`submodule_head_first_parent_abbrev` (= sub HEAD^1) and emits the recursive-merge
advice via `print_submodule_recursive_merge_advice` (stderr). Also guarded
`record_submodule_merge_conflict` to NOT stage a stage-1 entry when the base oid is
null — upstream leaves only stages 2/3 for the no-merge-base case.
