# t1302-repo-version — repository format version check

Ticket: 7e6aaa (config thread, repo-version / early-config)

## Problem

`git config` runs with `RUN_SETUP_GENTLY`. When setup discovers a repository
whose `config` declares an unsupported `core.repositoryformatversion` (e.g. 99,
or unsupported extensions), real git prints

    warning: Expected git repo version <= 1, found 99

to stderr and proceeds as if no repository were present (`have_repository = 0`).
Operations that read/write the local repo config therefore fail with exit 1,
while `--file`/`--global`/`--system` operations still succeed using their
explicit files.

Failing subtest (subtest 3, "gitdir selection on unsupported repo"):

    test_expect_code 1 git -C test2 config core.repositoryformatversion

`test2` has `core.repositoryformatversion = 99`. grit was printing `99` and
exiting 0; real git (2.52.0) prints the warning and exits 1.

## Root cause

grit's `config` command read the local repo config directly without performing
git's `check_repository_format_gently` step during setup.

## Fix

1. `grit-lib/src/repo.rs`: factored the repo-format parsing out of
   `validate_repository_format` into a `RepositoryFormat` struct +
   `parse_repository_format`, and added a public
   `repository_format_warning(git_dir) -> Result<Option<String>>` that returns
   the exact git `verify_repository_format` message (version message
   `Expected git repo version <= 1, found N`, or the unknown / v1-only
   extension messages). Mirrors git/setup.c `verify_repository_format`.
   (NOTE: this library change was also landed by a concurrent agent on the
   shared `grit-t5-progress` branch; HEAD already contains an identical
   `repository_format_warning`, so only the CLI wiring below is mine to commit.)

2. `grit/src/commands/config.rs`: after resolving the discovered git_dir, call
   `repository_format_warning`. On `Some(message)`, print `warning: {message}`
   to stderr and clear `git_dir` (so local-config operations behave as if no
   repository were present, producing exit 1 on the now-missing key, while
   explicit `--file`/`--global`/`--system` reads still work).
