## Fixed: test 115 "updateInstead with push-to-checkout hook" (120 -> 122)

### Root cause
The push-to-checkout hook installed by the test runs:
```sh
git update-index -q --refresh &&
git read-tree -u -m HEAD "$1" || { ...exit }
```
In test 115 phase 3 the work tree has a modified tracked file (`path1` edited but
not staged). git's `update-index -q --refresh` returns **0** for a content-modified
file whose index entry has valid stat data (`REFRESH_QUIET` => read-cache.c
`refresh_index`: `if (quiet) continue;` without setting `has_errors`). grit was
returning **1** (it hardcoded `quiet=false` into its `refresh_index` and exited 1 on
any "needs update"), which short-circuited the `&&`, so `read-tree` never ran and the
hook declined the push -> `! [remote rejected] (push-to-checkout hook declined)`.

### Verified git semantics (homebrew git 2.52 + reference git/ v2.53 source)
- real-stat content-modified file, `-q`  -> exit 0 (suppressed)
- genuinely empty tracked file now modified, `-q` -> exit 0
- deleted tracked file (real stat), `-q` -> exit 0
- `read-tree --reset` / `--cacheinfo` entry (zeroed ctime/mtime/dev/ino), `-q`
  -> exit 1 (NOT suppressed) — this is exactly what t4002 `update-index --refresh -q`
  (`test_must_fail`) depends on.
- no `-q` -> exit 1 + `<path>: needs update` for all of the above.

So `-q` suppresses the non-zero exit for "needs update" EXCEPT for never-lstat'd
entries (the read-tree-reset/cacheinfo signature: stat all-zero).

### Fix (grit/src/commands/update_index.rs)
- Pass `args.quiet` into `refresh_index` (was hardcoded `false`).
- Added `entry_stat_is_zeroed(entry)` (ctime/mtime/dev/ino all 0).
- Gated each `all_uptodate = false` (the five "needs update" branches: submodule,
  symlink-content, regular stat-changed-content, regular stat-matched-content-diff,
  missing-file) on `!quiet || entry_stat_is_zeroed(entry)`.
- Rewrote the stale comment that wrongly claimed `-q` never affects the exit status.

- **103 "fetch follows tags by default"**: a `sort -k 4` test artifact. The default
  `for-each-ref` line `<oid> <type>\t<refname>` has no 4th whitespace field, so
  `sort -k 4` ties on an empty key and falls back to WHOLE-LINE comparison; that
  orders by OID prefix. The annotated-tag OID happens to be lexically smaller than the
  main-commit OID in this run, so `expect` lists the tag first while `git for-each-ref`
  (refname order) lists it last -> `test_cmp` mismatch. Confirmed grit produces
  byte-identical commit AND tag OIDs to real git for the same input, and that real git
  on this machine (both BSD `sort` and GNU coreutils 9.10 `gsort`) produces the SAME
  mismatch. Purely OID-ordering / `sort -k4` dependent; not fixable in grit without
  editing the test. (Matches prior agent findings.)
