## Root cause

Both failing tests run `git -C super ... pull --no-rebase` where `super`'s `origin`
is a **local filesystem path** (`parent`). The superproject commit is intentionally
omitted, so the only effect that should happen is the **fetch phase recursing into
the submodule** and fetching the new submodule commit from the submodule's own remote
(`child`) — without updating the submodule working tree.

In grit, `pull` has a local-path fast path (`grit/src/commands/pull.rs`, the
`if let Some(remote_path) = local_remote_path` block) that copies objects directly
via `super::fetch::copy_objects_for_pull` instead of going through `super::fetch::run`.
That fast path completely bypassed the recursive-submodule-fetch machinery
(`fetch_submodule_recurse::recursive_fetch_submodules_after_fetch`), which the normal
fetch path invokes. So `fetch.recurseSubmodules=true` had no effect for local-path
pulls: the submodule commit was never fetched (`cat-file -e $sub_oid` failed).

`fetch.recurseSubmodules=true` maps to `RECURSE_SUBMODULES_ON`, which in
`recursive_fetch_submodules_after_fetch` iterates every populated index gitlink and
fetches it regardless of the changed-submodule record (git submodule.c
`get_fetch_recurse_config` → `default_argv = "yes"`). So even with no superproject
change, the submodule is fetched.

## Fix

In `grit/src/commands/pull.rs`:

- Added `pull_fetch_recurse_mode(args, config)` — resolves the submodule *fetch*
  recursion mode for a pull (mirrors `fetch_recurse_submodules_mode`):
  `--no-recurse-submodules` → Off, explicit `--recurse-submodules[=v]`, then
  `fetch.recurseSubmodules`, else `Default` (on-demand).
- Added `recurse_fetch_submodules_for_local_pull(...)` — wraps the object-copy +
  ref/FETCH_HEAD writing closure with `begin_fetch_submodule_record` /
  `finish_record_tips_after`, then calls
  `recursive_fetch_submodules_after_fetch`. This reproduces, for the local-path
  shortcut, exactly what `super::fetch::run` does for network remotes.
- Added `fetch_args_for_recurse(config)` — builds a minimal `fetch::Args` (all
  defaults) for the recurse call.
- Wrapped the local-path block's `copy_objects_for_pull` + FETCH_HEAD/ref writes in
  the closure passed to `recurse_fetch_submodules_for_local_pull`.

The submodule **working tree** is intentionally left untouched here — that update is
governed separately by `maybe_update_submodules_after_pull` (which correctly does NOT
run when only `fetch.recurseSubmodules` is set, per the existing
`submodule_update_after_pull_will_run`).
