## Root cause of #121

`git checkout -b beta` with `core.logAllRefUpdates = false` correctly creates no
reflog for `beta`, but `git rev-parse --verify beta@{0}` still succeeded (it must
fail).

In `grit-lib/src/rev_parse.rs::resolve_reflog_oid`, when the parsed reflog entry
list was empty and the index was 0, the code unconditionally fell back to
`refs::resolve_ref` (the ref's current value). That made `beta@{0}` resolve even
when `beta` had no reflog file at all.

Git's behavior (`git/refs.c::repo_dwim_log` + `git/object-name.c::get_oid_basic`):
`ref@{N}` is only resolvable when the **reflog file exists**. With no reflog,
`dwim_log` returns 0 → `get_oid_basic` returns -1 → failure. Only when the reflog
file exists but is empty does `ref@{0}` fall back to the ref value (the
`nth == co_cnt && !is_null_oid` case in object-name.c).

## Fix

`grit-lib/src/rev_parse.rs`, in `resolve_reflog_oid`, the `index == 0 && len == 0`
branch now checks `crate::reflog::reflog_exists`. If the reflog file does not
exist, it errors (`log for '<ref>' is empty`) instead of falling back to the ref
value. The empty-but-existing-file case still falls back to the ref value,
preserving the `@{0}` fallback semantics.

This is a narrow guard: it only changes the case where there are zero parsed
entries AND no reflog file. All non-empty-reflog resolution is unchanged.
