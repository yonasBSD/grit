# t5505-remote mop-up (ticket 26a08e)

Continuing from prior agent (121/130 -> their note). Fresh run: 122/130 passing
(7 fails + 1 known-breakage #26).

## Fixed

### fetch: apply ALL matching primary fetch refspecs to a source ref (tests 27, 61)
Root cause: `git fetch` with two configured refspecs whose dst differ but whose src
overlap (e.g. default `+refs/heads/*:refs/remotes/origin/*` PLUS an explicit
`refs/heads/main:refs/heads/upstream`) only created ONE destination. `git remote show`
no-query mode reverse-maps local refs through fetch refspecs to list "Remote branches
(status not queried)"; with `refs/heads/upstream` missing, `main` never appeared (test
27). Test 24's "Local branches configured for git pull" depended on the same
`refs/heads/upstream` ref existing.

Fix in `grit/src/commands/fetch.rs`:
- New `map_ref_through_refspecs_all()` returns every distinct destination for a source
  ref (mirrors Git `get_fetch_map`, one ref-map entry per matching refspec; negative
  refspec still shields).
- Configured-refspec apply loop now expands the **primary** remote's refspecs with
  `_all` (one FETCH_HEAD entry per source, ref update per destination). Coalesced
  (shared-URL) remotes keep first-match only, so a `git fetch megaloprepus` does NOT
  spill into a co-located `apis/*` tracking namespace (would have regressed test 61
  "update with arguments"; preserves test 121).

Verified minimal repro matches real git: `[new branch] main -> upstream`.

### fetch: default tag auto-following only follows reachable tags (test 56)
Root cause: the local-repo direct-copy fetch path copied *every* advertised tag's object
(`merged.extend(tags…)`) and then wrote every tag whose object existed locally. A dangling
tag like `foobar-tag` (its commit `reset --hard HEAD~1`-ed off the remote branch) was thus
pulled even under default tagopt. Git's `TAGS_DEFAULT` auto-follow only pulls tags whose
target is reachable from a fetched head; `--tags`/`tagopt = --tags` (`TAGS_SET`) pulls all.

Fix in `grit/src/commands/fetch.rs`:
- New `follow_all_tags` flag (true only for `--tags` / `tagopt = --tags`).
- New `peel_tag_target()` peels nested tags to the underlying object.
- After the object-existence tag filter, when auto-following (`!follow_all_tags`), retain
  only tags whose peeled target equals or is an ancestor of a fetched head (existing
  same-named local tags are always kept). t5503-tagfollow (12/12) and t5510-fetch
  (215/215) still fully pass.

### remote rename: on ref-conflict, do NOT rewrite fetch refspec (test 129)
Root cause cascade from test 128. `git remote rename parent parent/child` with a stray
`refs/remotes/parent/child` ref hits a NAME_CONFLICT. Git renames the config *section*
first, then `die(NULL)`s on the failed ref transaction **before** rewriting the fetch
refspec / branch / pushDefault config (builtin/remote.c `mv`). So on conflict the section
is renamed but `remote.parent/child.fetch` keeps the OLD `refs/remotes/parent/*` dst —
which is what lets test 128's finisher `git remote remove parent/child` reverse-map and
delete the stray `refs/remotes/parent/child`. grit instead rewrote the refspec dst to
`refs/remotes/parent/child/*` before bailing, so the stray ref survived and test 129's
`git update-ref refs/remotes/parent/child/branch` then hit a loose-ref D/F "File exists".

Fix in `grit/src/commands/remote.rs` `cmd_rename`: move the conflict check ahead of the
refspec/branch/pushDefault rewrites; on conflict, write config with only the section
renamed and return `ExplicitExit{code:128}` carrying the "renaming remote references
failed … conflicting references" message. Tests 126-129 all pass; t5510 (215/215) clean.

### fetch: bare mirror adopts remote HEAD branch (test 43)
Root cause: `git remote add --mirror -f` runs a mirror fetch; Git's `set_head` for a
`is_bare_repository() && remote->mirror` repo updates the repo's own `HEAD` to
`refs/heads/<remote-head>` (not `refs/remotes/<remote>/HEAD`) and unconditionally
(`create_only = !baremirror`). grit only ever wrote `refs/remotes/<remote>/HEAD`, so a
`init --bare -b notmain` mirror kept HEAD at `notmain` instead of adopting `main`.

Fix in `grit/src/commands/fetch.rs`: in the `set_head` block, add a bare-mirror branch
(`is_bare_repo && remote.<name>.mirror == true`) that writes local `HEAD -> refs/heads/
<default_branch>` when that branch arrived. Mirror tests 42-54 (incl. 44 "non-mirror
fetch does not interfere with mirror") all pass.

### Follow-up: tag reachability gate restricted to commit targets
The default-tag-follow reachability filter (test 56) initially over-pruned t5515's
tags pointing at trees/blobs (`tag-one-tree`, `tag-three-file`), regressing t5515 to
11/65. Refined: only gate tags whose peeled target is a *commit* on commit-ancestry from
a fetched head; tree/blob-targeted tags (content-reachable) are kept once their object is
present. t5515 back to 65/65, test 56 still passes.
