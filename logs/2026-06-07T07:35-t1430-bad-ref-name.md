## The 21-27 blocker (re-confirmed, unfixable in grit)

Subtests 21-27 set up dangling/broken symrefs via `test-tool ref-store
create-symref` and then assert the precondition `test_ref_exists <ref>` before
deleting.

The **local** `tests/test-lib.sh` implements:
```sh
test_ref_exists () { git rev-parse --verify -q "$1" >/dev/null 2>&1; }
```
The **upstream** `git/t/test-lib-functions.sh` implements:
```sh
test_ref_exists () { git ... show-ref --exists "$1"; }
```

`rev-parse --verify -q` resolves/verifies the ref to an object; on a dangling
or broken-named symref it exits **non-zero**, so the precondition fails before
the delete is even attempted. `show-ref --exists` only checks the ref file
exists (exit 0).

Verified against **real git 2.52.0** on the identical setups:
- `rev-parse --verify -q refs/heads/badname` (dangling symref) → exit **1**
- `rev-parse --verify -q refs/heads/broken...symref` → exit **1**
- `show-ref --exists ...` for both → exit **0**

So **even real git fails subtests 21-27 under the local test-lib.sh helper.**
The actual delete operations all succeed in grit (verified: `update-ref
--no-deref -d`, `branch -d` produce correct results / output / empty stderr,
and the ref is then missing). Only the divergent local-helper precondition
fails. Fixing it would require editing `tests/test-lib.sh`, which is forbidden.
Conclusion identical to prior agents (comments 3-6).

## Fidelity fix shipped (rev-parse `--verify` range handling)

While confirming the blocker I found a genuine grit-vs-git divergence:

`grit rev-parse --verify refs/heads/broken...symref` exited **128** with
`fatal: ambiguous argument 'refs/heads/broken'...` — grit unconditionally split
the spec on `...` (symmetric-diff operator) and tried to resolve the left
endpoint `refs/heads/broken` as a range end. Real git exits **1** (quiet) /
prints `fatal: Needed a single revision` (non-quiet).

Root cause: in `grit/src/commands/rev_parse.rs` the `--verify` block split on
`..`/`...` and errored if either endpoint failed, instead of mirroring git's
`try_difference()` (which only treats `A..B`/`A...B` as a range when *both*
endpoints resolve; otherwise it restores the string and resolves the whole spec
as a single revision).

Fix (in `grit/src/commands/rev_parse.rs`, `--verify` branch):
1. Resolve range endpoints with `.ok()` and only emit range output when both
   resolve; otherwise fall through to single-spec resolution.
2. Before single-spec resolution, if the spec still contains `..`/`...` and is
   not a valid ref name (`check_refname_format` rejects it), `fail_verify` —
   matching git, which refuses to follow a broken-named symref and reports
   "Needed a single revision". This avoids the opposite error of resolving
   `refs/heads/broken...symref` to its target OID.

After fix:
- `rev-parse --verify -q refs/heads/broken...symref` → exit **1** (was 128)
- `rev-parse --verify refs/heads/broken...symref` → `Needed a single revision`
- valid ranges (`HEAD~1..HEAD`, `HEAD~1...HEAD`) and normal refs unchanged.

This is pure fidelity — the t1430 subtest outcome is unchanged (both old-128
and new-1 are non-zero under the `-q` local helper).
