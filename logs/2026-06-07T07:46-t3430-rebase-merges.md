# t3430-rebase-merges (ticket 029fa4) — mop-up round 2

Subsystem: rebase-core (interactive rebase / sequencer, `--rebase-merges`) + cherry-pick range.

## Baseline this session
28/34 passing at start. Remaining failures: 20, 22, 26, 27, 28, 33.

## Fixes landed

### 1. Cherry-pick `A..B` range expansion (grit/src/commands/cherry_pick.rs) — test 28
`walk_commit_range(base, tip)` did a naive first-parent walk from `tip` toward root, stopping
ONLY when it hit `base` exactly. When `base` is off the first-parent chain (e.g. `E..G` where E is
a MERGE commit and G is on a separate branch off C), the walk never encountered E and ran all the
way to the root, picking A, C, F, G instead of just F, G. Test 28's `git cherry-pick E..G` then
re-picked A/C (already in E) → "The previous cherry-pick is now empty" and aborted.
Fix: rewrote `walk_commit_range` to compute proper `git rev-list base..tip` semantics — commits
reachable from `tip` but NOT in `ancestor_closure(base)` (grit_lib::merge_base) — then topo-sort
(Kahn) oldest-first, tie-breaking by committer date. Added `commit_date_for` helper.

### 2. `-x`/`--exec` after merge steps (grit/src/commands/rebase.rs) — test 33 (part 1)
Git inserts an `exec` after every `pick` AND `merge` (`todo_list_add_exec_commands`). grit's pick
path runs the persisted `rebase-merge/exec` commands inline, but the MergePlain / MergeReuseMessage
handlers never did, so a `-x "cp $done out"` capturing the post-merge `done` (test 33 greps it for
the merge-tip label) only saw state through the last PICK. Added `run_global_exec_after_merge` and
called it in the MergePlain success path and the MergeReuseMessage `Completed` arm.

### 3. Label truncation uses byte length, not display width (rebase.rs) — test 33 (part 2)
`sanitize_merge_label` truncated non-ASCII UTF-8 chars by display width
(`UnicodeWidthChar::width`), but git's `label_oid` measures `buf->len` in BYTES and keeps a UTF-8
char whole only if `buf->len + (p - label) <= max_len`. For `0123456789 我 123`: at maxLabelLength=14
→ `0123456789-我` (10+1+3=14, accepted); at 13 → `0123456789-` (10+1+3=14 > 13, `我` rejected).
grit's width=2 gave 13 and wrongly kept `我` at maxLabelLength=13. Fixed to use `ch.len_utf8()`.

### 4. Autosquash rearrange for `--rebase-merges` (rebase.rs) — test 27
`generate_rebase_merge_script` never ran the `fixup!`/`squash!` rearrange, so `git rebase -ir
--autosquash` left the fixup as a trailing standalone `pick … # fixup! B` instead of folding it into
B. Git's `complete_action` runs `todo_list_rearrange_squash` over the FULL merge script (picks AND
merges) before the editor. Added `rearrange_autosquash_merge_script` (+ helpers
`pick_or_merge_commit_oid`, `rewrite_todo_command_keyword`) mirroring `todo_list_rearrange_squash`:
match a fixup/squash pick to its target by commit subject (or commit name for a one-token suffix,
or a subject-prefix fallback), convert the line to `fixup`/`squash`, thread it after the target via
next/tail, and rebuild. Wired in at the merge-script call site, gated on `want_autosquash`.

### 5. Octopus merge replay: message + author (rebase.rs + merge.rs) — test 26
`merge -C <octopus>` (3+ parents) recreated the merge with the auto-generated `Merge branches '…'`
subject and the wrong author. Two fixes:
- merge.rs `do_octopus_merge`: honor `-F <file>`/`-m` (compute `octopus_custom_msg` from `args.file`
  like the non-octopus path) so the original message is reused. A bare octopus `-F` is ignored by
  the summary auto-generation; Git's `do_merge` passes `--no-log --no-stat`.
- rebase.rs `run_rebase_merge_subprocess`: for an octopus step (`merge_args` has >1 head) invoke the
  child merge with `--no-log --no-stat -s octopus` (unless an explicit `-s` is forwarded) and export
  `GIT_AUTHOR_NAME/EMAIL/DATE` from the original merge commit's author (Git's `read_env_script`),
  via new `author_fields_from_ident`. A 2-parent merge keeps its author through the in-process
  amend, but the octopus child merge strips GIT_AUTHOR_*.

### 6. `--exec` woven into the merge todo (rebase.rs) — fixes a sibling regression to 27 & 33
A sibling commit gated the global `rebase-merge/exec` file on `!rebase_interactive`; since
`--rebase-merges` is always "interactive", the exec file stopped being written and `-x`/`--exec`
never ran for the merge path (regressing 27 and 33). Switched to Git's actual model: after the
autosquash rearrange, `insert_exec_into_merge_script` weaves `exec <cmd>` lines into the merge todo
after each pick/merge (fixup chains stay attached; trailing exec appended), mirroring
`todo_list_add_exec_commands`. The replay's existing `RebaseReplayStep::Exec` path then drives them.
(My earlier `run_global_exec_after_merge` is now inert for the interactive merge path — the exec
file is never written there — but left in place; harmless.)
