## Root cause
Subtest 7 compares:
```
git rev-list --objects mytag   >list.expect   # tag named by ref
git rev-list --objects $tag    >list.actual   # same tag named by raw SHA
```
The two outputs differed in the *first line* (the underlying commit):
- by ref (`mytag`):  `<commit-oid>\n`            (printed as a commit line)
- by SHA (`$tag`):   `<commit-oid> \0\n`          (printed as an object with empty name)

In `grit-lib/src/rev_list.rs`, `resolve_specs_for_objects_with_options` handled a tag
given by raw SHA whose target is a commit by pushing the *commit* as a `RootObject`
with `wrap_with_tag`. `collect_root_object` then emitted that commit via the object
list with name `"\0"`, which the CLI renders as `<oid> ` + NUL — i.e. `<oid> \0`.

The by-ref path instead peels the tag to a commit, makes the commit a normal walk
tip (`commits`), and emits the tag through `tip_annotated_tag_by_commit`, so the
commit prints as a bare-OID commit line. Git's behavior matches the by-ref path:
`rev-list --objects <tag-sha>` walks the peeled commit as a commit (no name) and
emits the tag alongside it as `<tag-oid> tag`.

## Fix
In the `ObjectKind::Tag` arm (raw-SHA spec branch): when the tag's target peels to a
commit (`expected_kind == ExpectedObjectKind::Commit`), push `tag.object` into
`commits` (walk tip) and register the tag in `tip_annotated_tag_by_commit` instead of
creating a commit `RootObject`. Tags whose target is a tree/blob keep the existing
`RootObject { wrap_with_tag }` path unchanged.
