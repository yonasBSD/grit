# t0614-reftable-fsck.sh — reftable fsck (ticket bc27f1)

## Symptom
Harness reported 1/7 for `t0614-reftable-fsck.sh`, yet running the file directly
(`./t0614-reftable-fsck.sh -v -i`) passed 7/7. The `refs verify` reftable fsck
implementation (table-name validation, well-formed-repo fsck, worktree stack
verification, invalid symref reporting) already existed in
`grit/src/commands/refs.rs` and was correct.

## Root cause (env precedence, not fsck)
The discrepancy was purely about which ref backend `git init repo` created under
the harness.

- The test file exports `GIT_TEST_DEFAULT_REF_FORMAT=reftable` before sourcing
  `test-lib.sh`.
- Upstream `git/t/test-lib.sh` line 551 remaps this:
  `GIT_DEFAULT_REF_FORMAT="${GIT_TEST_DEFAULT_REF_FORMAT:-files}"`, so the
  test-only override wins and git creates a reftable repo.
- Our local `tests/test-lib.sh` does NOT do that remap, and `scripts/run-tests.sh`
  hardcodes `GIT_DEFAULT_REF_FORMAT="${GIT_TEST_DEFAULT_REF_FORMAT:-files}"`
  evaluated at runner level where `GIT_TEST_DEFAULT_REF_FORMAT` is empty → it
  passes `GIT_DEFAULT_REF_FORMAT=files`.
- grit's `init` (`grit/src/commands/init.rs`) read `GIT_DEFAULT_REF_FORMAT`
  FIRST, then `GIT_TEST_DEFAULT_REF_FORMAT`. So at init time both were visible
  (`GIT_DEFAULT_REF_FORMAT=files`, `GIT_TEST_DEFAULT_REF_FORMAT=reftable`) and
  grit chose `files`. No `.git/reftable/tables.list` existed → every reftable
  subtest failed; only the trivial setup-ish first block didn't.

Cannot fix test-lib.sh or the test file. The faithful grit-side fix is to honor
the upstream precedence: `GIT_TEST_DEFAULT_REF_FORMAT` (the test override) wins
over `GIT_DEFAULT_REF_FORMAT`.

## Fix
`grit/src/commands/init.rs`: swap the env lookup order so
`GIT_TEST_DEFAULT_REF_FORMAT` (when non-empty) takes precedence over
`GIT_DEFAULT_REF_FORMAT`, mirroring `git/t/test-lib.sh`.

In normal (non-test) runs `GIT_TEST_DEFAULT_REF_FORMAT` is unset, so the change
is a no-op.
