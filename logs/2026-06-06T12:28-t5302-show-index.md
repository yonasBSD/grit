## Diagnosis
Subtest 14 body:

```sh
show_count=$(git show-index <"$idx" | wc -l) &&
verify_count=$(env -u GIT_EXEC_PATH "$REAL_GIT" verify-pack -v "$idx" | grep -cE "^[0-9a-f]{40}") &&
test "$show_count" = "$verify_count"
```

Instrumented the block (printed to fd 3): `show=[      44] verify=[44]`.

The two object counts are numerically EQUAL (44 == 44, 39 == 39 on smaller
fixtures). The test does a *string* comparison. On macOS/BSD, `wc -l` pads its
output with leading spaces (`      44`) whereas `grep -c` emits no padding
(`44`), so `test "      44" = "44"` is false.

Proof this is a test/environment bug, not a grit bug:
- Ran the exact comparison substituting **real upstream git** for both
  `show-index` and `verify-pack`: it FAILS identically (`show=[      39]
  verify=[39]`). Real git's `show-index` produces the same leading-space `wc`
  output.
- `diff <(grit show-index <idx | sort) <(real-git show-index <idx | sort)` is
  IDENTICAL — grit's show-index output is byte-for-byte equal to upstream's.

This is the same BSD `wc` portability class already documented for
t5300-unpack-objects.

## Resolution
No grit code change is warranted — grit show-index is correct and matches
upstream exactly. The failure is unfixable in grit (cannot edit the test except
expect_failure flips, and this is a genuine expect_success that the host's BSD
`wc` breaks). Left the test as-is; documented findings on the ticket. Status
TOML reflects honest 16/17.
