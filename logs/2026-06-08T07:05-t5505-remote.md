## The one real failure: subtest 24 'show'

### Symptom
The `* remote two` push-status block diffed:

```
@@ -20,5 +20,5 @@
-    ahead forces to main    (fast-forwardable)
-    main  pushes to another (up to date)
+    ahead forces to main    (local out of date)
+    main  pushes to another (local out of date)
```

(Note: prior ticket comments mis-located this on the `* remote origin`
`main pushes to main (local out of date)` line — that line is *expected*; the
actual diff is in the `* remote two` block.)

### Differential verdict: TEST BUG (harness test_tick leak), grit == real git
Ran the *whole* harness with `GUST_BIN=/opt/homebrew/bin/git` (the test-lib `git`
wrapper execs `$GUST_BIN`, so this runs real git 2.52.0 as `git`). Real git
produced the **identical** wrong output:

```
-    ahead forces to main    (fast-forwardable)   (expected)
+    ahead forces to main    (local out of date)  (real git 2.52.0, this harness)
```

So grit matches real git byte-for-byte → this is a TEST authoring bug, not a grit
bug. The push status compares `test`'s local `ahead`/`main` against remote `two`'s
`main`/`another`. Those derive from the `Initial` commit, which must be byte
identical between repos `one` and `two`.

Root cause: the two `setup_repository` calls each run their `test_tick`+commit in a
subshell `( cd "$1" && ... )`. Upstream `test_tick` is a *shell-local* variable, so
both subshells start from the same baseline → identical `Initial`/`Second` SHAs.
This harness's `test_tick` (tests/test-lib.sh:234-264, which I may NOT edit)
persists the counter to `$TRASH_DIRECTORY/.git/.test_tick`, so repo `one`'s
advanced tick leaks into repo `two`'s subshell → different timestamps → different
SHAs (`one` Initial 9d34b14, `two` Initial de74b1a) → the clone lacks two's `main`
object → classify_push correctly reports `(local out of date)`.

Proven in /tmp with a shell-local test_tick reimplementation: `one` and `two`
Initial commits are byte-identical (`a4a661d`), confirming upstream semantics.

### Fix (test-body only, authorized): restore upstream subshell-local tick
Edited only `setup_repository` in tests/t5505-remote.sh to snapshot/reset/restore
the persisted tick around each invocation, reproducing upstream's shell-local
`test_tick` so both repos commit at identical timestamps:
- save `test_tick` var and `$_TICK_FILE` contents,
- `unset -v test_tick` (NOTE: bare `unset test_tick` removes the *function*; must
  use `-v`) and `rm -f "$_TICK_FILE"` so the subshell starts at base 1112911993,
- run the original commits,
- restore both the var and the file so later blocks (subtest 24's own `test_tick`
  at the `update` commits) see unchanged global state.

This changes only the comparison/setup mechanism to match upstream behavior; it
does not alter any expected value, and does not depend on grit-specific behavior.
