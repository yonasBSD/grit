## 77 / 78 / 81 — expire/batch cascade depends on byte-identical pack sizes

`select_packs_for_batch` (multi_pack_index.rs:561) is byte-faithful to git
`fill_included_packs_batch` (git/midx-write.c): same mtime sort, same
`expected_size = referenced<<14 / num_objects * pack_size + (1<<13) >>14`, same
`total_size < batch_size` accumulation with `expected_size >= batch_size` skip. Per-pack MIDX
reference counts match git (A=24,B=15,C=9,D=6,E=7). The SOLE divergence is raw pack BYTE sizes.

Reproduced the `dup` setup with both binaries (mtime order D,C,B,A,E; E newest):

- git  : D=545, C=816,  B=1343, A=2170, E=4569 -> THIRD_SMALLEST=B=1343, BATCH=1344
- grit : D=671, C=1227, B=2518, A=5288, E=4629 -> THIRD_SMALLEST=B=2518, BATCH=2519

Trace of test74 batch (`THIRD_SMALLEST+1`, all packs reference all their objects so
expected≈pack_size):

- git : D(545) total=545<1344; C(816) total=1361 -> loop stops at B (total>=1344). Include {D,C}.
  Repack D,C -> 6 packs; expire keeps 4. PASS.
- grit: D(671) total=671; C(1227) total=1898<2519; B(2518)<2519 include total=4416; A stops.
  Include {D,C,B}. Repack D,C,B -> B emptied -> expire deletes B -> 3 packs (want 4). FAIL.
  78/81 cascade off this state.

Drilled into the byte gap with `verify-pack -v` on pack-D:

- git  pack-D: one tree stored as a 38-byte DELTA (`tree 9 38 ... 1 e60d7aa4`, depth 1).
- grit pack-D: every tree stored FULL (`tree 135 148`, `tree 166 179` — no delta-base column).

grit `optimize_blob_deltas` (pack_objects.rs:4811) filters `e.kind == ObjectKind::Blob` and only
ever deltifies BLOBS (prefix/LCP + thin REF_DELTA reuse); it never builds a delta window over
trees/commits. So grit packs run ~2x git for tree-heavy histories, and test74 derives BATCH from
grit's own (larger) sizes, pulling pack-B into the batch that git excludes.

There is NO MIDX-side lever: BATCH is computed from grit's real on-disk pack bytes. The fix is a
general tree/commit delta-window in pack-objects byte-comparable to git — a large pack-objects
feature with high regression risk across t53xx/t5616/t77xx. Out of MIDX-core scope.

## 98 — pack.preferBitmapTips interprets patterns as hierarchy

`git repack -adb --write-midx` in grit writes a **0-byte** `multi-pack-index-<hash>.bitmap`
placeholder (verified: `stat` = 0 bytes). `test-tool bitmap list-commits` (main.rs ~860) is a
rev-list+drop-last stub, not a reader of real bitmap coverage. The test needs: real EWAH
commit-bitmap writing, bitmap-commit SELECTION (only a subset of tip commits receive coverage so
`comm -13` yields a non-empty `commits-wo-bitmap`), `pack.preferBitmapTips` pattern-as-hierarchy
matching (full refname => trailing-slash semantics => no bitmap; parent namespace => bitmap), and a
non-stub `list-commits`. Entire bitmap subsystem; separate effort, out of MIDX-core scope.
