# t5537-fetch-shallow — mop-up round 2

Ticket 34a2d7 was closed at 7518b1770 claiming 16/16, but a fresh run reported
13/16. The prior fix (subtest 16, HTTP shallow connectivity) is intact; the three
new failures (subtests 10, 12, 13) were `fetch --update-shallow` bugs that the
earlier agent's runs masked (their non-isolated runs raced a concurrent binary
swap and never re-ran to see the real failures).

## Failures and root causes

### Subtests 10 & 13 — `fetch --update-shallow` drops annotated tags

`heavy-tag` (annotated, pointing at `HEAD^` = a commit inside the shallow history)
was downloaded but never written as `refs/tags/heavy-tag` (`git fsck` reported it as
a `dangling tag`). The opportunistic tag-follow reachability gate in
`fetch.rs::run` (`remote_tags.retain`, the `should_fetch_tags && !follow_all_tags`
branch) calls `merge_base::is_ancestor(tag_target, fetched_head_tip)`. Over a
**shallow** clone this returned an *error* (not `false`), so `unwrap_or(false)`
dropped the tag.

Root cause in `grit-lib/src/merge_base.rs::CommitGraphCache::parents_of`: walking
past the shallow boundary commit hits a parent commit that is absent locally.
`peel_to_commit_for_merge_base` reads that parent and returns `Error::ObjectNotFound`,
which the existing `.map_err` only remapped for `InvalidRef` and otherwise
propagated — bypassing the `odb.read` `ObjectNotFound` arm right below it. Fix:
treat `ObjectNotFound` from the peel call as a shallow/missing boundary (no parents),
mirroring Git's graph walk over a shallow clone.

### Subtest 12 — upload-pack rejects a non-tip want over protocol v2

`fetch --update-shallow ../shallow/.git "$SHALLOW":refs/heads/a-shallow` requests the
server's shallow boundary commit, which is *not* a ref tip. grit's v2 server
(`serve_v2.rs`) was applying upstream's v0-only `check_non_tip` /
`allow{Tip,Reachable,Any}SHA1InWant` gating and answered `ERR upload-pack: not our
ref`. Upstream `upload-pack.c` `parse_want` (the v2 path) only verifies the wanted
object is *present* locally — any held object may be wanted in v2; the non-tip
rejection is explicitly v0-only (confirmed against git 2.52: the same fetch succeeds,
exit 0, only warning that the shallow-root ref is not updated). Fix: in
`serve_v2.rs` reduce the want check to "object exists, else `not our ref`" and drop
the now-dead `serve_our_ref_oids` / `serve_is_reachable_from_our_refs` helpers.
