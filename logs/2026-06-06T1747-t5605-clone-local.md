# t5605-clone-local — file:// clone unborn HEAD preflight

Ticket: 1f1895 (test, t5)

## Status
- Before: 22/23 passing.
- After: 23/23 passing (fully passing).

## Failing subtest
19 - "cloning file:// does not hardlink":
```
git clone --bare file://"$(pwd)"/a non-local &&
! repo_is_hardlinked non-local
```
By the time subtest 19 runs, `a.git` (which `file://.../a` resolves to) has a
dangling/unborn HEAD: subtest 8 ran `git -C a.git symbolic-ref HEAD refs/heads/nonexistent`.

## Root cause
A `file://` clone uses the protocol-v2 pack transport (so objects are written fresh,
not hardlinked). The v2 clone preflight sends `ls-refs ... unborn`, and the server
replies with an unborn HEAD advertised per gitprotocol-v2 as:
```
unborn HEAD symref-target:refs/heads/nonexistent
```
i.e. the literal token `unborn` in place of the object id (`obj-id-or-unborn`).

`parse_ls_refs_v2_line` (grit/src/commands/ls_remote.rs) called
`ObjectId::from_hex("unborn")`, which errored, so the whole preflight failed with
`file:// protocol v2 clone preflight: bad oid in ls-refs: unborn`. The clone then
fell back to the local hardlink path, producing a hardlinked repo — failing the test.

## Fix
1. `parse_ls_refs_v2_line` (grit/src/commands/ls_remote.rs): treat the literal
   `unborn` OID token as the null OID (`ObjectId::zero()`) instead of erroring.
2. `collect_clone_ls_refs_metadata` (grit/src/file_upload_pack_v2.rs): when HEAD's
   advertised OID is zero (unborn), leave `head_oid` unset (`None`) and let the
   symref-target drive checkout; also skip zero-OID head/tag refs from `wants`.

The on-disk HEAD fallback (`read_source_head_info`) and `guess_checkout_branch`
already handle the unborn-symref case, so once the preflight no longer crashes the
clone proceeds over the pack transport and the result is not hardlinked.
