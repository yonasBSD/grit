# t5510-fetch — subtest 187 "all boundary commits are excluded"

Ticket: 7ace33. Started at 214/215 (only subtest 187 open per prior comments).

## Root cause

`git bundle create main --since=<merge-date>` over a merge tip must emit a *thin*
pack that `git index-pack --fix-thin` expands to exactly 3 objects: the merge
commit, the merge tree, and one boundary base appended by fix-thin.

The merge tree (`base.t | oneside.t | otherside.t`, 110 bytes) has the oneside
boundary tree (`base.t | oneside.t`, 71 bytes) as a strict byte prefix; git
writes the merge tree as a REF_DELTA against that boundary tree, and fix-thin
re-materialises the one boundary base → 3 objects.

grit's bundle pack writer (`build_pack_data`) emitted only full objects and the
`--since` path re-added prerequisite commit roots as full objects, producing 7
objects. Removing that heuristic naively regressed t6020 (which relies on the
heuristic giving 13).

## Fix (grit/src/commands/bundle.rs)

Added `try_build_thin_boundary_pack` for the `--since`/`--until` boundary case.
It fires only when:
- there are no genuinely-new content blobs (every blob is boundary-reachable),
- there is at least one new tree and no new tags, and
- every new tree is a strict byte prefix-extension of some boundary tree.

In that shape it builds a thin pack: new commits full + new trees as REF_DELTA
(via `delta_encode::encode_prefix_extension_delta`) against the boundary tree,
omitting blobs. fix-thin then appends exactly the referenced boundary base.

In every other shape it returns `None`, so the existing full-object heuristic
(`include_prerequisite_commit_roots`) is kept unchanged. t6020's `--since --all`
bundle has genuinely-new blobs and same-size (non-prefix) new trees, so it takes
the old path and stays at 13.

Also added `push_pack_obj_header` helper to share the pack object-header encoding.
