# t5300-unpack-objects — investigation log

Ticket: 1b3636 (test, t5). Status TOML last scan: 22/23 (failing=1).

## Repro
`./scripts/run-tests.sh t5300-unpack-objects.sh` → `✗ (22/23)`.

Verbose direct run pinpoints the single failure:
- `not ok 3 - unpack-objects -n: dry run writes no loose objects`

All other 22 subtests pass, including the other two dry-run subtests
(16 `-q -n together`, 22 `-n --strict together`).

## Root cause: BSD vs GNU `wc`, NOT a grit bug

Subtest 3 body:
```sh
grit init dry.git --bare &&
grit -C dry.git unpack-objects -n <test.pack &&
count=$(find dry.git/objects -type f | wc -l) &&
test "$count" = "0"
```

- grit's `unpack-objects -n` is correct: exit 0, writes **zero** loose objects
  (`find dry.git/objects -type f` is empty). Confirmed in grit-lib
  `unpack_objects.rs` (`write_or_hash`, `if dry_run { hash only }`, line ~651)
  and its unit test `test_unpack_objects_dry_run_writes_nothing`.
- The failure is purely shell-portability. On macOS BSD `wc -l` of empty input
  prints `       0` (left-padded). In an **assignment** context
  (`count=$(...)`) command substitution preserves leading whitespace (POSIX:
  word-splitting/IFS stripping only happens on *unquoted* expansions, not on
  assignment), so `count` = `       0` and `test "       0" = "0"` is false.
- Proven side-by-side on this box:
  - `find <empty> -type f | /usr/bin/wc -l`  → `       0`  (BSD, this macOS)
  - `find <empty> -type f | gwc -l`           → `0`        (GNU coreutils)
- **The identical assertion FAILS with `/usr/bin/git` too** — real git's
  `unpack-objects -n` behaves exactly like grit's. So there is no behavioral
  difference between grit and Git here.
- Sibling dry-run subtests 16 and 22 avoid the bug by piping through
  `| tr -d " "`; subtest 3 omits it. This is a custom (non-upstream) test file
  ("grit unpack-objects basic tests"); upstream t5300 is the unrelated
  `t5300-pack-object.sh`. Subtest 3 has used the non-portable form since the
  file was first written (commit f0bf384e7) — it never regressed.

## Conclusion
On the canonical Linux grading environment (GNU `wc`), subtest 3 — and the whole
file — passes 23/23. The 22/23 here is a macOS-only artifact of BSD `wc`
padding. There is **no grit-side fix**: grit's `unpack-objects -n` already
matches Git exactly. The only real fix would be adding `| tr -d " "` to the test
body (as siblings 16/22 do), which is a forbidden test-body edit under this
repo's rules (only `test_expect_failure`→`test_expect_success` flips are
allowed, and this subtest is already `test_expect_success`).

Left the file unchanged and the ticket open with these findings for the mop-up
agent / for a maintainer decision on the test-script portability fix.
