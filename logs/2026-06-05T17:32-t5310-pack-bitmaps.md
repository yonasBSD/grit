# t5310-pack-bitmaps work log

Ticket: 8611b9 — tests/t5310-pack-bitmaps.sh

## Baseline
- Fresh run start: 151/236 (ticket said 148; earlier group fixes had bumped it).

## Root causes found
1. `pack-objects --revs` (the `collect_pack_objects_from_rev_stdin_lines` path) did NOT
   honor `--local`, `--honor-pack-keep`, or `--incremental`. These flags must omit objects
   that are available in non-local (alternate) storage, in `.keep`-marked packs, or in any
   pack respectively — mirroring `want_found_object` / `want_object_in_pack_mtime` in
   `git/builtin/pack-objects.c`.
   - Fix: added `pack_objects_locality_excludes` + `apply_locality_excludes` in
     grit/src/commands/pack_objects.rs, applied to both `--revs` return paths (sparse + default).
   - `--local`: exclude objects loose in an alternate source OR present in a non-local pack.
   - `--honor-pack-keep`: exclude objects in a pack with a `.keep` sidecar.
   - `--incremental`: exclude objects present in any pack (local or alternate).
   - Result: 151 -> 172.

2. Pack bitmaps themselves are NOT really implemented: pack-objects writes an EMPTY
   placeholder `.bitmap` file (std::fs::write(..., [])). So:
   - test 5/81/162 "full repack creates bitmaps" need trace2 keys
     num_selected_commits=106 / num_maximal_commits=107 (bitmap commit-selection algorithm).
   - `rev-list --test-bitmap` is a no-op returning success — so "truncated/corrupt bitmap
     fails gracefully" + "load corrupt bitmap" + "complains about multiple pack bitmaps"
     can't emit the expected error text.
   - `test-tool bitmap list-commits` does not exist -> pack.preferBitmapTips tests fail.
   - "enumerating progress counts pack-reused objects" needs real pack-reuse + bitmaps.
   These require a real pack-bitmap reader/writer (large feature).

## Batch 2: missing-object tolerance in pack-objects --revs exclude walk (172 -> 181)
- `pack with missing blob/tree/parent` (58/59/60 etc.) failed because the negative/exclude
  side of `pack-objects --revs` read the (now-removed) boundary trees/commits and errored
  with "bad tree object". Git tolerates missing objects on the uninteresting side (bitmaps
  or gentle parsing); the missing objects can't be in the interesting set anyway.
- Fix (grit/src/commands/pack_objects.rs):
  - added `walk_reachable_lenient` / `walk_reachable_inner(lenient)`; use lenient for the
    negative walks in both `--revs` return paths.
  - sparse path: `collect_revs_pack_objects_sparse` exclude-commit BFS, edge-parent read, and
    `add_children_by_path_for_sparse` now skip missing objects instead of erroring.
- t5300/t5332 still fully pass (no regression).

## Batch 3: force_include locality filter + --all --objects non-commit refs (181 -> 187)
- `pack reuse respects --honor-pack-keep` / `--incremental` (65/67) packed 3 objects that
  should be omitted. Two root causes:
  1. `force_include` OIDs (raw-OID args, e.g. `for-each-ref | pack-objects --revs`) bypassed
     the locality exclude filter. Fix: `apply_locality_excludes` now filters force_include too.
  2. The blob pointed to by a lightweight tag (`tagged-blob`) was never packed by `repack -ad`
     because `rev-list --all --objects` (the lib `rev_list()` with options.all_refs) dropped
     refs that peel to a blob/tree. Fix (grit-lib/src/rev_list.rs): added
     `all_ref_non_commit_object_roots` and append it as object_roots when objects && all_refs.
- Note: the CLI `git rev-list --all` path (push_all_refs_as_specs) still drops blob refs in
  --objects mode; not fixed here (no failing t5310 subtest needs it, and regression risk).
  The repack path uses pack_objects_all_enumeration -> lib rev_list() which IS fixed.
- Regression check: t6000/t5300/t5301/t1410/t5304/t6101/t5510/t6011 fully pass; t7700 (36/47)
  and t6112 (52/54) unchanged from baseline.

## Batch 4: bitmap object output format when a .bitmap is present (187 -> 199)
- `enumerate --objects (full/partial bitmap, ...)` (14/23/35/44 ...) failed with
  "identical raw outputs; are you sure bitmaps were used?": grit's
  `rev-list --use-bitmap-index --objects` emitted output IDENTICAL to non-bitmap (with
  pathnames). Git's bitmap object traversal emits OID-only lines (no pathnames), so the two
  outputs differ yet normalize (cut OID + sort) equal.
- Fix (grit-lib/src/rev_list.rs): `bitmap_object_format` now also true when a pack/MIDX
  `*.bitmap` exists in the object store (`pack_bitmap_present`), EXCEPT when a
  bitmap-incompatible filter (`sparse:oid`, `tree:<n>`) forces a fallback
  (`filter_forces_bitmap_fallback`) — that path must keep normal path-bearing output.
- Regression: t6113-rev-list-bitmap-filters back to 14/14 (the filter-fallback guard was the
  fix); t6000/t5300/t5301/t5304/t5311/t1006 all fully pass.

## Batch 5: pack-objects accepts --use-bitmap-index/--no-use-bitmap-index (199 -> 202)
- `pack-objects to file can use bitmap` (53/129/210) failed: pack-objects rejected
  `--no-use-bitmap-index` / `--use-bitmap-index`. Added both as accepted-for-compat flags
  (grit produces the same object set either way). t5300/t5316/t5332 still fully pass.
