# t6110-rev-list-sparse — case-insensitive merge spelling fix

Date: 2026-05-29
Branch: wf/p2/t6110-rev-list-sparse
Target: tests/t6110-rev-list-sparse.sh

## Result
- Before: 1/2 passing (subtest 1 "setup" failing).
- After: 2/2 passing.

## Root cause
On case-insensitive repos (`core.ignorecase=true`, the default `grit init`
writes on macOS), the merge machinery in `grit/src/commands/merge.rs`
ASCII-lowercases each path component so cross-side case-only renames collapse
to a single merge key (needed for t6419). `tree_to_map_for_merge` also
overwrote the stored entry path with the lowercased key
(`e.path = key.clone()` at the old line 8306), so `merge_trees` emitted
lowercased result paths (`a.t` instead of the tracked `A.t`).
`bail_if_merge_would_overwrite_local_changes` then compared those lowercased
paths byte-for-byte against the real index (original spelling) and
misclassified the tracked files as untracked, aborting an otherwise clean
merge with "The following untracked working tree files would be overwritten by
merge: a.t ...". t6110 subtest 1 ends in `git merge main`, so setup failed.

## Fix (grit/src/commands/merge.rs only)
Keep the lowercased keys/paths inside the merge engine (still required for
case-fold identity), but rewrite the merge result back to the original
on-disk spelling after `merge_trees` returns, guarded by `core_ignorecase`:

- `build_original_spelling_table` builds a canonical-lowercase ->
  original-spelling map from the un-lowercased side trees. Priority matches
  `git merge-ort`: a side whose spelling differs from base (the side that did
  the case-rename) wins; among equally-eligible sides prefer ours, then
  theirs, then base. Each side map uses `or_insert` so first-seen spelling
  wins per side.
- `restore_entry_spelling` / `restore_index_spelling` / `restore_map_spelling`
  / `restore_string_path_spelling` / `restore_merge_result_spelling` rewrite
  the result index entries, conflict-file keys, conflict-description path
  fields, and the ours/base/theirs maps used downstream, recomputing the
  0xFFF path-length flag bits exactly as `tree_to_map_for_merge` does.
- Applied in `do_real_merge` right after the `merge_trees` call.

Case-sensitive repos take the early `None` return and are byte-for-byte
unchanged.
