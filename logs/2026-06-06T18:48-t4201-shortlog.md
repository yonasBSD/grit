# t4201-shortlog — ticket 80b9b4

## Status
- Before: 29/32. After: 30/32. Fixed subtest 7 ("shortlog wrapping").
- Still failing: 8 ("shortlog from non-git directory"), 9 ("shortlog can read --format=raw output").

## Root causes

### Subtest 7 (FIXED) — commit stored non-UTF-8 body lossily
The wrapping test commits five messages, two of which contain bytes that are not
valid UTF-8 once `i18n.commitEncoding=non-utf-8` is set:
- commit 3: `\360\235\204\236` (U+1D11E treble clef — valid UTF-8)
- commit 4: `\370\235\204\236` (invalid UTF-8 5-byte-style lead)

`git shortlog -w` is then compared against a hardcoded `expect` heredoc that
contains the verbatim bytes. Git stores commit bodies *verbatim* and never
transcodes them (for an unknown encoding like `non-utf-8`, `logmsg_reencode`
is a no-op). grit's `commit` command instead ran the message through a lossy
`String` (`String::from_utf8_lossy`), turning commit 4's `\370\235\204\236`
into four U+FFFD (`\357\277\275`) before writing the object. The stored object
itself was wrong, so every downstream reader (shortlog, log) showed mojibake.

Fix (grit/src/commands/commit.rs):
- Added byte-level cleanup helpers: `git_vertical_stripspace_bytes`,
  `cleanup_edited_commit_message_bytes`, `cleanup_message_bytes`.
- In `prepare_commit_message`, for the non-editor `-m` path, when
  `args.raw_messages` contains invalid UTF-8, build `MessageResult.raw_bytes`
  from the verbatim argv bytes (joined with `\n\n`) cleaned at the byte level,
  instead of discarding them.
- Same for the non-editor `-F` path: when the file content is invalid UTF-8,
  preserve verbatim bytes via byte-level cleanup.
These `raw_bytes` flow through the existing `raw_message` plumbing into
`serialize_commit`, so the object is stored byte-exact. Valid-UTF-8 messages
are untouched (no regression); only invalid-UTF-8 payloads take the new path.

Verified: commit objects now store `\360\235\204\236` and `\370\235\204\236`
verbatim, and `git shortlog -w HEAD` matches the expect file exactly.
No regressions: t7502-commit-porcelain (82/82), t3900-i18n-commit (38/38),
grit-lib unit tests pass (modulo the 2 known ignore::gitignore_glob failures).

### Subtests 8 & 9 (STILL FAILING) — `grit log` mangles non-UTF-8 bodies
Both pipe `git log` output into `shortlog -w` from stdin and compare to the same
`expect`. The commit objects are now stored correctly, but `grit log` itself
re-emits commit 4's body as U+FFFD instead of the raw `\370\235\204\236` bytes.

Root cause: `grit log`'s entire formatting pipeline is `String`-based. In
`grit-lib/src/objects.rs::parse_commit`, the body is decoded with
`commit_encoding::decode_bytes(encoding, body)`, which for an unknown encoding
falls back to `String::from_utf8_lossy` (→ U+FFFD). `CommitInfo.message`,
`commit_pretty`, the grep filters, and `format_commit` in log.rs all operate on
that `String`, so the invalid bytes cannot survive. `parse_commit` already keeps
the verbatim bytes in `CommitData.raw_message`, but the log output paths use
`.message`, not `.raw_message`.

Why not fixed here: making `grit log` byte-clean for non-UTF-8 bodies means
threading raw bytes through `CommitInfo`, `format_commit` (a ~ hundreds-of-lines
function with many `--format`/`--pretty`/raw/oneline branches), and the grep
pipeline — shared thread-B log-pretty machinery. That is a large, risky change
well beyond the shortlog ticket and could regress many passing log tests. It
should be its own ticket: "grit log emits commit bodies verbatim (no lossy
UTF-8 conversion for unknown i18n.commitEncoding)". The fix would route the
default/`--format=raw` body output (and ideally `%b`/`%s` raw output) through
`raw_message` bytes when present rather than the lossy `message` String.
