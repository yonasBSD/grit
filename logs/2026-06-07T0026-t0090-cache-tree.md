# t0090-cache-tree (ticket fbeb9b)

## Symptom
Harness reported `status=error`, 0/0 â€” the script aborted before a `# Tests:`
summary. Log stopped after `ok 11`, then `FATAL: Unexpected exit with code 0`.

## Root cause
Not a grit crash. Test 12's `test_cache_tree` helper calls the recursive shell
function `generate_expected_cache_tree`, which runs
`git ls-tree --name-only -d HEAD -- "$pathspec/$subtree/"` to find sub-subtrees.

grit's `ls-tree -d HEAD -- dir/` (trailing slash) incorrectly emitted the
directory entry `dir` **itself** in addition to entries inside it. Real git's
`show_tree_common`/`show_recursive` (builtin/ls-tree.c) only prints an
intermediate tree node when `LS_SHOW_TREES` is set; for a plain `-d` with a
trailing-slash pathspec the directory entry itself is NOT a leaf and must be
suppressed (only entries strictly below it are shown).

Because grit kept re-emitting `dir`, the shell recursion `dir -> dir/ -> dir ->
...` never terminated; eventually a subshell `exit 0` propagated and tripped
test-lib's EXIT trap â†’ FATAL.

Confirmed against real git:
- `ls-tree -d HEAD -- dir/`     -> `dir/sub`         (NOT `dir`)
- `ls-tree -d HEAD -- dir`      -> `dir`
- `ls-tree -r -d HEAD -- dir/`  -> `dir`, `dir/sub`  (-d -r auto-sets LS_SHOW_TREES)
- `ls-tree -t HEAD -- dir/`     -> `dir`, ...         (-t = LS_SHOW_TREES)

## Fix
grit/src/commands/ls_tree.rs, in `list_tree`'s recurse block: the
intermediate-node print was gated on `args.show_trees || args.only_trees`.
Dropped the `|| args.only_trees` so it only prints when `show_trees` is set
(which `-t` sets directly and `-d -r` auto-sets at line ~125), matching git's
`LS_SHOW_TREES` semantics.

## Second fix: test 10 `commit --interactive gives cache-tree on partial commit`
`run_commit_interactive_mode` (grit/src/commands/commit.rs) was a stub: it
printed the menu, read ONE line, only handled `2`/`u`/`q`, and `exit(1)`'d on
anything else â€” so `p` (patch) just aborted. The test drives the full add--i
menu (`p 1 "" s n y q`).

Rewrote `run_commit_interactive_mode` to mirror Git's `interactive_add`:
1. Seed a temp index (`$GIT_DIR/next-index-<pid>`) from the current index.
2. Point `GIT_INDEX_FILE` at it and delegate to the existing
   `add_interactive::run_add_i` engine (already drives status/update/revert/
   add-untracked/patch/diff correctly â€” confirmed via `git add -i`).
3. Reload the temp index (now carrying the staged hunks) and return it as the
   index the commit tree is built from. The temp file is removed; the REAL
   on-disk index is deliberately NOT mutated here.
4. The staged index is promoted to the real index only in the post-commit
   refresh (commit.rs ~1796), i.e. AFTER the commit succeeds. Abort paths
   (empty message etc.) `exit` before that, so an aborted `commit --interactive`
   leaves the index unchanged (t7501 "doesn't change index if editor aborts").

Verified manually: partial commit stages only the 2nd hunk, HEAD has foo=42 /
bar=44, `git status` shows ` M foo.c`; editor-abort leaves the index unchanged.
