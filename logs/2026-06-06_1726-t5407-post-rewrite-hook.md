# t5407-post-rewrite-hook — obstructed-pick re-apply on rebase --continue

Ticket: d16474 (tags: test, t5)
Stem: t5407-post-rewrite-hook.sh
Final: 17/17 passing.

## Symptom

`t5407` subtest 11 "git rebase with failed pick" failed (16/17). The test runs an
interactive rebase whose todo creates untracked files via `exec` (`exec >G`,
`exec >H 2>I`) before the `merge -C merge-E E`, `pick G`, `pick H`, `fixup I`
steps. Each step aborts with "untracked working tree files would be overwritten",
the user removes the file and `git rebase --continue`s, and the final
post-rewrite hook input is checked. The expected mapping is:

    merge-E -> HEAD~2
    G       -> HEAD~1
    H       -> HEAD
    I       -> HEAD

Grit instead recorded every old commit collapsed onto the *previous* HEAD, e.g.
`G -> <merge-E commit>` instead of `G -> <G's new commit>`, and dropped the
`I -> HEAD` line entirely.

## Root cause

When a pick/fixup aborts because materializing it would overwrite an untracked
working tree file, the step is never applied, so the worktree still equals HEAD.
On the following `git rebase --continue` the continue path saw `stopped-sha` set
and `worktree_matches_head()` true and took the "empty conflict resolution"
shortcut: it recorded `old_oid -> current_HEAD` and advanced without re-running
the pick. So the obstructed commit was never actually reproduced and its mapping
was wrong. For the trailing `fixup`, a separate fixup/squash no-op shortcut
dropped it without recording anything.

## Fix (grit/src/commands/rebase.rs, `do_continue` + `cherry_pick_for_rebase`)

The bulk of the obstruction handling (writing an `obstructed-pick` marker in
`cherry_pick_for_rebase` and re-running the step on `--continue` via
`rebase_pick_was_obstructed` + `cherry_pick_for_rebase`) was already present on
the workspace branch and already brought t5407 to 17/17 in isolation. The
remaining bug was that the re-apply path also fired for `squash` steps, whose
editor-driven message folding does not re-run cleanly here — that left a dangling
rebase in t3404 tests 95/96 (already-failing obstruction tests) which cascaded
into "a rebase is already in progress" failures of t3404 tests 97 ("rebase
--continue removes CHERRY_PICK_HEAD") and 98 ("drop").

This commit restricts the obstructed re-apply to `pick` and `fixup` only; `squash`
keeps the pre-existing baseline no-op path. Two hunks in `do_continue`:
the fixup/squash no-op block now only skips its shortcut for an obstructed
`fixup` (not squash), and the pick re-apply gate is `Pick | Fixup` (not Squash).
