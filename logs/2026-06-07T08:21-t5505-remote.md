## Fix for subtest 130 (rename moves around the reflog)

`git for-each-reflog-ent` must emit reflog entries OLDEST-first (upstream
`files_for_each_reflog_ent`, git/refs/files-backend.c:2341, streams the reflog file
front-to-back; appended files are chronological, so front-to-back == oldest-first). Only
the `-reverse` variant (`files_for_each_reflog_ent_reverse`) emits newest-first.

grit's test-tool wiring in `grit/src/main.rs` reversed the WRONG variant: it reversed for
plain `for-each-reflog-ent` (making it newest-first) and left `-reverse` un-reversed.
`grit_lib::reflog::read_reflog` already returns entries in file order (oldest-first), so
the correct behavior is to reverse ONLY for `for-each-reflog-ent-reverse`.

Fix (grit/src/main.rs ~line 1106):
```rust
if sub == "for-each-reflog-ent-reverse" {
    entries.reverse();
}
```
(was `if sub == "for-each-reflog-ent"`).

This makes t5505 #130 pass (the expected reflog after `remote rename` is the original
creation entry followed by the `remote: renamed ...` entry — oldest-first).

## Remaining failure: subtest 24 (show) — NOT a grit bug, unfixable
Root cause is in forbidden `tests/test-lib.sh`. The ported `test_tick` (lines 227-262) adds
`_TICK_FILE` PERSISTENCE across subshell boundaries, which upstream
`git/t/test-lib-functions.sh:133-141` does NOT have (upstream `test_tick` is a plain
shell-local variable). In the `show` test the inner `(cd ../one && test_tick && git commit)`
subshell advances the on-disk tick; upstream's shell-local tick leaves `setup_repository one`
and `two` with byte-identical Initial commits, so the clone holds two's main and push status
is "up to date". grit's persisted tick leaks one's advanced value into two's subshell ->
different Initial SHAs -> clone lacks two's main -> classify_push correctly reports
"(local out of date)". Real git run in THIS harness produces the same wrong output, proving
the divergence is the harness file, not the implementation. Unfixable without editing the
forbidden `tests/test-lib.sh`. grit is correct.
