## Failures and fixes

### Cluster C ‚Äî bisect/show-branch `-h` / `--help-all` (tests 17, 18, 113, 114)
`bisect` and `show-branch` are dispatched manually in `grit/src/main.rs`, so they bypass
the centralized `-h`/`--help-all` synopsis path and discover the repo first, erroring
`not a git repository` (exit 1) outside a repo. Added a call to
`commands::upstream_synopsis_help::try_print_upstream_help_and_exit(subcmd, rest)` before
each manual dispatch. That helper only fires when `rest` is exactly one of `-h`/`--help`/
`--help-all`, so the manual `--bisect-*` flag parsing is untouched. It prints the vendored
`usage: git <cmd> ...` synopsis to stdout and exits 129 (0 for `--help`), matching git.

### Cluster B ‚Äî implicit `diff --no-index` outside a repo (test 8)
Outside a repo there is no work tree, so `parse_rev_and_paths` cannot classify args as paths
and funnels everything into `revs`, leaving `raw_path_args` empty ‚Äî the existing
classification-based `--no-index` fallback never fired. Added a raw-argv check before that
fallback: when `repo_opt.is_none() && !args.cached`, no `--` separator, exactly two
non-option operands that both exist on disk, call `run_no_index`. The existing
classification-based fallback is kept for the in-repo / with-`--` cases.

### Cluster A ‚Äî bare `git diff` emitted an empty patch body (test 4)
Test 4 was mislabeled as an apply bug. The real cause: test 1's `git diff >sample.patch`
produced a 0-byte file, so `git apply` correctly failed. A plain `git diff` (no
`--stat`/`--raw`/`--name-only`/..., no `-p`, no `-s`) detected the change but printed no
hunks because `show_unified_after_stat` only fired on an explicit `-p` alongside a stat or a
`--submodule` format. Broadened the gate to also fire when no competing format is active
(`|| !format_besides_unified_patch`), restoring git's default of emitting hunks. Contract
preserved: plain `--stat` still omits hunks; `--stat -p` still appends them; `-s`/`--no-patch`
still suppress them (those already clear `show_unified_patch`).
