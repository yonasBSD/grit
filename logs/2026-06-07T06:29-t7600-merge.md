## Bug fixed: cache-tree corruption on conflicted autostash apply (#66)

Repro (clean env): set up c0/c1/c3, dirty the worktree with `merge-file`, then
`git merge --autostash c3` where applying the stash on top of the merge result
conflicts. With `GIT_TEST_CHECK_CACHE_TREE=true` (the harness default) grit aborted:

    Created autostash: ...
    error: corrupted cache-tree has entries not present in index   (exit 128)

Root cause: `apply_stash_impl` (grit/src/commands/stash.rs) clones the current
index (which carries the on-disk `TREE` cache-tree extension) and then adds
unmerged stage-1/2/3 entries via the local `add_stage_entry` helper, which pushes
directly to `index.entries` WITHOUT invalidating the cache-tree. The resulting
index has fewer stage-0 entries than the stale cache-tree records, so
`verify_cache_tree` (run on every write under GIT_TEST_CHECK_CACHE_TREE) rejected
it. Git's `add_index_entry` invalidates the affected path's cache-tree, and a
conflicted index keeps no valid cache-tree at all.

Fix (grit/src/commands/stash.rs):
1. After staging the conflict entries in the three-way content-merge branch, call
   `new_index.invalidate_cache_tree_for_path(path_bytes)`.
2. As a robust catch-all, in the `if has_conflicts` block before write, call
   `new_index.clear_cache_tree()` — a conflicted (unmerged) index can never have a
   valid stage-0 cache-tree, matching Git which only keeps one for a fully merged
   index.

With the fix the merge succeeds (exit 0), prints "Applying autostash resulted in
conflicts.", leaves `UU file`, HEAD:file == result.1-9, and the stash holds the
original diff. #66 now passes.

## #70 was environment pollution, not a grit bug

`merge --no-ff --edit` runs `EDITOR=./editor git merge --no-ff --edit c1` and
expects the fake editor to prepend a line to the merge message. The editor never
ran. Traced via temporary probes: grit's editor resolution order (correctly, like
upstream `git_editor()`) is GIT_EDITOR > core.editor > VISUAL > EDITOR. My
interactive shell had `GIT_EDITOR=true` exported globally; test-lib.sh sets
`EDITOR=:` but does NOT unset `GIT_EDITOR`, so my `GIT_EDITOR=true` leaked into the
harness and overrode the test's `EDITOR=./editor`, resolving the editor to the
no-op `true`. Confirmed: `env -u GIT_EDITOR ./t7600-merge.sh` => all 83 pass even
before considering #66, and grit's `--no-ff --edit` works correctly in any clean
environment. No code change needed for #70.
