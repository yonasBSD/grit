# t5574-fetch-output — fetch output format

Ticket: e96bc3. Stem: t5574-fetch-output.sh.

## Root causes found

1. `fetch.output` config validation: value-less `-c fetch.output` (boolean-true) must
   produce `missing value ... unable to parse ... from command-line config`; empty/garbage
   must produce `invalid value for 'fetch.output': '<v>'`. Previously printed `'true'`.
   Fix: use `config.get_last_entry` (distinguishes `value: None`) in `fetch::run`.

2. Spurious `Total N (delta 0), reused 0 (delta 0)` on stderr for non-progress fetch.
   The local upload-pack (serve_v2 / upload_pack) always spawned `pack-objects --progress`,
   ignoring the client's `no-progress` request. Fix: thread a `progress` flag through
   `spawn_pack_objects_upload_shallow` (pass `--quiet` when client sent `no-progress`).

3. `would write to .git/FETCH_HEAD` printed on dry-run — git never prints this. Removed.

4. `--porcelain` + `--recurse-submodules=<yes|on-demand>` must `die` with
   "options '--porcelain' and '--recurse-submodules' cannot be used together"; porcelain
   otherwise forces recurse off. Added validation in `run` (before remote resolution so it
   fires even with no remote — test 8).

5. `--no-porcelain` was unrecognized. Added flag with `overrides_with` so the last of
   `--porcelain`/`--no-porcelain` wins (test 10).

6. Aligned/compact/porcelain output: rewrote the implicit-fetch + tag + prune report path to
   accumulate `FetchDisplayRecord`s into a `FetchDisplay` that computes the refcol width
   (max remote-ref name, floor 10) before printing, mirroring git's `display_state`. Added
   compact `*` substitution (`find_and_replace`) and porcelain stdout format
   `<code> <old> <new> <localref>`. Forced-update detection via merge-base ancestor check.

## Status

ALL 14/14 passing (was 1/14).

7. CLI explicit-refspec path (`fetch origin HEAD`, `HEAD:foo`, `<oid>:object-id`): routed the
   dst and no-dst update branches through FetchDisplay. No-dst now emits the FETCH_HEAD-only
   line (` * branch  HEAD -> FETCH_HEAD` / porcelain `* <zero> <commit> FETCH_HEAD`); new-ref
   classification uses the resolved remote ref (raw OID -> `[new ref]`).

8. dry-run was writing refs — `apply_single_ref_update`/`apply_single_ref_delete` ignored
   `args.dry_run`, so `--dry-run origin HEAD:foo` created `foo`; the following real fetch then
   saw it up-to-date and printed nothing. Added `dry_run` early-return to both.

9. Glob CLI-refspec path (tests 5,6): routed through FetchDisplay. A non-ff reject to a
   non-force destination now records a `!` line and (non-atomic) continues, exiting 1 quietly
   via ExitCodeError (no "could not be updated" hint → empty stderr in porcelain). With
   `--atomic` the same records are reported but the transaction is aborted (no refs / FETCH_HEAD
   written). FetchDisplay::flush stable-sorts deletes first to match Git's prune-before-updates.
   Added opportunistic remote-tracking updates (configured `refs/remotes/origin/*` applied
   alongside CLI wildcard refspecs) so the `origin/*` lines appear.

10. Restored a narrowly-gated `would write to .git/FETCH_HEAD` dry-run note for anonymous-path
    fetches only (`git fetch .`, no FETCH_HEAD display line, no CLI refspecs) so t5510 #69's
    `grep FETCH_HEAD` still matches without adding a spurious line to named-remote dry runs.
