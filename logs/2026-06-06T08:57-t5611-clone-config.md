## Failing tests
- `not ok 6 - clone -c remote.origin.fetch=<refspec> works`
- `not ok 8 - clone -c remote.<remote>.fetch=<refspec> --origin=<name>`

Both: `git clone -c "remote.<name>.fetch=+refs/grab/*:refs/grab/*" . child` failed to copy
`refs/grab/it` into the child. Test 7 (`git -c remote.origin.fetch=... clone`) already passed,
which gave the clue.

## Root cause
In `grit/src/commands/clone.rs`, the local (non-`file://`) clone path is:
1. `apply_clone_config` (line ~1026) writes `remote.origin.fetch=+refs/grab/*:...` into the dest
   config early (before fetch).
2. `setup_origin_remote` / `setup_origin_remote_url` then did
   `config.set("remote.<name>.fetch", "+refs/heads/*:refs/remotes/<name>/*")` — and `set` REPLACES
   the last matching entry, clobbering the user's `clone -c` refspec.
3. `copy_configured_remote_fetch_refspecs` (line ~1532) then read `remote.<name>.fetch` from the
   dest config to copy extra refs — but the grab refspec had already been overwritten, so
   `refs/grab/it` was never copied.

`git -c` (test 7) survived because that path reads the refspec from `GIT_CONFIG_PARAMETERS`
(handled separately in `command_line_config_values`), not from the dest config.

Upstream Git (`builtin/clone.c` `write_refspec_config`) appends the default tracking refspec with
`git_config_set_multivar_*`, producing a multi-valued `remote.<name>.fetch`: the user's spec first,
then the default. Verified against system git — child config is:
```
[remote "origin"]
	fetch = +refs/grab/*:refs/grab/*
	url = ...
	fetch = +refs/heads/*:refs/remotes/origin/*
```

## Fix
Added `write_default_remote_fetch(config, remote_name, refspec)` helper: if a
`remote.<name>.fetch` already exists (set by `clone -c`), `add_value` (append, multi-valued);
otherwise `set`. Wired it into both `setup_origin_remote` and `setup_origin_remote_url` in place
of the clobbering `config.set(...fetch...)`. grit child config now matches git byte-for-byte.
