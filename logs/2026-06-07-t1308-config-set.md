# t1308-config-set.sh — ticket 289f24

## Goal
Make `tests/t1308-config-set.sh` fully pass (was 38/39).

## Failing subtest
- 35: "proper error on error in default config files"
  - Test appends a stray `[` line to `.git/config`, then runs
    `test-tool config get_value foo.bar` and expects exit 128 with
    `fatal: bad config line N in file .git/config`.
  - grit printed `Value not found for "foo.bar"` and exited 1.

## Root cause
The config *parser* already detects the malformed `[` line and returns
`Error::Message("fatal: bad config line N in ...")` (grit-lib/src/config.rs
`ConfigFile::parse`, the `else` branch around line 716). The bug was upstream of
that: the `test-tool config get_value` handler resolved the repo git dir via
`grit_lib::repo::Repository::discover(None).ok()`. `discover` *reads and parses
the repository config* during discovery, so the syntax error made discovery
fail and return `None`. With `git_dir = None`, `ConfigSet::load(None, true)` read
no local config, found nothing, and printed "Value not found" (exit 1) — never
reaching the parse error.

The real `git config` command avoids this: its `resolve_git_dir()`
(grit/src/commands/config.rs) falls back to a config-free directory walk
(`precompose_config::locate_git_dir_from_cwd`) when `discover` fails, so the
local config file is still located and its parse error surfaces.

## Fix
grit/src/main.rs: added helper `test_tool_config_locate_git_dir()` that mirrors
`config.rs`'s `resolve_git_dir`: honor `GIT_DIR`, else `Repository::discover`,
else fall back to `locate_git_dir_from_cwd`. `run_test_tool_config` now uses it
to resolve `git_dir` instead of `Repository::discover(None).ok()`. So when the
config has a syntax error, `ConfigSet::load` is given the real git dir, parses
the broken `.git/config`, and returns the fatal error the handler already maps
to exit 128.

Left `read_early_config` branch unchanged (its discover-based path is what the
t1309 ceiling tests expect).

## Result

- `cargo test -p grit-lib --lib`: only the 2 known pre-existing
  `ignore::gitignore_glob_tests` failures (unrelated to this ticket).
- No new clippy warnings on changed code.
