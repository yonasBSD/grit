## Root cause (differential-verified vs git 2.52.0)
Subtest 3 did:
    count=$(find dry.git/objects -type f | wc -l) &&
    test "$count" = "0"

macOS BSD `wc -l` left-pads its output ("       0"), and command substitution
in an assignment preserves leading whitespace, so `count` is "       0", and the
string compare `test "       0" = "0"` fails. GNU wc (Linux) emits "0", so this
passed on Linux (23/23).

Differential proof:
- grit `unpack-objects -n` on the real pack exits 0 and writes ZERO loose object
  files (`find dry.git/objects -type f` prints nothing). This MATCHES real git
  2.52.0 (/opt/homebrew/bin/git) and the harness's REAL_GIT (/usr/bin/git 2.39.5):
  both also write zero loose objects in dry-run.
- The assertion's failure is purely the BSD wc padding. /usr/bin/git FAILS the
  exact same assertion identically; adding `| tr -d " "` makes the count "0" and
  the compare pass for both. This is a pure comparison-mechanism fix — no expected
  value changes — so no further differential proof is required.

This is a sanctioned BSD-wc-padding test bug. Sibling dry-run subtests 16 (qn)
and 22 (ns), plus subtests 11, 14, 23, already use `| tr -d " "`; subtest 3 was
the lone authoring oversight that omitted it.

## Fix applied (test body only)
tests/t5300-unpack-objects.sh subtest 3:
    -count=$(find dry.git/objects -type f | wc -l) &&
    +count=$(find dry.git/objects -type f | wc -l | tr -d " ") &&
