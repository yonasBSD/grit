## Failing subtest
6 - rebase --continue update from eucJP to ISO-2022-JP

## Diagnosis
The conflict commit was created with `i18n.commitencoding=eucJP` (so the commit
object stores an `encoding eucJP` header and an eucJP-encoded body). The config was
then switched to `ISO-2022-JP`, and a merge-backend rebase (`git rebase -m`) was run,
which conflicts.

During the conflicted pick:
- `write_rebase_conflict_message` writes `.git/rebase-merge/message` correctly in the
  *output* encoding (ISO-2022-JP): it decodes the original eucJP body to Unicode via
  the commit's own `encoding` header, then re-encodes to the configured encoding.
- The adjacent code, however, wrote `.git/MERGE_MSG` from `commit.message` — the
  *UTF-8* Unicode form — plus the `# Conflicts:` block.

On `rebase --continue`, `read_rebase_continue_message` prefers `MERGE_MSG` (it carries
the conflicts hint). It read the UTF-8 bytes but decoded them with the configured
non-UTF-8 encoding (`ISO-2022-JP`), turning the multibyte UTF-8 sequences into U+FFFD
replacement characters in the final committed message body.

This only bit eucJP→ISO-2022-JP (subtest 6), not eucJP→UTF-8 (subtest 5), because with
a UTF-8 output encoding `read_rebase_continue_message` takes the `String::from_utf8`
branch and the UTF-8 MERGE_MSG bytes survive intact.

## Fix
grit/src/commands/rebase.rs

1. New helper `append_rebase_conflicts_comment_block_bytes` — byte-level version of
   `append_rebase_conflicts_comment_block`, so the ASCII `# Conflicts:` block can be
   appended to non-UTF-8 message bytes without a lossy UTF-8 round-trip.
2. The conflicted-pick MERGE_MSG write now uses `transcoded_replayed_message(&commit,
   &config)` (the same transcoding `write_rebase_conflict_message` uses for
   `rebase-merge/message`) and appends the conflicts block via the byte-level helper.
   MERGE_MSG and `rebase-merge/message` are now in the same charset, so
   `read_rebase_continue_message` decodes correctly.

For the common UTF-8 case the bytes are identical to before (`raw_opt` is None →
`unicode.into_bytes()` + ASCII conflict block).
