Embedding Git fetch & push with grit-lib

The transport stack from PR #838 · grit-lib 0.4+

grit has always implemented every Git transport and the full wire protocol — but that code lived inside the CLI crate, shaped around argv, subprocesses, and stdout, so no other program could call it. PR #838 lifts that core into reusable, embedder-shaped grit-lib APIs behind a Transport trait. You can now fetch and push over git://, ssh, http(s), and local repositories entirely in-process — no gix, and no git subprocess on the wire — from any Rust program.

1. What PR #838 introduced

The PR adds four cooperating pieces to grit-lib, all designed so that the public surface is plain data and traits — the I/O lives inside the transport implementations, not in your call sites.

  • grit_lib::transport — a Transport / Connection trait pair plus three wire transports: GitDaemonTransport (git://), SshTransport (with a pluggable ssh command), and SmartHttpTransport<C> over a pluggable HttpClient trait.
  • grit_lib::fetch — fetch_remote drives the negotiation over any Connection; transport::http::http_fetch does the stateless-HTTP equivalent. Both return a structured FetchOutcome.
  • grit_lib::push — push_remote (streaming) and push_http, returning a PushOutcome with per-ref status (report-status parsed for you).
  • grit_lib::credentials — a CredentialProvider trait and a Git-compatible default that runs configured credential.helper programs, with typed non-interactive failure (it never blocks on a TTY prompt).

Fetch negotiates protocol v2 (streaming and over HTTP, with transparent downgrade to v0/v1); push uses v0/v1 (Git has no v2 receive-pack). Everything is hash-algorithm aware (SHA-1 and SHA-256), and packs are built from the negotiated want/have set, so a push never packs the whole object graph.

2. The transport matrix

The same two operations — fetch and push — work across all four transports. You pick the transport by constructing the matching type (or, more often, by letting a small helper choose from the URL scheme); the rest of your code is identical.

3. Adding the dependency

Add grit-lib to your Cargo.toml. The http-ureq feature gives you a ready-made HTTP client (UreqHttpClient); leave it off if you want to plug in your own HTTP stack instead.

# Cargo.toml
[dependencies]
grit-lib = { version = "0.4", features = ["http-ureq"] }

4. Core concepts

There are two layers. The transport layer opens a Connection to a remote; the operation layer (fetch_remote / push_remote) runs the protocol over that connection. For local and HTTP remotes there are direct entry points that bundle both.

The Transport trait

pub trait Transport {
    fn connect(&self, url: &str, service: Service, opts: &ConnectOptions)
        -> Result<Box<dyn Connection>>;
}

pub enum Service { UploadPack, ReceivePack }  // fetch, push

pub struct ConnectOptions {
    pub protocol_version: u8,        // 0/1/2; the server may downgrade
    pub server_options: Vec<String>,
}

Inputs & outputs

Every operation takes a typed options struct and returns a typed outcome — no parsing of porcelain, no stdout scraping.

pub struct FetchOptions {
    pub refspecs: Vec<String>,           // e.g. "+refs/heads/*:refs/remotes/origin/*"
    pub negative_refspecs: Vec<String>,
    pub tags: TagMode,                  // All / None / Following
    pub prune: bool,
    pub dry_run: bool,
    pub depth: Option<u32>,            // shallow fetch
    pub deepen_since: Option<String>,
    pub deepen_not: Vec<String>,
    pub unshallow: bool,
}  // implements Default

pub struct FetchOutcome {
    pub updates: Vec<RefUpdate>,         // remote_ref, local_ref, old/new oid, mode
    pub default_branch: Option<String>,   // from HEAD symref
    pub new_shallow: Vec<ObjectId>,
    pub new_unshallow: Vec<ObjectId>,
}
Progress. Every operation takes a &mut dyn Progress. Pass grit_lib::fetch::NoProgress to ignore sideband messages, or implement the one-method Progress trait to surface them.

5. Fetching

The pattern for a wire transport is always: build the transport, connect for UploadPack, then call fetch_remote with the connection. Here is a complete git:// fetch into an existing local repository:

use grit_lib::fetch::{fetch_remote, NoProgress};
use grit_lib::transfer::{FetchOptions, TagMode};
use grit_lib::transport::{ConnectOptions, GitDaemonTransport, Service, Transport};
use std::path::Path;

fn fetch_git_daemon(git_dir: &Path, url: &str) -> grit_lib::error::Result<()> {
    let transport = GitDaemonTransport::new();
    let opts = ConnectOptions { protocol_version: 2, ..Default::default() };
    let mut conn = transport.connect(url, Service::UploadPack, &opts)?;

    let fetch_opts = FetchOptions {
        refspecs: vec!["+refs/heads/*:refs/remotes/origin/*".to_owned()],
        tags: TagMode::Following,
        ..Default::default()
    };

    let outcome = fetch_remote(git_dir, conn.as_mut(), &fetch_opts, &mut NoProgress)?;
    for u in &outcome.updates {
        if let Some(local) = &u.local_ref {
            println!("{:?}  {}", u.mode, local);
        }
    }
    Ok(())
}

Over ssh the only change is the transport — SshTransport::new() spawns ssh from your environment (override it with SshTransport::with_program(...) or ::with_shell_command(...)):

use grit_lib::transport::SshTransport;

let transport = SshTransport::new();
let mut conn = transport.connect("git@github.com:owner/repo.git", Service::UploadPack, &opts)?;
let outcome = fetch_remote(git_dir, conn.as_mut(), &fetch_opts, &mut NoProgress)?;

For HTTP there is a direct entry point that handles discovery and the stateless POST loop for you — pass an HttpClient instead of a connection:

use grit_lib::transport::http::http_fetch;
use grit_lib::transport::http::ureq_client::UreqHttpClient;

// `.with_git_protocol("version=2")` asks the server for protocol v2.
let client = UreqHttpClient::new().with_git_protocol("version=2");
let outcome = http_fetch(&client, git_dir, "https://github.com/owner/repo.git",
                         &fetch_opts, &mut NoProgress)?;

And a local (file:// or path) fetch skips the transport entirely:

let outcome = grit_lib::transfer::fetch_local(git_dir, remote_git_dir, &fetch_opts)?;

6. Pushing

Push takes a slice of PushRefSpec (one per ref you want to update or delete) and a PushOptions, and returns a PushOutcome whose results carry a per-ref status you can inspect — fast-forward, forced, rejected (non-fast-forward / stale lease), or remote-rejected.

use grit_lib::push::push_remote;
use grit_lib::fetch::NoProgress;
use grit_lib::transfer::{PushOptions, PushRefSpec};
use grit_lib::push_report::PushRefStatus;
use grit_lib::transport::{ConnectOptions, GitDaemonTransport, Service, Transport};

// Resolve the local commit we want to publish.
let oid = grit_lib::refs::resolve_ref(git_dir, "refs/heads/main")?;

let spec = PushRefSpec {
    src: Some(oid),
    dst: "refs/heads/main".to_owned(),
    force: false,
    delete: false,
    expected_old: None,    // Some(oid) ⇒ force-with-lease (compare-and-swap)
    expect_absent: false,  // true ⇒ lease that the ref must not yet exist
};

// receive-pack is v0/v1, so connect with the default protocol version.
let transport = GitDaemonTransport::new();
let mut conn = transport.connect(url, Service::ReceivePack, &ConnectOptions::default())?;

let outcome = push_remote(git_dir, conn.as_mut(), &[spec],
                          &PushOptions::default(), &mut NoProgress)?;

for r in &outcome.results {
    match r.status {
        PushRefStatus::Ok | PushRefStatus::UpToDate =>
            println!("pushed {}", r.remote_ref),
        _ => eprintln!("rejected {}: {:?} {}",
                       r.remote_ref, r.status, r.message.as_deref().unwrap_or("")),
    }
}

The same PushRefSpec / PushOutcome shapes are used everywhere. For ssh, swap in SshTransport. For HTTP, use the convenience method on the transport (or push::push_http directly):

use grit_lib::transport::http::{SmartHttpTransport, ureq_client::UreqHttpClient};

let transport = SmartHttpTransport::new(UreqHttpClient::new());  // receive-pack is v0/v1: no version header
let outcome = transport.push(git_dir, "https://github.com/owner/repo.git",
                            &[spec], &PushOptions::default(), &mut NoProgress)?;

And local: grit_lib::transfer::push_local(git_dir, remote_git_dir, &[spec], &opts)?.

Server-side push options (git push --push-option) go in PushOptions::push_options; they are sent only when the remote advertises the capability, otherwise the push fails with a typed Error::PushOptionsUnsupported.

7. One helper for every transport

Because every transport produces the same FetchOutcome / PushOutcome, you can hide the choice behind a single function that dispatches on the URL scheme. This is exactly what the bundled gritx-fetch / gritx-push examples do — discover the transport (and therefore the authentication) from the remote URL, then run the operation:

use grit_lib::fetch::{fetch_remote, NoProgress};
use grit_lib::transfer::{FetchOptions, FetchOutcome};
use grit_lib::transport::http::{http_fetch, ureq_client::UreqHttpClient};
use grit_lib::transport::{ConnectOptions, GitDaemonTransport, Service, SshTransport, Transport};
use std::path::{Path, PathBuf};

fn fetch_any(git_dir: &Path, url: &str, opts: &FetchOptions)
    -> grit_lib::error::Result<FetchOutcome>
{
    let v2 = ConnectOptions { protocol_version: 2, ..Default::default() };
    if url.starts_with("http://") || url.starts_with("https://") {
        let client = UreqHttpClient::new().with_git_protocol("version=2");
        http_fetch(&client, git_dir, url, opts, &mut NoProgress)
    } else if url.starts_with("git://") {
        let mut c = GitDaemonTransport::new().connect(url, Service::UploadPack, &v2)?;
        fetch_remote(git_dir, c.as_mut(), opts, &mut NoProgress)
    } else if grit_lib::transport::is_ssh_url(url) {
        let mut c = SshTransport::new().connect(url, Service::UploadPack, &v2)?;
        fetch_remote(git_dir, c.as_mut(), opts, &mut NoProgress)
    } else {
        let remote = PathBuf::from(url.strip_prefix("file://").unwrap_or(url));
        grit_lib::transfer::fetch_local(git_dir, &remote, opts)
    }
}

The push side mirrors this with push_remote / SmartHttpTransport::push / push_local. See the example crate for the full version, including pushurl handling and remote-name resolution.

8. HTTP clients & credentials

The HTTP transport is generic over an HttpClient — grit-lib owns the smart-HTTP protocol, you supply the transport. The trait is deliberately tiny, so you can wrap any HTTP stack:

pub trait HttpClient: Send + Sync {
    fn get(&self, url: &str, git_protocol: Option<&str>) -> Result<Vec<u8>>;
    fn post(&self, url: &str, content_type: &str, accept: &str,
            body: &[u8], git_protocol: Option<&str>) -> Result<Vec<u8>>;
    fn git_protocol_header(&self) -> Option<&str> { None }
    fn smart_http_enabled(&self) -> bool { true }
}

With the http-ureq feature you get UreqHttpClient for free. To authenticate against a private remote, wire a CredentialProvider; the default HelperCredentialProvider runs the repo's configured credential.helper programs (e.g. osxkeychain) and is filled on a 401 then retried as HTTP Basic:

use grit_lib::config::ConfigSet;
use grit_lib::credentials::HelperCredentialProvider;
use grit_lib::transport::http::ureq_client::UreqHttpClient;

// Load the repo's git config (system + global + local), build the helper-backed
// provider, and also honor http.proxy / http.cookieFile / http.extraHeader.
let config = ConfigSet::load(Some(git_dir), true)?;
let provider = HelperCredentialProvider::new(config.clone());
let client = UreqHttpClient::from_config(&config)?
    .with_credential_provider(Box::new(provider))
    .with_git_protocol("version=2");
Non-interactive by design. When no helper can supply a credential, the provider returns a typed Error::Auth — it never opens a TTY prompt or hangs. Interactive prompting, if you want it, is an embedder concern you layer on top.

10. Runnable examples

The grit-examples crate ships two complete CLIs built entirely on these APIs:

  • gritx-fetch — fetch from a repo's default remote, auto-discovering the transport and authentication from its URL.
  • gritx-push — the push counterpart, with [+]<src>[:<dst>] refspecs, --force, and --push-option.
# from a checkout of grit
cargo run -p grit-examples --bin gritx-fetch -- origin
cargo run -p grit-examples --bin gritx-push  -- origin

Their shared src/remote.rs is a compact, real-world reference for the scheme-dispatch and credential wiring sketched above.
